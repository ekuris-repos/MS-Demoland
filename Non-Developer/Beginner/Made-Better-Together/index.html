Part 1

The Two-Platform Reality

Let's start with the situation many of you are living right now: your organization uses Azure DevOps for work tracking and GitHub for source code, or you are somewhere in the middle of a transition between the two.

Why Organizations Use Both

Azure DevOps

Mature work tracking with Boards, Backlogs, and Sprints. Deep integration with Azure Pipelines and Test Plans. Familiar to enterprise teams that have used TFS or VSTS for years.

GitHub

Industry-standard platform for source code, pull requests, and open-source collaboration. Home of GitHub Copilot, Actions, and the largest developer community in the world.

Most organizations did not choose to run two platforms on purpose. It happened naturally. Azure DevOps (and its predecessors, TFS and VSTS) has been the enterprise standard for work tracking, sprint planning, and test management for over a decade. Meanwhile, GitHub became the default for source code hosting, pull requests, and developer collaboration. When Microsoft acquired GitHub in 2018, these two ecosystems started converging, but many organizations still have significant investment in both. The good news is that Microsoft has built robust integration points between the two platforms, and Copilot adds an AI layer that works across both.

Common Friction Points

  • Context switching: PMs update work items in ADO while developers work in GitHub; neither sees the full picture in one place
  • Manual status syncing: Someone has to remember to update the ADO board when a PR is merged in GitHub
  • Duplicate tooling: Teams maintain Azure Pipelines and GitHub Actions for different parts of the same project
  • Reporting gaps: Stakeholder dashboards in ADO do not automatically reflect activity happening in GitHub
  • Security silos: Vulnerability findings live in one platform while the code and work items live in the other
Before we talk about solutions, let's name the friction. The number one complaint we hear is context switching. A project manager opens Azure Boards to check sprint progress, but the actual development work is happening in GitHub pull requests. Status updates require manual effort, and things fall through the cracks. Teams sometimes end up maintaining two CI/CD systems because Pipelines are configured in ADO and Actions are configured in GitHub. Reporting becomes fragmented. And security findings from tools like Dependabot or CodeQL may not be visible to the people managing work items in ADO. These are real problems, and the rest of this session is about solving them.
Part 2

GitHub Advanced Security for Azure DevOps

Now let's talk about one of the most important integration points: GitHub Advanced Security for Azure DevOps, commonly known as GHAzDO.

What Is GHAzDO?

  • GitHub Advanced Security features delivered natively inside Azure DevOps
  • Secret scanning: Detects credentials and tokens committed to ADO repositories
  • Dependency scanning: Identifies known vulnerabilities in third-party packages
  • Code scanning: Powered by CodeQL, the same engine used on GitHub.com
  • Security alerts appear directly in the Azure DevOps UI; no need to leave your workflow
  • Available for Azure DevOps Services (cloud) with per-active-committer licensing
GHAzDO brings three pillars of GitHub Advanced Security into Azure DevOps: secret scanning, dependency scanning, and code scanning. Secret scanning watches for accidentally committed credentials, API keys, and tokens. Dependency scanning checks your project's third-party packages against known vulnerability databases. Code scanning uses CodeQL, the same static analysis engine that powers security scanning on GitHub.com, to find bugs and vulnerabilities in your own code. All of these results appear directly in the Azure DevOps interface. Your security team and your project managers can see findings without needing a GitHub account. The licensing model is per active committer, so you only pay for people who actually push code to the repository.

Why GHAzDO Matters for You

Visibility Without GitHub

Security alerts surface in ADO dashboards. You can monitor risk, track remediation, and report to stakeholders entirely within Azure DevOps.

Unified Compliance

One set of security policies applied consistently across your ADO repos. Audit logs, alert trends, and remediation timelines all in one place.

For PMs and team leads, the value of GHAzDO is visibility. You do not need to manage a separate GitHub account or learn a different interface. Security alerts show up right where you already work. You can add security widgets to your ADO dashboards, track remediation progress during sprint reviews, and report compliance metrics to leadership. It also simplifies your governance story. Instead of maintaining security scanning configurations in two places, GHAzDO gives you a single pane of glass for all repositories hosted in Azure DevOps.
Part 3

Azure Boards + GitHub Integration

Let's move on to the integration that directly impacts your sprint planning and work tracking: Azure Boards connected to GitHub.

How Azure Boards Connects to GitHub

  • Install the Azure Boards app on your GitHub organization from the GitHub Marketplace
  • Link work items to GitHub activity using AB# syntax (e.g., AB#1234) in commits, PRs, and branches
  • Automatic state transitions: Work items move from "Active" to "Resolved" when a linked PR is merged
  • Status badges: GitHub PRs display the linked work item's state directly in the PR conversation
  • Bi-directional traceability: From a work item, see every commit and PR; from a PR, see the linked requirements
The Azure Boards integration with GitHub is straightforward to set up. You install the Azure Boards app from the GitHub Marketplace onto your GitHub organization, then connect it to your Azure DevOps project. Once connected, developers use the AB# syntax in their commit messages and pull request descriptions to link activity to work items. For example, writing "Fixes AB#1234" in a PR description creates a link back to work item 1234 in Azure Boards. When that PR is merged, the work item state can automatically transition. This gives you bi-directional traceability: from any work item in ADO, you can see all associated GitHub commits and PRs, and from any PR in GitHub, you can see which requirements it addresses. No more manual status updates.

Impact on Sprint Planning

Before Integration

  • PM asks developers for status updates in standup
  • Board updates happen hours or days after actual work
  • Sprint burndown charts do not reflect reality
  • End-of-sprint scramble to close items

After Integration

  • Work items update automatically when PRs merge
  • Boards reflect real-time development progress
  • Burndown charts stay accurate throughout the sprint
  • PMs spend standup time on blockers, not status
Let's talk about the practical impact on your daily work. Before this integration, sprint boards were only as current as the last time someone remembered to drag a card. Status updates in standup often sounded like "I think that's done, let me check." After connecting Azure Boards to GitHub, work items update automatically based on real development activity. When a developer opens a PR linked to a work item, the item moves to "In Review." When the PR merges, it moves to "Done." Your burndown chart stays accurate because it reflects actual completed work, not manual updates. This frees up standup time to focus on blockers and decisions instead of going around the room asking "what did you do yesterday?"

Copilot in Azure DevOps

Now let's look at how AI is showing up directly inside Azure DevOps itself.

Copilot Features in ADO Today

  • AI-generated PR descriptions: Copilot summarizes code changes into a clear, readable PR description automatically
  • Work item suggestions: Get AI-assisted recommendations for work item fields, descriptions, and acceptance criteria
  • Test plan generation: Copilot drafts test cases based on requirements described in work items
  • Pipeline troubleshooting: AI-assisted explanations of pipeline failures help teams resolve build issues faster
  • Natural language search: Ask questions about your project data in plain English
Copilot capabilities are arriving in Azure DevOps and expanding regularly. Today, some of the most useful features include AI-generated pull request descriptions, which save developers time by automatically summarizing what changed and why. For PMs, the work item assistance is particularly valuable: Copilot can help write acceptance criteria, suggest descriptions, and even draft test cases based on the requirements you have written. Pipeline troubleshooting is another area where Copilot helps. When a build fails, Copilot can explain the error in plain language and suggest fixes. And natural language search lets you query your project data conversationally. These features continue to expand, so check the Azure DevOps release notes regularly for updates.

Copilot in GitHub

Even if Azure DevOps is your primary work tracker, GitHub Copilot offers capabilities that benefit your entire team.

GitHub Copilot for Non-Developer Workflows

  • Copilot Chat in GitHub.com: Ask Copilot questions about any repository, PR, or issue directly in the browser
  • PR summaries: Click a button to generate a plain-language summary of any pull request's changes
  • Issue triage: Copilot can suggest labels, assignees, and related issues based on the content of a new issue
  • Documentation drafts: Generate release notes, README updates, and project documentation from recent activity
  • Copilot Extensions: Third-party extensions can connect Copilot to external tools like Jira, Sentry, or your own internal systems
GitHub Copilot is not just for writing code. On GitHub.com, Copilot Chat lets anyone ask questions about a repository in plain English. You can ask "What changed in the last sprint?" or "Summarize this pull request for a non-technical stakeholder," and Copilot will give you a clear answer. PR summaries are especially useful for PMs who need to review changes without reading code. Issue triage features help route incoming issues to the right team. Documentation generation can draft release notes from merged PRs. And Copilot Extensions allow your organization to connect Copilot to other tools in your workflow. Even if your boards live in ADO, these GitHub Copilot capabilities add real value to your planning and communication processes.

Copilot Chat for Stakeholder Communication

For PMs and Leads

  • "Summarize all PRs merged this sprint"
  • "What are the open security alerts in this repo?"
  • "Draft release notes for version 2.4"
  • "Which files changed the most this month?"

For Stakeholders

  • "Explain this PR in non-technical terms"
  • "What is the risk level of this change?"
  • "How does this feature relate to our Q3 goals?"
  • "Show me the test coverage for this feature"
Here are some real prompts you can use with Copilot Chat on GitHub.com today. As a PM or team lead, you might ask Copilot to summarize all the pull requests merged during a sprint, which gives you a quick status report without asking each developer individually. You can ask about open security alerts to prepare for a compliance review. Drafting release notes becomes a 30-second task instead of a 30-minute one. For stakeholders who need to understand technical changes, Copilot can translate code changes into business language. It can assess risk, connect changes to project goals, and surface test coverage data. This transforms GitHub from a tool that only developers use into a communication platform that serves the whole team.

Bridging Workflows

Now let's put it all together. How do you create a seamless workflow that spans both platforms?

Syncing Work Items and Pipelines

  • Azure Boards GitHub App: Automatically links GitHub commits and PRs to ADO work items using AB# references
  • GitHub Actions triggering ADO Pipelines: A merge in GitHub can kick off a release pipeline in Azure DevOps
  • ADO Pipelines triggering GitHub Actions: A successful build in ADO can trigger deployment workflows in GitHub Actions
  • Shared variables and secrets: Use Azure Key Vault as a common secret store accessed by both platforms
  • Webhook integrations: Custom webhooks connect events in one platform to actions in the other
The bridging layer between Azure DevOps and GitHub has several components. The Azure Boards GitHub App handles work item linking, which we already discussed. For CI/CD, you can configure cross-platform triggers so that a merge in GitHub triggers an Azure DevOps release pipeline, or a successful ADO build triggers a GitHub Actions deployment workflow. Secrets and configuration can be centralized in Azure Key Vault, which both platforms can access. For custom integrations, webhooks let you respond to events on one platform by taking action on the other. The key principle is to let each platform do what it does best and use integration points to keep everything connected. You do not have to pick one platform for everything.

Shared Dashboards and Reporting

ADO Dashboards

  • Sprint burndown with auto-updated work items
  • GHAzDO security alert widgets
  • Pipeline status and deployment history
  • Custom queries linking to GitHub PR data

GitHub Insights

  • Copilot usage metrics across your organization
  • PR cycle time and review throughput
  • Security overview for all repositories
  • Actions workflow success rates
Reporting is often the hardest part of a multi-platform setup, but it does not have to be. Azure DevOps dashboards can incorporate data from both platforms. Sprint burndown charts automatically reflect GitHub PR activity when Boards integration is enabled. GHAzDO widgets surface security alerts directly on your ADO dashboard. On the GitHub side, organization-level insights show Copilot adoption metrics, PR cycle times, and security posture across all repositories. For executive reporting, you can pull data from both platforms into Power BI, which has connectors for both Azure DevOps and GitHub. This gives you a single reporting layer even when the underlying activity spans two platforms.
Part 7

Migration Considerations

A question that comes up in almost every conversation: should we migrate everything to one platform, or keep using both?

When to Stay on Both Platforms

  • Heavy investment in ADO Test Plans: If your QA team relies on Test Plans extensively, there is no direct equivalent in GitHub
  • Complex process customization: ADO supports deeply customized work item types, states, and rules that GitHub Issues/Projects cannot replicate
  • Regulatory or compliance requirements: Some industries require audit trails and controls that ADO provides out of the box
  • Large-scale TFVC repositories: If you still use TFVC (Team Foundation Version Control), migration requires careful planning
  • Organizational readiness: Teams that are not ready for change benefit from a gradual, integrated approach
There are legitimate reasons to continue using both platforms. If your QA organization depends on Azure DevOps Test Plans for manual testing, exploratory testing, or test case management, moving away from ADO means finding an alternative for those workflows. If you have spent years customizing your ADO process templates with specific work item types, state transitions, and business rules, replicating that in GitHub Projects would require significant effort. Regulated industries sometimes have audit and compliance requirements that ADO's enterprise features address directly. And if you have legacy TFVC repositories, migrating version control is a separate project from migrating work tracking. The integration approach lets you get value from both platforms without forcing a big-bang migration.

When Consolidation Makes Sense

  • Reducing tool sprawl: If your organization is paying for and maintaining two platforms with overlapping features, consolidation saves cost and complexity
  • Maximizing Copilot value: GitHub Copilot's deepest integrations are with GitHub natively; consolidating unlocks the full feature set
  • Simplifying onboarding: New team members learn one platform instead of two, reducing ramp-up time
  • Open-source alignment: If your organization contributes to or consumes open source, GitHub is the natural home
  • GitHub Projects maturity: GitHub Projects has grown significantly and now supports many enterprise planning scenarios
On the other hand, consolidation makes sense when the overhead of maintaining two platforms outweighs the benefits. If your teams are not using ADO's advanced features like Test Plans or custom process templates, you may be paying for capabilities you do not need. GitHub Copilot works most deeply with GitHub's native features, so moving your work tracking to GitHub Issues and Projects gives you the tightest Copilot integration. Onboarding is simpler when new team members only need to learn one tool. And GitHub Projects has matured significantly over the past two years, adding views, custom fields, workflows, and roadmap capabilities that serve many enterprise planning needs. The decision is not all-or-nothing; many organizations migrate incrementally, moving one team or project at a time.

Practical Demo Scenarios

Let's walk through two practical scenarios that show how these integrations work in real life.

Demo: Sprint Planning with Copilot

  • Step 1: Open Azure Boards and review the product backlog for the upcoming sprint
  • Step 2: Use Copilot in ADO to draft acceptance criteria for high-priority work items
  • Step 3: Assign work items to the sprint and confirm capacity
  • Step 4: Developers create branches in GitHub linked to work items using AB# references
  • Step 5: As PRs merge throughout the sprint, Azure Boards updates automatically
  • Step 6: Use Copilot Chat on GitHub.com to generate a mid-sprint progress summary for stakeholders
Let's walk through a sprint planning scenario. You start in Azure Boards where the product backlog lives. During backlog refinement, you can use Copilot to help draft acceptance criteria for user stories, saving time and ensuring consistency. Once items are assigned to the sprint, developers create branches in GitHub and include the AB# reference in their branch names or commit messages. Throughout the sprint, as pull requests are opened and merged, the linked work items in Azure Boards transition automatically. At the mid-sprint checkpoint, instead of manually compiling a status report, you open Copilot Chat on GitHub.com and ask it to summarize all PRs merged since the sprint started. You get a stakeholder-ready summary in seconds. This workflow uses both platforms to their strengths, with Copilot adding an AI assistance layer at every step.

Demo: Stakeholder Reporting

  • ADO Dashboard: Open your project dashboard showing sprint burndown, velocity trends, and GHAzDO security widgets
  • Copilot PR Summaries: Click into any recent PR and use Copilot to generate a non-technical summary of the changes
  • Release Notes Draft: Ask Copilot Chat to compile release notes from all PRs merged since the last release tag
  • Security Posture: Review GHAzDO alerts on the ADO dashboard and use the trend data to show improvement over time
  • Export to PowerPoint: Use ADO's built-in export features or Power BI to create presentation-ready materials
The second scenario focuses on stakeholder reporting. Imagine you have a steering committee meeting tomorrow. You open your ADO project dashboard, which shows the sprint burndown chart, velocity trends from the last several sprints, and the GHAzDO security widgets showing open and resolved alerts. For any PR that stakeholders might ask about, you click into it and use Copilot to generate a plain-language summary. Before the meeting, you ask Copilot Chat to compile release notes from all PRs merged since the last tagged release. You review the GHAzDO security trend data to show that your team is reducing vulnerabilities over time. Finally, you export the dashboard or connect it to Power BI for a polished presentation. All of this takes minutes, not hours, and you never had to ask a developer for a status update.
