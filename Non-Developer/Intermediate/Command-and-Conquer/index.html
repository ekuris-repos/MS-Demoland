  • Developer satisfaction surveys capture sentiment, not output

What you need are metrics that connect AI tool usage to engineering outcomes you already care about: velocity, quality, and cost. That is exactly what we are going to build toward in this session.

Part 2

Copilot Metrics & Dashboards

Now let's get into the data layer. GitHub provides a comprehensive Metrics API and built-in dashboards that give you exactly the visibility we just discussed.

The Copilot Metrics API

  • REST API available at the organization and enterprise level
  • Returns daily aggregated metrics: suggestions, acceptances, active users, and language breakdowns
  • Data available for the last 28 days with daily granularity
  • Supports filtering by team, allowing you to compare adoption across business units
  • Enterprise endpoints aggregate data across all organizations in your GitHub Enterprise account
  • Authentication uses a personal access token or GitHub App with the manage_billing:copilot scope
The Copilot Metrics API is your primary programmatic interface to usage data. It operates at two levels: organization and enterprise. At the organization level, you get daily metrics for a single GitHub org, including total suggestions made, total acceptances, the number of active users, and breakdowns by programming language and IDE. At the enterprise level, the API aggregates data across every organization in your GitHub Enterprise account, giving you a single view for the entire company. The data covers a rolling 28-day window with daily granularity, and you can filter by team to compare adoption across different engineering groups. Authentication is straightforward; you need a token with the manage_billing:copilot scope. Many organizations build automated dashboards by calling this API on a schedule and pushing the results into their existing BI tools.

Built-In Copilot Dashboards

Organization Dashboard

Navigate to your GitHub org settings and select Copilot to see seat usage, acceptance rates, and active user trends. No API calls or custom tooling required.

Enterprise Dashboard

For Enterprise Cloud customers, the enterprise-level dashboard rolls up metrics from every organization, giving executives a single pane of glass across the company.

If you do not want to build your own dashboard, GitHub provides built-in views in both the organization and enterprise settings pages. The organization-level dashboard shows you seat utilization at a glance: how many licenses are assigned, how many are actively in use, daily acceptance rates, and trend lines over the past 28 days. For Enterprise Cloud customers, the enterprise dashboard aggregates these metrics across all organizations, which is critical if your company has multiple GitHub orgs. These dashboards are available today with no configuration or additional cost. They are the fastest way for a non-technical executive to get a snapshot of Copilot adoption. For more advanced analysis, such as correlating Copilot usage with DORA metrics or sprint velocity, you would use the API to feed data into your own BI platform.

Understanding Copilot Usage Data

  • Total Seats - Licenses assigned vs. licenses actively used. Identify shelfware before renewal.
  • Active Users - Developers who used Copilot at least once in the reporting period. Track adoption curves over time.
  • Suggestions & Acceptances - How many code completions Copilot offered and how many developers kept. Higher acceptance signals better prompt context.
  • Acceptance Rate - The percentage of suggestions accepted. Typically ranges from 25% to 40% across organizations.
  • Language Breakdown - Which programming languages see the most Copilot activity. Helps you target training and enablement.
  • Chat & IDE Engagement - How often developers use Copilot Chat, inline completions, and other features.
Let's walk through the key data points the API and dashboards provide. Total seats tells you how many licenses you are paying for versus how many are actually in use. This is your first line of defense against wasted spend; if 30% of your seats are idle, you need to either reallocate them or investigate why adoption is lagging. Active users counts how many developers used Copilot at least once in the reporting window. This is your adoption metric. Suggestions and acceptances tell you how much value developers are getting from Copilot's code completions. The acceptance rate, typically between 25% and 40%, is a useful proxy for how well Copilot understands your codebase. Language breakdown is particularly useful for targeting enablement efforts; if Copilot is heavily used in JavaScript but barely used in Java, you may need Java-specific training. Finally, chat and IDE engagement metrics help you understand which Copilot features are resonating with your teams.

Key Metrics That Matter to Leadership

Adoption Velocity

Track active users as a percentage of total seats over time. A healthy rollout shows steady week-over-week growth toward 80%+ utilization within the first 90 days.

Developer Hours Saved

Multiply accepted suggestions by average time-per-suggestion (typically 30-90 seconds) to estimate hours returned to developers each week. This is the number your CFO wants.

Two metrics consistently resonate with executive audiences. The first is adoption velocity: what percentage of your licensed developers are actually using the tool, and how quickly is that number growing? A successful pilot should see 60-80% utilization within the first 30 days. If you are below 40% after a month, something is blocking adoption and you need to investigate. The second metric is developer hours saved. This requires a simple calculation: take the number of accepted suggestions, multiply by an estimated time savings per suggestion (industry benchmarks range from 30 to 90 seconds depending on complexity), and convert to hours. Even using conservative estimates, the numbers are compelling. A team of 100 developers accepting 200 suggestions per day at 45 seconds each saves roughly 37 developer-hours per day. Over a year, that is more than 9,000 hours. Translate that into fully loaded developer cost and you have a dollar figure your finance team can work with.
Part 3

Building the Business Case

With the metrics in hand, let's build the ROI framework that will justify your Copilot investment to finance and the board.

ROI Frameworks for Copilot

  • Cost Avoidance - Developer hours saved multiplied by fully loaded cost per hour. The most straightforward and defensible calculation.
  • Throughput Gains - Measure increase in PRs merged, features shipped, or story points completed per sprint after Copilot adoption.
  • Quality Impact - Track changes in bug rates, code review turnaround, and production incidents before and after rollout.
  • Retention & Satisfaction - Developer experience surveys and attrition rates. Developers increasingly expect AI tools; not providing them is a recruiting disadvantage.
  • Time-to-Onboard - Measure how quickly new developers make their first meaningful contribution with and without Copilot assistance.
There are five lenses through which you can build a Copilot ROI case, and the strongest presentations use at least three. Cost avoidance is the simplest: multiply hours saved by your average fully loaded developer cost. Throughput gains require a before-and-after comparison of engineering output metrics; PRs merged per developer per week is a good starting point. Quality impact is harder to attribute directly to Copilot, but organizations consistently report faster code review cycles and fewer boilerplate bugs after adoption. Retention and satisfaction matter more than many executives realize. In a competitive job market, developers actively seek employers that provide modern AI tooling. Not offering Copilot is increasingly seen as a red flag during recruiting. Finally, time-to-onboard is a powerful metric for organizations with high hiring volumes. If Copilot helps a new developer contribute meaningfully two weeks sooner, that is real value you can quantify.

Calculating Real Cost Savings

  • Step 1: Determine your fully loaded developer cost per hour (salary + benefits + overhead, divided by working hours per year)
  • Step 2: Pull accepted suggestions per day from the Copilot Metrics API
  • Step 3: Multiply by estimated time savings per accepted suggestion (use 55 seconds as a conservative baseline)
  • Step 4: Annualize and compare against total Copilot license cost
  • Example: 500 developers × 150 accepted suggestions/day × 55 seconds = 1,145 hours/day saved
  • At $85/hour fully loaded: $97,000/day in developer time recovered, against ~$600/day in license cost
Let's walk through a concrete example. Start with your fully loaded cost per developer hour. For most US-based tech companies, this ranges from $70 to $120 per hour when you include salary, benefits, office costs, and tooling. Next, pull the accepted suggestions count from the Metrics API. Multiply that by a conservative estimate of time saved per suggestion. Industry data suggests 55 seconds is a reasonable baseline for code completions, though complex multi-line suggestions save considerably more. Annualize the result and compare it against your total license spend. In our example with 500 developers, even using conservative numbers, the daily value of time recovered is roughly $97,000, while the daily cost of Copilot Business licenses for 500 seats is approximately $600. That is a return of more than 160x. Even if you discount the time savings by 75% to account for context switching and suggestions that required modification, the ROI remains overwhelming. This is the kind of analysis that gets CFO sign-off.
Part 4

Governance & Policies

ROI only matters if you can deploy Copilot responsibly. Let's talk about the governance controls that make enterprise adoption possible.

Copilot Usage Policies

What to Allow

  • Code completions in standard development environments
  • Copilot Chat for code explanation, refactoring, and test generation
  • Pull request summaries and review assistance
  • Documentation generation from existing code

What to Restrict

  • Copilot in repositories containing classified or export-controlled data
  • AI-generated code in safety-critical systems without human review
  • Sharing proprietary algorithms or trade secrets in prompts
  • Bypassing code review for AI-generated changes
Every organization needs a clear Copilot usage policy, and it should be documented, communicated, and enforced. On the "allow" side, the vast majority of development activities benefit from Copilot: writing code, explaining code, generating tests, summarizing pull requests, and producing documentation. These are high-value, low-risk use cases. On the "restrict" side, you need to think about regulatory and compliance boundaries. If your organization handles classified information, export-controlled technology, or sensitive health data, you may need to exclude specific repositories from Copilot entirely. Safety-critical code, such as avionics, medical devices, or financial trading systems, should always require human review regardless of whether AI assisted in writing it. The key principle is that Copilot is a tool that assists developers; it does not replace engineering judgment, code review, or compliance processes.

Content Exclusions & IP Protection

  • Content exclusions let you specify files and repositories that Copilot will not read or use as context
  • Configured at the organization level in GitHub settings; applies to all members automatically
  • Use patterns to exclude entire directories: **/secrets/**, **/proprietary/**
  • Copilot Business and Enterprise do not retain prompts or suggestions, and do not train on your code
  • For additional assurance, Copilot Enterprise offers IP indemnity from Microsoft
  • Review your exclusion patterns quarterly to keep them current with your repository structure
Content exclusions are one of the most important governance features for enterprise customers. They allow you to define file patterns and repository paths that Copilot will completely ignore. It will not read those files for context, and it will not generate suggestions based on their content. This is configured at the organization level, so it applies automatically to every developer without requiring individual setup. Common exclusion patterns include secrets directories, proprietary algorithm implementations, and third-party code with restrictive licenses. It is also important to understand the data handling guarantees. With Copilot Business and Enterprise, your code is not used to train the model, prompts and suggestions are not retained after processing, and telemetry is limited to aggregate usage metrics. For organizations with strict IP concerns, Copilot Enterprise includes IP indemnity coverage from Microsoft, providing an additional layer of legal protection.

Org-Level Controls

  • Seat management: Assign Copilot to individual users, teams, or the entire organization
  • Feature toggles: Enable or disable specific Copilot capabilities (Chat, CLI, PR summaries) at the org level
  • Policy inheritance: Enterprise policies cascade to all organizations; org admins can only be more restrictive, not more permissive
  • Suggestion matching: Enable or disable the filter that blocks suggestions matching public code
  • Audit logs: All Copilot administrative actions are captured in the GitHub audit log for compliance
  • SSO integration: Copilot access is governed by your existing SAML SSO and SCIM provisioning
GitHub gives you comprehensive administrative controls for managing Copilot at scale. Seat management lets you assign licenses to specific users, to entire teams, or to the whole organization. Feature toggles let you control which Copilot capabilities are available; for example, you might enable code completions company-wide but restrict Copilot Chat to specific teams during an initial rollout. Policy inheritance is particularly important for enterprises with multiple organizations. Policies set at the enterprise level cascade down, and organization admins can only add restrictions, never remove them. This ensures consistent governance across the company. The suggestion matching filter blocks any suggestion that closely matches publicly available code, which is a common requirement for legal and compliance teams. All administrative actions are captured in the audit log, giving your security team full visibility into who changed what and when. Finally, Copilot access integrates with your existing SSO and SCIM setup, so you manage it through the same identity infrastructure you already use.
Part 5

Reporting & Strategic Planning

You have the data and the controls. Now let's talk about how to package all of this into reports that drive action and strategy that drives adoption.

Creating AI Adoption Reports

For the Board

  • Total investment vs. estimated hours saved
  • Adoption rate trend (% of seats actively used)
  • Competitive positioning statement
  • Risk posture: IP protections and compliance controls in place

For Engineering Leadership

  • Team-by-team adoption breakdown
  • Acceptance rates by language and repository
  • Feature usage: completions vs. Chat vs. PR summaries
  • Recommended actions: training, enablement, seat reallocation
Not all stakeholders need the same report. Your board of directors cares about three things: cost justification, competitive advantage, and risk. Give them the ROI calculation, the adoption trajectory, and a clear statement on IP protections and compliance. Keep it to one or two pages. Engineering leadership needs more granularity. They want to know which teams are adopting quickly, which are lagging, and what the acceptance rates look like across different languages and repositories. They also want actionable recommendations: which teams need training, where should you reallocate unused seats, and which Copilot features should you enable next. You can generate both reports using data from the Metrics API. Several organizations use Copilot itself to help draft these reports by feeding it the raw metrics and asking it to produce executive summaries, which is a fitting use of the tool to report on itself.

Reporting to Stakeholders

  • Cadence: Monthly reports for engineering leadership; quarterly reports for the board and C-suite
  • Automation: Use the Metrics API to auto-generate reports on a schedule. Reduce manual effort to near zero.
  • Benchmarking: Compare your metrics against published industry benchmarks from GitHub's research
  • Narrative: Numbers without context are noise. Frame every metric around a business outcome.
  • Recommendations: Every report should end with 2-3 specific actions, not just data
The best reporting cadence is monthly for engineering leadership and quarterly for the board. Monthly reports catch adoption problems early, while quarterly board reports show meaningful trends rather than noise. Automate data collection using the Metrics API so you are not spending hours pulling numbers manually. Benchmarking is valuable for context; GitHub publishes research on average acceptance rates and adoption curves that you can compare against. The most important advice here is to lead with narrative, not numbers. A slide that says "Acceptance rate: 32%" means nothing to a board member. A slide that says "Developers are keeping one in three AI suggestions, saving an estimated 4,200 hours per quarter, equivalent to $360,000 in developer time" tells a compelling story. Always end your reports with specific recommendations. Data without action items is just decoration.

Scaling from Pilot to Enterprise

Phase 1: Pilot (4-6 Weeks)

  • Select 2-3 willing teams (50-100 developers)
  • Define success metrics before you start
  • Assign a Copilot champion on each team
  • Collect baseline metrics for comparison

Phase 2: Expand (6-12 Weeks)

  • Roll out to all engineering teams with training
  • Publish usage policies and content exclusions
  • Stand up automated reporting
  • Address team-specific adoption blockers

Phase 3: Optimize (Ongoing)

  • Reclaim unused seats quarterly
  • Invest in prompt engineering training
  • Correlate Copilot data with DORA metrics
  • Evaluate Copilot Enterprise features

Decision Gates

  • Pilot → Expand: 60%+ active users, positive developer feedback
  • Expand → Optimize: 75%+ seat utilization, measurable throughput gains
  • Review governance policies annually at minimum
A phased rollout reduces risk and builds organizational confidence. In Phase 1, you pilot with two or three teams that are willing and enthusiastic. Define your success metrics upfront so you know what "good" looks like before the pilot begins. Collect baseline data on development velocity and quality so you have a fair point of comparison. Assign a Copilot champion on each pilot team to drive adoption and surface issues. In Phase 2, you expand to the full engineering organization. This is where governance becomes critical: publish your usage policies, configure content exclusions, and stand up automated reporting. Address adoption blockers team by team, because the reasons one team is not using Copilot may be completely different from another team's reasons. Phase 3 is ongoing optimization: reclaim unused seats, invest in advanced training, and start correlating Copilot metrics with your existing engineering metrics like DORA. Use clear decision gates between phases. Do not expand from the pilot until you see at least 60% active utilization and positive developer feedback.

Competitive Landscape

Executives often want to know how Copilot compares to alternatives. Let's address that directly.

How Copilot Compares

GitHub Copilot Strengths

  • Native integration with GitHub: PRs, issues, Actions, security
  • Enterprise governance: content exclusions, audit logs, SSO, policy inheritance
  • IP indemnity from Microsoft (Enterprise tier)
  • Multi-model architecture with access to OpenAI, Anthropic, and Google models
  • Copilot Extensions ecosystem for custom tool integrations

Evaluation Criteria

  • Code quality and suggestion accuracy in your primary languages
  • Enterprise administration and compliance controls
  • Integration with your existing DevOps toolchain
  • Data privacy, IP protection, and indemnification
  • Total cost of ownership including training and onboarding
When your board or peers ask "Why Copilot instead of alternatives?", focus on what differentiates the platform for enterprise customers. GitHub Copilot's primary advantage is its native integration with the GitHub platform. Copilot does not just generate code; it understands your pull requests, your issues, your Actions workflows, and your security findings. No other tool has that level of integration with the platform where your code already lives. Enterprise governance is another key differentiator: content exclusions, audit logging, policy inheritance across organizations, and SSO integration are table stakes for large companies, and Copilot delivers all of them. The multi-model architecture means you are not locked into a single AI provider; Copilot can leverage models from OpenAI, Anthropic, and Google. When evaluating any AI coding tool, focus on five criteria: suggestion quality in your specific languages, enterprise controls, DevOps integration, data privacy, and total cost of ownership. Run a structured evaluation if your organization requires it, but ensure you are comparing like-for-like on the governance and compliance dimensions, not just completion accuracy.
