Part 2

GitHub Issues & Projects

Now let's get into the primary tools you will use every day: GitHub Issues and GitHub Projects.

Planning Sprints with GitHub Projects

  • GitHub Issues are your work items: bugs, features, and tasks
  • GitHub Projects provide board, table, and roadmap views for organizing work
  • Custom fields let you track priority, effort, sprint, and team in one place
  • Built-in automations move issues through columns based on PR status
  • Everything stays connected: issues link to PRs, PRs link to code, code links to deployments
  • Sub-issues let you break epics into trackable child items, and issue types (Bug, Feature, Task) provide structured categorization
GitHub Issues are your fundamental work items. Every bug, feature request, task, or spike lives as an issue. GitHub Projects sits on top of issues and gives you flexible views: a kanban board for daily standups, a table view for bulk editing and filtering, and a roadmap view for timeline-based planning. You can add custom fields for priority, story points, sprint number, team assignment, and anything else your process needs. Built-in automations can move issues to "In Progress" when a linked PR is opened, and to "Done" when it merges. The real power is traceability. An issue links to a PR, the PR links to code changes, and those changes link to a deployment. As a PM, you get end-to-end visibility from requirement to production.

Tracking Milestones and Progress

  • Use Milestones to group issues by release or deadline
  • The milestone progress bar shows percentage complete at a glance
  • Combine milestones with Project views to filter by sprint or release train
  • Use labels consistently: bug, feature, blocked, needs-review
Milestones let you group related issues under a release or deadline. Each milestone includes a progress bar that shows how many issues are open versus closed, giving you an instant snapshot of where things stand. Pair milestones with Project views and you can filter your board to show only Sprint 12 issues, or only items tagged for the Q3 release. Labels are equally important for a PM. Establish a consistent labeling strategy with your team: bug, feature, blocked, needs-review, and so on. When labels are consistent, you can filter, search, and report on them reliably. Inconsistent labels create noise; consistent labels create insight.

Understanding Technical Work

This is where Copilot becomes your secret weapon. Let's talk about understanding what your developers are actually building.

Reading Pull Requests with Copilot

  • Pull requests (PRs) show every code change before it goes live
  • Copilot generates PR summaries automatically on github.com
  • Ask Copilot Chat: "Summarize this PR for a non-technical audience"
  • Identify scope: "How many files changed and what areas of the app are affected?"
  • Assess risk: "Does this PR touch any security-sensitive files?"
Pull requests are where code changes live before they go into production. Every PR has a diff showing exactly what changed, but you do not need to read the diff yourself. Copilot can generate PR summaries automatically on github.com, and you can also paste a PR link into Copilot Chat and ask for a plain-English summary. Try prompts like "Summarize this PR for a non-technical audience" or "How many files changed and what areas of the app are affected?" You can even ask risk-oriented questions: "Does this PR touch any security-sensitive files?" or "Is this a large change that might need extra testing?" These answers used to require a developer to explain things in a meeting. Now you can get them in seconds.

What to Ask Copilot

  • "What could break if this is wrong?"

What You Get Back

  • Plain-language explanations
  • Impact and scope assessments
  • Risk factors and dependencies
  • Context for standup and retro conversations
You do not need to understand code to understand the impact of a change. The left column shows questions you can ask Copilot about any PR or commit. The right column shows what you get back. "What does this change do?" gives you a summary you can use in a standup. "Why might this take longer than expected?" helps you anticipate delays. "What could break if this is wrong?" gives you risk context for your stakeholder report. These questions turn you from a passive observer of development into an informed, engaged partner.
Part 4

Writing Better Issues

Now that you understand how to read what developers produce, let's improve what you give them. Good issues save everyone time.

Anatomy of a Great Issue

  • Clear title: action-oriented, specific, and searchable
  • Context: why this work matters and who it serves
  • Acceptance criteria: concrete, testable conditions for "done"
  • Scope boundaries: what is included and, equally important, what is not
  • Resources: links to designs, specs, related issues, or relevant PRs
A well-written issue saves hours of back-and-forth. The title should be specific and action-oriented: "Add export button to dashboard" is better than "Dashboard improvements." Context explains why the work matters: who requested it, what problem it solves, and how it connects to broader goals. Acceptance criteria define "done" in testable terms: "User can click Export and receive a CSV file within 5 seconds." Scope boundaries are just as important. State what is not included to prevent scope creep. Finally, link to relevant designs, specs, or related issues so the developer has everything in one place.

Using Copilot to Clarify Requirements

  • Draft an issue, then ask Copilot: "Improve this issue description for clarity"
  • Generate acceptance criteria: "Write acceptance criteria for this feature"
  • Identify missing details: "What questions would a developer ask about this issue?"
  • Break down large features: "Split this epic into smaller, independently deliverable issues". Link the resulting issues as sub-issues under the original epic for built-in progress tracking.
  • Standardize language: Copilot helps keep terminology consistent across your backlog
Copilot is excellent at refining issue descriptions. Start with a rough draft of what you want built, paste it into Copilot Chat, and ask it to improve the description for clarity. It will tighten the language, add structure, and flag ambiguity. You can also ask it to generate acceptance criteria based on your description. One of the most valuable prompts is "What questions would a developer ask about this issue?" This surfaces gaps you might not have noticed. For larger features, ask Copilot to break an epic into smaller, independently deliverable issues. This produces better sprint planning and more predictable delivery. Over time, Copilot also helps standardize how you write issues, keeping terminology consistent across the entire backlog.
Part 5

Stakeholder Reporting

Reporting is one of the areas where Copilot delivers the most immediate time savings. Let's look at how to generate leadership-ready reports in minutes.

Status Reports and Sprint Summaries

  • Ask Copilot to summarize completed work from your closed issues and merged PRs
  • Generate sprint summary narratives from raw data points
  • Create executive summaries that highlight outcomes, not implementation details
  • Adjust tone and detail level: technical for engineering leads, strategic for VPs
Status reports used to mean an afternoon of copying ticket titles into a slide deck. With Copilot, you gather your data points, such as the list of closed issues, merged PRs, and outstanding blockers, then ask Copilot to draft a sprint summary. The key is adjusting for your audience. For engineering leads, include technical details: which services were updated, which bugs were resolved, and what remains. For VPs and executives, focus on outcomes: features shipped, user-facing improvements, and progress toward quarterly goals. You can literally run the same data through two different prompts and get two perfectly tailored reports. That flexibility is what makes this so powerful.

Velocity Charts and Visual Reporting

  • GitHub Projects provides built-in Insights with configurable historical and current-state charts. You can set these up to track sprint progress over time
  • Export project data to CSV for further analysis or custom dashboards
  • Use Copilot to interpret trends: "What does this velocity trend suggest?"
  • Generate chart descriptions for reports: "Write a paragraph explaining this burndown chart"
GitHub Projects includes built-in Insights with configurable historical and current-state charts that you can set up to track sprint progress over time. If your team uses story points or issue counts, these charts update automatically as work progresses. You can also export your project data as a CSV and build custom charts in Excel or Google Sheets. Where Copilot adds value is in interpretation. Paste your velocity numbers into Copilot Chat and ask "What does this velocity trend suggest about our release timeline?" or "Write a paragraph explaining this burndown chart for my weekly stakeholder update." Copilot turns raw numbers into narratives that stakeholders can understand and act on. This is especially useful for PMs who present data regularly but want to ensure the narrative is clear and consistent.

Risk Identification

Great PMs do not just report on what happened. They anticipate what might go wrong. Let's talk about using Copilot for proactive risk identification.

Analyzing Project Health Signals

  • Aging issues: issues open for 2+ sprints may signal unclear requirements or hidden complexity
  • Blocked items: look for the blocked label or stalled PR reviews
  • PR review bottlenecks: PRs waiting 3+ days for review slow the entire team
  • Sprint carryover: consistently carrying work forward suggests estimation problems
  • Inactive contributors: team members with no commits may need support or reassignment
Every project has health signals hiding in plain sight. Aging issues, those sitting open for two or more sprints, are a red flag. They often indicate unclear requirements, underestimated complexity, or a feature nobody wants to tackle. Blocked items are another signal: look for the "blocked" label or PRs that have been waiting for review for days. PR review bottlenecks deserve special attention. If PRs sit for three or more days without a review, the entire team slows down. Sprint carryover is a planning signal: if you consistently carry 20% of your sprint into the next one, your estimation process needs adjustment. Finally, watch for inactive contributors. Someone with no commits for a sprint might be stuck, underloaded, or overwhelmed with meetings. Each of these signals is actionable if you catch them early.

Using Copilot to Surface Risks

Prompts to Try

  • "List all issues labeled 'blocked' and suggest next steps"
  • "Which PRs have been open for more than 5 days?"
  • "Summarize the top 3 risks for this sprint"
  • "What patterns do you see in our overdue items?"

Actions to Take

  • Escalate blocked items in standup
  • Reassign stalled PR reviews
  • Add risk items to the sprint retrospective
  • Adjust upcoming sprint scope proactively
The left column shows prompts you can use with Copilot to surface risks. Feed it your issue list or project board data and ask targeted questions. "Which PRs have been open for more than 5 days?" gives you a concrete list to act on. "Summarize the top 3 risks for this sprint" generates a risk narrative you can share in your standup or steering meeting. The right column shows what to do with those insights. Escalate blockers immediately. Reassign stalled reviews so PRs keep moving. Add systemic risk patterns to your retrospective so the team can address root causes. And when the data says your sprint is overcommitted, adjust scope proactively rather than waiting for things to slip. The difference between a reactive PM and a proactive one is often just having the right data at the right time.
Part 7

Communication

PMs are the communication hub of a project. Let's look at how Copilot helps you draft the right message for every audience.

Drafting Release Notes and Updates

  • Use merged PRs and closed issues as source material for release notes
  • Ask Copilot: "Write release notes for version 2.4 based on these merged PRs"
  • Separate entries by category: Features, Bug Fixes, Improvements, Breaking Changes
  • Generate both internal (detailed, technical) and external (user-facing, benefit-oriented) versions
Release notes are a communication touchpoint that reaches multiple audiences. Developers want to know what changed technically. Product owners want to see feature progress. End users want to understand what is new and what is fixed. Start by gathering your merged PRs and closed issues for the release. Paste that list into Copilot Chat and ask it to write release notes organized by category: Features, Bug Fixes, Improvements, and Breaking Changes. You can generate two versions: an internal one with technical details for the engineering team, and an external one with user-friendly language focused on benefits. This dual-output approach saves you from writing the same information twice in different styles.

Technical Summaries for Non-Technical Audiences

  • Ask Copilot to translate technical jargon into business language
  • Prompt: "Rewrite this technical update for a VP who does not write code"
  • Focus on impact and outcomes rather than implementation details
  • Include: what shipped, what it means for users, and what is next
  • Copilot preserves accuracy while adjusting the level of abstraction
One of the hardest parts of a PM's job is translating between technical and business audiences. Developers think in terms of APIs, database migrations, and refactors. Executives think in terms of revenue impact, user experience, and risk. Copilot bridges this gap. Take a technical update from your engineering lead and prompt Copilot with "Rewrite this for a VP who does not write code." The result focuses on impact and outcomes: what shipped, how it affects users, and what comes next. Copilot preserves the factual accuracy while adjusting the level of abstraction. This means fewer misunderstandings, fewer follow-up questions, and more credibility for you as the person who always delivers clear, audience-appropriate communication.

Metrics That Matter

Not everything that can be measured should be measured. Let's focus on the metrics that actually drive decisions.

What to Track and Why

Delivery Metrics

  • Velocity: story points or issues completed per sprint
  • Cycle time: time from "In Progress" to "Done"
  • Sprint completion rate: percentage of planned work delivered
  • PR merge time: time from PR open to merge

Health Metrics

  • Carryover rate: work that spills into the next sprint
  • Bug-to-feature ratio: time spent on fixes vs. new work
  • Blocker frequency: how often work gets stuck
  • Review turnaround: time waiting for PR reviews
Not all metrics are equal. On the left, we have delivery metrics. Velocity tells you how much work the team completes per sprint, which is essential for forecasting. Cycle time measures how long it takes from when work starts to when it is done, and shorter cycle times generally mean better flow. Sprint completion rate shows whether you are planning accurately. PR merge time reveals how quickly code moves from review to production. On the right, health metrics. Carryover rate flags persistent overcommitment. Bug-to-feature ratio shows whether the team spends too much time on reactive fixes versus proactive development. Blocker frequency tells you how often work stalls. And review turnaround highlights bottlenecks in the code review process. Track these metrics consistently and you will have the foundation for data-driven conversations with your team and leadership.

Presenting Metrics to Leadership

  • Lead with the story, not the data: "We shipped 12 features this quarter" beats a raw velocity chart
  • Use trends over snapshots: show direction, not just a single number
  • Connect metrics to business outcomes: "Faster cycle time means features reach users sooner"
  • Highlight actions taken, not just observations: "We identified a review bottleneck and reduced wait time by 40%"
  • Ask Copilot: "Turn these metrics into an executive summary paragraph"
Data without context is just numbers. When presenting to leadership, lead with the story. "We shipped 12 features this quarter, a 25% increase from Q2" is more compelling than a velocity chart with no commentary. Use trends rather than snapshots. A single sprint's velocity is noise; a six-sprint trend line is a signal. Always connect metrics to business outcomes. Executives do not care about cycle time as an abstract concept, but they care when you say "Faster cycle time means features reach users two weeks sooner." Highlight actions, not just observations. Saying "We identified a review bottleneck and reduced PR wait time by 40%" demonstrates that you use data to drive improvement, not just to report. Copilot can help with all of this: paste your metrics and ask it to generate an executive summary paragraph.
