Token Optimization - The Credit Crunch

Conserve AI Credits by deciding which workflows AI should own, then building the skills, instructions, and agent personas that make those workflows lean

Welcome to Token Optimization, the second part of our usage-based billing story. The first course explained what GitHub AI Credits are and how spend works. This course answers the natural next question: now that every token has a price, how do we get the most value from each one? The answer is not to use Copilot less. The answer is to be deliberate. We will define the workflows AI should actually own, and then build durable artifacts, skills, instructions, and agent personas, that make those workflows token-efficient by default. This session is for platform teams, Copilot administrators, and senior engineers who want to operationalize spend-aware AI across the organization.

Part 1

The Credit Crunch

Let us set the stage. If your organization already understands the billing model, this part will feel familiar, and that is intentional. We only need enough of a recap to make the optimization argument land. The real message of Part 1 is that billing literacy was step one. Workflow discipline is step two, and it is where the durable savings actually come from.

Recap: AI Credits in One Slide

  • GitHub Copilot usage is now metered in GitHub AI Credits, where 1 credit = $0.01 USD
  • Every billed interaction consumes input, output, and sometimes cached tokens
  • Token usage is converted to credits using the published rate for the model you used
  • Chat, CLI, cloud agent, and code review are billed; completions and Next Edit Suggestions stay included
  • Business and Enterprise seats contribute to a shared, pooled credit balance
This is the entire prior course compressed into one slide. If your audience took the billing course, treat this as a five-second refresher. If they did not, this is the minimum they need. The key line is that credits map to tokens and tokens map to behavior. More context, longer output, and heavier models cost more. That single fact is what makes workflow design a financial decision and not just an engineering preference.

Why Optimization Is the Next Frontier

  • The promotional included usage from the billing transition does not last forever
  • Agentic and multi-step workflows are becoming the default, and they consume the most tokens
  • Budgets and pooling control the ceiling; workflow design controls the baseline
  • Coaching individual prompts scales poorly; durable artifacts scale across the whole team
  • The goal is not minimal usage. The goal is high-value usage per credit
Budgets and pooling are reactive. They tell you when you are close to a limit, but they do not change why you got there. Workflow design is proactive. It lowers the baseline cost of getting work done, which is far more durable than chasing individual prompts. The reason we emphasize artifacts over coaching is simple math. You can coach one developer's prompt today, but a well-built skill or instruction file improves every interaction from every developer, every day, with no repeated effort.

What Actually Consumes Tokens

Token source What it includes Why it grows
Instruction and context System prompts, instruction files, referenced files, repo context Broad scope and bloated instruction files inflate every request
Conversation history Prior messages carried forward in a session Long sessions re-send stale turns on every new message
Tool calls and results Agentic loops, file reads, terminal output, search results Each tool round-trip adds input and output tokens
Model output The generated response itself Verbose, unbounded answers cost more than constrained ones
Many teams assume the prompt they type is the cost. It is only a fraction. The assembled request includes system instructions, every instruction file in scope, referenced files, the running conversation history, and the results of any tool calls the agent makes. Output tokens add to that. When you see these four sources laid out, the optimization strategy becomes obvious. Shrink the context, trim the history, bound the tool use, and constrain the output. Defined workflows and good artifacts do exactly that, automatically.

The Hidden Cost of Unstructured AI Use

Unstructured

  • Every developer re-explains the same context from scratch
  • Vague prompts force the agent to explore the repo to understand intent
  • Mega-sessions drag stale history into every new question
  • No reusable knowledge, so the same work is paid for repeatedly

Workflow-Driven

  • Shared artifacts carry context once, not per developer
  • Clear scope means the agent acts instead of exploring
  • One task, one focused session with only relevant context
  • Knowledge is captured once and reused at near-zero marginal cost
The most expensive pattern in most organizations is not a single heavy agent run. It is repetition. Ten developers independently re-explaining the same build process, the same coding standards, and the same migration steps, over and over, every session. Each explanation is paid for in input tokens, every single time. Workflow-driven teams pay for that knowledge once by capturing it in an artifact, and then reuse it for almost nothing. That contrast is the entire thesis of this course.
Part 2

Define Workflows First

This is the heart of the course. Before you build a single artifact, before you pick a model, before you write a prompt, you decide what work AI should actually own. Skipping this step is the root cause of most wasted spend. Teams that buy the tools first and figure out the workflows later end up paying for a lot of expensive wandering.

The "Tools Before Workflows" Mistake

  • Teams often adopt agents, MCP servers, and models, then look for places to use them
  • This inverts the right order and produces expensive solutions in search of a problem
  • Without a defined workflow, the agent has no clear success criteria, so it over-explores
  • Over-exploration means more context, more tool calls, and more output, all billed
  • Start with the outcome you want, then choose the smallest capable tool to deliver it
There is a strong temptation, especially in enterprises, to lead with capability. We bought the agent, so let us find work for it. That is backwards, and under usage-based billing it is also costly. An agent without a clearly defined workflow does not know when it is done, so it keeps gathering context and trying approaches. Defining the workflow first gives the agent a destination, which is the single most effective way to keep a multi-step run from ballooning in cost.

What Makes a Good AI Workflow

  • Repeatable: the team performs it often enough that automation pays off
  • Well-scoped: clear inputs, clear outputs, and a definition of done
  • Rules-based: it follows conventions that can be written down and reused
  • Verifiable: success can be checked by tests, a build, or a clear acceptance bar
  • High-leverage: it saves meaningful human effort or reduces real risk
Not every task deserves an AI workflow. The best candidates are repeatable, well-scoped, rules-based, verifiable, and high-leverage. When a task has all five properties, you can capture it in an artifact and the AI can execute it reliably and cheaply. When a task is a one-off, ambiguous, or impossible to verify, automating it usually costs more than it saves because you pay for exploration and then pay again for human rework. Use these five criteria as a filter before you invest in any artifact.

Workflows AI Should and Should Not Undertake

AI should own Keep human-led
Dependency upgrades with a known migration path Architecture decisions with long-term trade-offs
Applying consistent coding standards and lint fixes Ambiguous product requirements that need clarification
Generating tests for well-specified behavior Security-sensitive changes without a defined checklist
Drafting changelogs and release notes from diffs One-off exploratory spikes with no clear definition of done
Repetitive refactors that follow a clear pattern Judgment calls that require business or domain context AI lacks
This is the slide to photograph. The left column shares a pattern: known steps, clear conventions, and a verifiable result. Those are exactly the workflows where AI is both effective and cheap, because the work is bounded. The right column shares the opposite pattern: ambiguity, long-term consequences, or missing context. Forcing AI into the right column tends to generate cost without confidence. Notice the very first item on the left, dependency upgrades with a known migration path. That is the worked example we will carry through the rest of the course.

Mapping and Prioritizing Your Workflows

graph TD
  A[List recurring team tasks] --> B{Repeatable and rules-based?}
  B -->|No| C[Keep human-led for now]
  B -->|Yes| D{Verifiable result?}
  D -->|No| C
  D -->|Yes| E{High frequency or high effort?}
  E -->|Low| F[Backlog: revisit later]
  E -->|High| G[Define workflow and build artifacts]
You do not have to define every workflow at once. Use a simple decision flow to prioritize. Start by listing the tasks your team repeats. Filter for the ones that are rules-based and verifiable, since those are automatable. Then prioritize by frequency and effort, because that is where the savings concentrate. The high-frequency, high-effort, rules-based, verifiable tasks are where you invest in artifacts first. Everything else goes to a backlog or stays human-led. This keeps your optimization effort focused on the workflows that actually move the credit needle.

Anatomy of a Defined Workflow

Worked example: a dependency upgrade and migration assistant.

Inputs and Triggers

  • A target package and version
  • The current lockfile and dependency manifest
  • The project's test command

Steps and Guardrails

  • Read release notes and breaking changes
  • Apply documented codemods and API changes
  • Stay within the affected modules only
  • Run tests and report what changed

Definition of done: the upgrade builds, tests pass, and a concise summary lists the breaking changes addressed.

A defined workflow is more than a sentence. It has explicit inputs, a bounded set of steps, guardrails that keep the agent from wandering, and a clear definition of done. Look at the guardrails in particular. Stay within the affected modules only is a token-saving instruction as much as a correctness one, because it stops the agent from reading the entire repository. This is the workflow we will turn into artifacts in the next part, and the one you will build in the lab.
Part 3

Skills, Instructions, and Agent Personas

Now that we have a defined workflow, we make it durable and cheap by capturing it in artifacts. You have already learned what instructions, skills, and agents are in the earlier courses. Here we focus on a different question: which artifact supports which part of a workflow, and when do you reach for the commands that scaffold them. The artifacts are the mechanism that turns a one-time definition into permanent token savings.

Three Artifacts That Support a Workflow

Artifact Role in the workflow Loaded when
Instructions Standing rules and conventions that always apply Automatically, scoped by path or repo
Skills Packaged, on-demand procedures for a specific task Only when the task matches the skill description
Agent personas A scoped role with a mindset and a defined tool set When the user invokes that agent
These three artifacts are complementary, not competing. Instructions are the standing rules that should always shape behavior, so they load automatically. Skills are heavier, detailed procedures that you only want loaded when they are relevant, which is why their description-based triggering matters so much for cost. Agent personas bundle a role, a mindset, and a constrained tool set for invoking on demand. The art is matching each part of your defined workflow to the right artifact, which is what the next few slides cover.

Instructions: Standing Rules

  • Use instructions for conventions that apply to every matching interaction
  • Scope them with an applyTo pattern so they only load for relevant files
  • Keep them lean: instruction files are sent on every request in scope
  • For the migration workflow: pin the package manager, the test command, and the rule to touch only affected modules
  • A bloated instruction file is a recurring tax on every prompt, so prune ruthlessly
Instructions are the cheapest artifact to reason about and the easiest to get wrong. Because they load automatically on every request in scope, a lean instruction file pays dividends and a bloated one quietly inflates every interaction. For our migration assistant, the instructions are short standing rules: which package manager we use, the exact test command, and the guardrail to stay within affected modules. Notice we do not put the full migration procedure here. That belongs in a skill, loaded only when needed, which we cover next.

Skills: Packaged Procedures, Loaded on Demand

  • Use a skill when a workflow needs a detailed, repeatable procedure that should not load every time
  • The skill description is the trigger, so write it to match the task precisely
  • Detailed steps stay out of the always-on context until the task actually calls for them
  • For the migration workflow: capture the upgrade steps, codemod patterns, and verification process
  • When to reach for it: scaffold one with /create-skill once the procedure is worth reusing across the team
Skills are where detailed knowledge lives, and their on-demand loading is a cost feature, not just an organizational one. The full migration procedure, the codemod patterns, the verification checklist, would be wasteful to send on every prompt. As a skill, it only enters the context when the description matches the task at hand. When you have a defined workflow whose procedure is worth reusing, that is your signal to reach for the create-skill command and scaffold it, rather than re-explaining the steps in chat every time. We are describing when to use it here. You already learned how it works in the earlier courses.

Agent Personas: A Scoped Role With a Tool Set

  • Use a persona when a workflow needs a consistent role, mindset, and a constrained set of tools
  • A focused tool set reduces unnecessary tool calls, and fewer tool calls mean fewer tokens
  • A defined role keeps the agent on task instead of exploring beyond the workflow
  • For the migration workflow: a "migration assistant" persona that reads release notes, edits affected modules, and runs tests
  • When to reach for it: scaffold one with /create-agent when the workflow recurs and benefits from a dedicated operator
Agent personas earn their keep when a workflow recurs and benefits from a dedicated operator with a fixed role and a constrained tool set. The constraint is the cost lever. An agent with access to every tool will explore with every tool. A migration assistant scoped to reading release notes, editing the affected modules, and running tests will not waste tokens spelunking through unrelated parts of the system. When a defined workflow fits that shape, reaching for the create-agent command to scaffold the persona is faster and more consistent than describing the role from scratch each time.

How Artifacts Cut Token Cost

    A2 --> A3[Many tool calls + large context]
    A3 --> A4[High credits per run]
  end
  subgraph With artifacts
    B1[Instructions carry standing rules] --> B2[Skill supplies the exact procedure]

This diagram makes the savings concrete. Without artifacts, the developer pays to re-explain context, the agent pays to explore and find intent, and the run accumulates many tool calls and a large context window. With artifacts, the standing rules are already present and lean, the skill supplies the exact procedure so no exploration is needed, and the persona limits the tools and scope so the agent acts directly. Same workflow, far fewer tokens. And critically, that reduction applies to every developer and every run, not just one.

Match the Workflow Signal to the Artifact

Signal in the workflow Artifact to build Command to reach for
A rule that should always apply Instructions file Author directly; keep it lean
A detailed procedure used only sometimes Skill /create-skill
A recurring role with a fixed tool set Agent persona /create-agent
A mix of all three Instructions + skill + persona together Author rules, then /create-skill and /create-agent
This is the decision table for the lab. Read the signal in your defined workflow, then build the matching artifact. A rule that always applies is an instruction file, and it is light enough to write directly. A detailed but occasional procedure is a skill, and create-skill is the fast way to scaffold it. A recurring role with a fixed tool set is a persona, and create-agent scaffolds it. Most mature workflows use all three together, which is exactly what you will build for the migration assistant in the hands-on lab.
Part 4

Token Optimization Techniques

Artifacts set up the structure. These techniques are the day-to-day habits that keep individual interactions efficient. Some of these were introduced in the billing course as developer habits. Here we connect them directly to tokens and to the artifacts we just built, so they reinforce each other rather than standing alone.

Right-Size the Context You Send

  • Reference the specific files or selection the task needs, not the entire workspace
  • A local change should start with a local scope, then widen only if required
  • Let scoped instructions supply standing rules instead of pasting them into the prompt
  • Let a skill supply the procedure instead of re-typing steps each time
  • Smaller, relevant context usually improves accuracy and lowers cost at the same time
Right-sizing context is the highest-frequency optimization there is, because it applies to nearly every interaction. The mistake is reaching for broad workspace context by default. Most tasks need a file or a selection, not the whole repository. And once you have built artifacts, you should lean on them: do not paste your coding standards into a prompt when a scoped instruction file already carries them, and do not retype a procedure when a skill supplies it. The artifacts and the habits are designed to work together.

Reuse Artifacts Instead of Re-Explaining

Re-Explain (Wasteful)

Each developer pastes the same standards, build steps, and migration process into chat, paying input tokens for the same knowledge every time.

Reuse (Efficient)

The knowledge lives in instructions and skills. It is captured once and applied automatically or on demand, at near-zero marginal cost per use.

  • Treat repeated explanations in chat as a signal to build or extend an artifact
  • If you explain the same thing twice, capture it once so you never pay to explain it again
This slide reframes a simple rule. The second time you find yourself explaining the same context to Copilot, stop and capture it. Re-explaining is paying full price for knowledge you already have. Reuse converts that into a one-time cost. This is also a practical trigger for artifact creation: your own repeated explanations are the clearest signal that a skill or an instruction file is overdue. The migration workflow is a perfect example. Nobody should be re-typing the upgrade steps in chat once the skill exists.

Practice Session Hygiene

  • One task, one chat. Start a fresh session when the task changes
  • Long sessions re-send accumulated history on every new message, inflating input tokens
  • A fresh, focused session is usually both cheaper and more accurate
  • If a tool supports transcript compaction, summarize and carry forward only the decisions that matter
  • Pair fresh sessions with scoped artifacts so each new chat still starts with the right standing context
Session length is a silent cost driver. Every message in a long conversation carries the entire prior transcript as input, so a sprawling session pays for its own history again and again. The fix is discipline: when the task changes, start a new chat. You lose nothing important because your instructions and skills still supply the standing context automatically. That combination, fresh sessions plus durable artifacts, gives you a clean slate without re-explaining anything.

Let Model Routing Match Cost to Complexity

  • Default to Auto so routing can reserve heavier reasoning models for tasks that need them
  • Straightforward steps in a workflow are routed to faster, lower-cost models
  • Override manually only when you know the task needs a specific model family
  • A well-defined workflow makes routing easier, because each step has a clear complexity profile
  • This builds on the model strategy from the billing course; here it is one lever among several
We covered model strategy in depth in the billing course, so this is a focused reminder rather than a deep dive. Default to Auto and let routing match model cost to task complexity. The connection to this course is that well-defined workflows make routing more effective. When a workflow has clear steps, each step has a predictable complexity, so the routing has better signal. Manual model selection should stay the exception, reserved for steps you know require a particular model family.

You cannot optimize what you cannot see, and the usage block is where you see it. Open Copilot's debug metadata for a single chat turn and you get an itemized receipt for that request. The headline to anchor on first is the ratio between input and output. Roughly sixty-two thousand tokens went into the model and only about two thousand came back. That is the whole reason this course focuses on context discipline rather than on writing shorter prompts or shorter answers: the input side is where the money goes and where you have the most leverage. Do not try to interpret every field on this slide. Use it to set up the two slides that follow, where we walk the fields one at a time and then explain the caching economics that make reuse so much cheaper than re-sending. In the lab you will open this exact view on a real request. Treat the menu path as a starting point and confirm it against your current VS Code version, since the Copilot tooling evolves quickly.

Anatomy of the Usage Block

Field What it represents Why it matters for cost
prompt_tokens All input sent: your message, chat history, attached files, instruction files, and every available tool schema The largest and most controllable cost; right-sizing context moves this number directly
completion_tokens The tokens the model generated in its reply Usually small next to input, so shortening answers rarely saves much
total_tokens Prompt plus completion The headline volume for the turn
cached_tokens The portion of the input served from cache Billed at a fraction of fresh input, so reuse is cheaper than re-sending
cache_creation_tokens New input written to the cache this turn Billed at a premium above standard input
copilotUsage The turn rolled up into AI Credits The single number engineering and finance can both track
Now walk the fields one at a time. Prompt tokens is the big one, and it is broader than people expect. It is not just the message you typed. It includes the running chat history, every file you attached, the instruction files in scope, and the JSON schema for every tool the agent can call. That is why an over-scoped agent or a bloated instruction file shows up here as real cost on every single turn. Completion tokens is the reply, and it is usually small by comparison, which is why trying to save money by asking for terse answers misses the point. Total tokens is just the sum. The two cache fields are the interesting part and we give them their own slide next, but the short version is that cached tokens are cheap reused input and cache-creation tokens are the premium you pay to write new entries. Finally, copilotUsage is the line that translates all of this into AI Credits, and it is the number worth tracking over time because both engineering and finance understand it.

Why Caching Changes the Math

  • In this turn, 57,198 of 61,729 prompt tokens (about 93%) were served from cache and billed at a fraction of fresh input
  • Cache hits are cheap; cache writes (cache_creation_tokens) are billed at a premium above standard input
  • Caching is calculated at the model provider, so an Anthropic model and an OpenAI model decide what is cacheable differently
  • You do not control caching directly. It is a provider optimization, not a setting you tune
  • What you do control is prompt_tokens: shrink the input and the whole turn gets cheaper, cache or no cache
  • The practical levers are summarization and compaction (for example a /compact command) to carry forward only what matters instead of the full history
This slide is the payoff, but it comes with an important clarification so nobody leaves with the wrong mental model. Caching is real and it matters: on this turn over ninety percent of the input was served from cache at a fraction of the price of fresh input. But caching is decided at the model provider, not by you. Anthropic and OpenAI implement prompt caching differently, with their own rules for what is eligible, how long entries live, and how hits and writes are priced. Treat the cache fields as something you observe and benefit from, not a dial you turn. The dial you actually control is the input itself. Prompt tokens is the lever. If you send less, the turn is cheaper whether or not the provider caches it, and you also give the provider a smaller, more stable surface to cache. The concrete tools for shrinking input are summarization and compaction. Many tools expose a compact command, often slash compact, that collapses a long transcript down to the decisions and state that still matter, so the next turn does not re-send the entire history. Tie this back to the habits from earlier in the course: lean instruction files, right-sized context, fresh focused sessions, and compaction when a session must run long. Those are the things you can actually do. Caching is the provider quietly rewarding you for doing them.

Optimization Has a Floor: Don't Starve the Model

Cutting tokens is only a win up to a point. Strip out too much and the model has to go find what you withheld, which costs more than you saved.

Too Little Context

  • The model lacks what it needs to answer
  • It compensates with extra tool calls to read files, search, and re-discover state
  • Each call returns output that becomes new input tokens on the next turn
  • More round-trips, more latency, and often a worse answer

Right-Sized Context

  • The model has the essentials up front and answers in fewer turns
  • Curated context beats both extremes: not the whole repo, not nothing
  • Durable artifacts supply standing context so each turn starts informed
  • Fewer discovery loops means fewer tokens overall

Optimize for the cheapest correct answer, not the smallest possible prompt.

This is the most important nuance in the whole optimization section, and it is the one teams get wrong once they get excited about cutting tokens. Token reduction is not a goal in itself. It has a floor. If you strip context too aggressively, the model does not simply fail quietly. It goes looking for what you withheld. In an agentic setting that means tool calls: it reads files, runs searches, lists directories, and re-discovers the state you could have just handed it. Every one of those tool results comes back as input tokens on the following turn, and the model may need several of these loops before it has enough to answer. So the under-context request that looked cheap on paper can end up more expensive than a well-formed one, and slower, and lower quality on top of that. The mental model to leave them with is a U-shaped cost curve. On the left, too much context, you pay for tokens the model did not need. On the right, too little context, you pay for the model to go re-acquire it through tools. The minimum sits in the middle: curated, right-sized context. This is exactly why the artifact discipline from earlier matters. A good instruction file or skill supplies the standing context the model would otherwise have to hunt for, so you land in that low-cost middle by default. The slogan is the takeaway: optimize for the cheapest correct answer, not the smallest possible prompt.

Anti-Patterns That Quietly Burn Credits

Anti-pattern Why it wastes tokens The fix
Over-scoped agents Access to every tool invites unnecessary exploration Constrain the persona to the tools the workflow needs
Skills that never trigger Vague descriptions mean the skill never loads, so work is redone Write precise, task-matching descriptions
Instruction bloat Long instruction files are re-sent on every request in scope Keep instructions lean; move procedures into skills
Redundant context Pasting files the agent can already reference duplicates tokens Reference, do not paste; right-size the scope
Mega-sessions Long transcripts re-send history on every message One task, one chat; start fresh when the task changes
These are the failure modes you will actually encounter, and most of them are quiet. Nobody notices a slightly bloated instruction file or a session that ran long, but they add up across a team and a month. The pattern to internalize is that good artifacts can be done badly. An over-scoped agent or a never-triggering skill can cost more than having no artifact at all. Each fix here ties directly back to the discipline from Parts 2 and 3: define the workflow tightly, then build lean, well-targeted artifacts.

Measure Workflow Efficiency, Not Just Total Spend

  • Track credits per workflow run, not only the organization-wide total
  • Compare before and after you introduce artifacts for a workflow
  • Watch model mix and tool-call counts as leading indicators of cost
  • Celebrate high-value spend; coach on high-cost, low-value patterns
  • Feed what you learn back into the workflow definitions and artifacts
Total spend is a lagging, blunt metric. Efficiency per workflow is what tells you whether your optimization work is paying off. The most powerful measurement is a simple before-and-after: capture the typical credit cost of a workflow, introduce the artifacts, then measure again. That number makes the value of this entire approach concrete to both engineering and finance. Model mix and tool-call counts are useful leading indicators. And the loop closes by feeding insights back into the workflow definitions, which is the operationalize step we turn to next.
Part 5

Operationalize Across the Team

We have the discipline, the artifacts, and the techniques. The final step is making this a repeatable organizational practice rather than a one-time cleanup. This part gives you a loop, a 30-day plan, and a short checklist so the work sticks after the meeting ends.

The Workflow-First Optimization Loop

graph LR
  A[Define the workflow] --> B[Build supporting artifacts]
  B --> C[Run with right-sized context]
  C --> D[Measure credits per run]
  D --> E[Refine workflow and artifacts]
  E --> A
Optimization is not a project with an end date. It is a loop. Define the workflow, build the artifacts, run it efficiently, measure the cost per run, and refine. Each lap lowers the baseline a little more and captures a little more knowledge. The reason we draw it as a cycle is to set the right expectation: you are establishing a practice, not checking a box. Teams that internalize this loop steadily drive down their cost per outcome while increasing the share of work AI handles well.

A 30-Day Workflow Optimization Plan

Window What to do
Week 1 List recurring tasks and pick two or three workflows AI should own using the should/should-not filter
Week 2 Define those workflows in detail: inputs, steps, guardrails, and a definition of done
Week 3 Build supporting artifacts: lean instructions, plus skills and personas via /create-skill and /create-agent
Week 4 Measure credits per run before and after, document wins, and feed lessons back into the definitions
A 30-day plan keeps this from becoming an abstract good intention. Week 1 is selection: do not try to automate everything, pick two or three high-value workflows. Week 2 is definition, the discipline from Part 2. Week 3 is artifact building, where the create commands earn their keep. Week 4 is measurement and feedback, which proves the value and starts the loop. After one month you will have a small set of well-optimized workflows and, just as importantly, a repeatable method for the next ones.

Team Optimization Checklist

  • Define the workflow before choosing tools or models
  • Use the should/should-not filter to decide what AI owns
