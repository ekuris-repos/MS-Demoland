Part 1

MCP Recap

Before we dive into advanced topics, let's do a quick refresher on the MCP fundamentals you learned in the Beginner course.

What Is MCP?

  • Model Context Protocol is an open standard for connecting AI models to external tools and data sources
  • MCP provides a structured, type-safe interface between clients (like Copilot) and servers (your tools)
  • Servers expose tools (actions the model can invoke), resources (data it can read), prompts (reusable templates), sampling (server-initiated LLM requests), and elicitation (server-initiated user input)
  • The protocol handles discovery, invocation, and response formatting automatically
  • Think of MCP as a universal adapter between AI and your organization's systems
MCP is the Model Context Protocol, an open standard originally created by Anthropic and now widely adopted across the AI tooling ecosystem. It defines how AI clients, such as GitHub Copilot, VS Code, or other model hosts, communicate with external servers that provide tools, resources, prompts, sampling, and elicitation. The key benefit is standardization: instead of writing custom integrations for every AI client, you build one MCP server and any compatible client can use it. This is the foundation everything else in this course builds on.

Why MCP Matters at Enterprise Scale

  • Standardization: One protocol replaces dozens of bespoke integrations
  • Discoverability: Clients automatically discover available tools and their schemas
  • Security boundary: The server controls what data and actions are exposed to the model
  • Composability: Multiple servers can be combined to build complex workflows
  • Governance: Centralized control over which tools are approved and how they are used
At enterprise scale, MCP solves several critical problems. First, it replaces ad-hoc integrations with a single, well-defined protocol. Second, it provides a natural security boundary; the MCP server decides exactly what data to expose and what actions to allow. Third, because tools are self-describing with JSON schemas, clients can discover and validate them automatically. Finally, composability means you can combine multiple MCP servers into sophisticated workflows without modifying any individual server. These properties make MCP uniquely suited for enterprise environments where governance, security, and maintainability are paramount.
Part 2

Architecture Deep Dive

Now let's go deeper into the architecture of MCP. Understanding the client-server model and transport layers is essential for making good design decisions when building production servers.

The Client-Server Model

  • MCP Host: The application the user interacts with (VS Code, GitHub.com, IDE)
  • MCP Client: A protocol client embedded in the host that manages server connections
  • MCP Server: A lightweight process that exposes tools, resources, and prompts
  • Each client maintains a 1:1 connection with a single server
  • A host can manage multiple clients, each connected to a different server
  • Communication uses JSON-RPC 2.0 as the message format
The MCP architecture has three layers. The host is the user-facing application, such as VS Code or GitHub Copilot Chat. Inside the host, one or more MCP clients are created, each maintaining a one-to-one connection with an MCP server. The server is a separate process that exposes capabilities through the protocol. All communication uses JSON-RPC 2.0, which provides a standardized request-response format with support for notifications. This separation of concerns means servers can be developed, tested, and deployed independently from clients.

Transport Layers

stdio

Server runs as a local subprocess. Communication happens over standard input/output. Best for local development and IDE integrations where the server runs on the same machine.

Streamable HTTP

Server runs as an HTTP endpoint. Supports both request-response and server-sent events for streaming. The recommended transport for remote and production deployments.

MCP supports two primary transport mechanisms. The stdio transport runs the server as a local child process, communicating through standard input and output streams. This is the simplest option and works well for local development in VS Code. The Streamable HTTP transport, which supersedes the older HTTP+SSE transport, runs the server as an HTTP endpoint. It uses standard HTTP POST for client-to-server messages and server-sent events for server-to-client streaming. Streamable HTTP is the recommended choice for remote deployments, shared team servers, and production environments. When planning your architecture, start with stdio for development and target Streamable HTTP for production.

Protocol Lifecycle

  • Initialize: Client sends capabilities; server responds with its capabilities and supported features
  • Discover: Client requests the list of available tools, resources, and prompts
  • Invoke: Client calls a tool with arguments; server validates, executes, and returns results
  • Notify: Either side can send notifications (e.g., tool list changed, progress updates)
  • Shutdown: Graceful disconnection with cleanup
Every MCP session follows a predictable lifecycle. The client initiates with an initialize request, declaring its capabilities. The server responds with its own capabilities, including which protocol features it supports. After initialization, the client discovers available tools, resources, and prompts by querying the server. During the active session, the client can invoke tools by sending requests with typed arguments. The server validates those arguments against the tool's schema, executes the operation, and returns structured results. Both sides can send notifications for events like tool list changes or progress updates. Finally, either side can initiate a graceful shutdown. Understanding this lifecycle helps you design servers that handle each phase correctly.
Part 3

Building Production MCP Servers

With the architecture understood, let's get practical. This section covers language choices, SDK options, and the project structure you need for a production-grade MCP server.

Language & SDK Options

TypeScript / JavaScript

Official @modelcontextprotocol/sdk package. Mature ecosystem, excellent for teams already using Node.js. Supports both stdio and Streamable HTTP out of the box.

Python

Official mcp Python SDK. Great for data-heavy integrations, ML pipelines, and teams with Python expertise. Async-first design using asyncio.

C# / .NET

Official ModelContextProtocol NuGet package. Ideal for enterprises with existing .NET infrastructure and Azure deployments. Integrates with ASP.NET Core.

Other Languages

Community SDKs available for Java, Go, Rust, and more. The protocol is language-agnostic; any language that can handle JSON-RPC over stdio or HTTP can implement a server.

You can build MCP servers in any language, but three have official SDK support. The TypeScript SDK is the most mature and is maintained by the MCP specification team. It provides decorators and helper functions that make server creation straightforward. The Python SDK uses an async-first design and is excellent for data science and ML integrations. The C# SDK integrates with ASP.NET Core and is the natural choice for .NET shops, especially those deploying to Azure. Community SDKs exist for Java, Go, Rust, and other languages. Choose based on your team's expertise and your existing infrastructure.

TypeScript Server Structure

import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import { z } from "zod";

const server = new McpServer({
  name: "inventory-service",
  version: "1.0.0",
});

server.tool(
  "lookup_product",
  "Find a product by SKU in the inventory system",
  { sku: z.string().describe("The product SKU to look up") },
  async ({ sku }) => {
    const product = await inventoryAPI.getProduct(sku);
    return { content: [{ type: "text", text: JSON.stringify(product) }] };
  }
);

const transport = new StdioServerTransport();
await server.connect(transport);
Here is a minimal TypeScript MCP server. We import the McpServer class and the stdio transport from the official SDK. We create a server instance with a name and version, then register a tool using the server.tool method. The tool has a name, description, a Zod schema for input validation, and an async handler function. The handler calls your internal API, in this case an inventory service, and returns structured content. Finally, we connect the server to the stdio transport. This pattern scales well; you simply add more tool registrations as your server grows. Notice how Zod handles input validation automatically, rejecting malformed requests before they reach your handler.

Tool Design Patterns

  • Single responsibility: Each tool should do one thing well. Prefer many focused tools over one monolithic tool
  • Descriptive naming: Use verb_noun format (e.g., get_ticket, create_incident, search_wiki)
  • Rich descriptions: The tool description is the model's primary guide; be specific about what it does and when to use it
  • Typed schemas: Define every parameter with types, descriptions, and constraints. The schema drives both validation and model understanding
  • Idempotent reads: GET-style tools should be safe to call multiple times without side effects
  • Explicit mutations: Tools that modify data should clearly state that in their description
Good tool design is critical because the AI model uses your tool names, descriptions, and schemas to decide when and how to call them. Name your tools with a clear verb-noun pattern so the model can infer intent. Write descriptions that are specific; instead of "query the database," say "search the product catalog by name, category, or price range." Define every parameter with a type, description, and any constraints like minimum values or allowed enum values. Make read operations idempotent so the model can call them freely without concern about side effects. For tools that modify state, be explicit in the description so the model and user both understand the consequences of invocation.

Error Handling & Validation

server.tool(
  "get_customer",
  "Retrieve customer details by their unique ID",
  { customer_id: z.string().uuid().describe("Customer UUID") },
  async ({ customer_id }) => {
    try {
      const customer = await customerAPI.findById(customer_id);
      if (!customer) {
        return {
          content: [{ type: "text", text: "No customer found with that ID." }],
          isError: true,
        };
      }
      return { content: [{ type: "text", text: JSON.stringify(customer) }] };
    } catch (err) {
      return {
        content: [{ type: "text", text: `Service error: ${err.message}` }],
        isError: true,
      };
    }
  }
);
Robust error handling is essential for production MCP servers. The schema layer provides your first line of defense; Zod validates inputs before your handler ever runs, rejecting invalid UUIDs, missing fields, or out-of-range values automatically. Inside your handler, use try-catch blocks to handle API failures gracefully. When something goes wrong, return a response with isError set to true and a clear, descriptive message. This tells the AI model that the operation failed and provides enough context for it to either retry with corrected inputs or explain the failure to the user. Never let unhandled exceptions crash your server; always catch and return structured errors.
Part 4

Connecting to Internal APIs

Now we get to the core value proposition of enterprise MCP: wrapping your organization's existing APIs so that Copilot can interact with them through natural language.

Wrapping REST APIs as MCP Tools

server.tool(
  "search_incidents",
  "Search ServiceNow incidents by state and priority",
  {
    state: z.enum(["new", "in_progress", "resolved", "closed"])
      .describe("Incident state filter"),
    priority: z.number().min(1).max(5).optional()
      .describe("Priority level, 1 = critical, 5 = planning"),
  },
  async ({ state, priority }) => {
    const params = new URLSearchParams({ state });
    if (priority) params.set("priority", String(priority));
    const res = await fetch(
      `${SERVICENOW_BASE}/api/now/table/incident?${params}`,
      { headers: { Authorization: `Bearer ${await getToken()}` } }
    );
    const data = await res.json();
    return { content: [{ type: "text", text: JSON.stringify(data.result) }] };
  }
);
This example shows how to wrap a REST API, in this case ServiceNow, as an MCP tool. The pattern is straightforward: define a tool with a descriptive name and schema, then implement a handler that translates the MCP request into an HTTP call to your internal API. Notice that the schema uses enums for the state parameter, which constrains the model's choices to valid values. The priority parameter is optional with min/max bounds. The handler constructs the URL, adds authentication headers, makes the request, and returns the response. This pattern works for any REST API: map the relevant endpoints to tools and let the schema guide the model's usage.

Wrapping GraphQL Services

server.tool(
  "get_team_members",
  "List members of an engineering team from the internal directory",
  { team_name: z.string().describe("Team name, e.g. 'platform-eng'") },
  async ({ team_name }) => {
    const query = `
      query ($team: String!) {
        team(name: $team) {
          members { name, email, role, location }
        }
      }`;
    const res = await fetch(DIRECTORY_GRAPHQL_URL, {
      method: "POST",
      headers: { "Content-Type": "application/json",
                 Authorization: `Bearer ${await getToken()}` },
      body: JSON.stringify({ query, variables: { team: team_name } }),
    });
    const { data } = await res.json();
    return { content: [{ type: "text", text: JSON.stringify(data.team) }] };
  }
);
GraphQL APIs follow the same wrapping pattern with one key difference: instead of constructing URL parameters, you build a GraphQL query string. The tool's schema captures the high-level parameters the model needs to provide, and your handler translates those into the appropriate GraphQL query and variables. This approach hides the complexity of the GraphQL schema from the model, giving it a simple, focused interface. For large GraphQL APIs, create multiple MCP tools that each handle a specific query or mutation rather than exposing the full GraphQL interface. This keeps each tool's purpose clear and makes it easier for the model to select the right one.
Part 5

Database Integration

Connecting Copilot to internal databases is one of the most powerful MCP use cases, but it requires careful attention to safety and security. Let's cover the patterns that make this work in production.

Safe Database Access Patterns

  • Read-only connections: Use a database user with SELECT-only permissions. Never grant INSERT, UPDATE, or DELETE to the MCP connection
  • Parameterized queries: Always use parameterized queries to prevent SQL injection. Never interpolate user input into query strings
  • Row limits: Enforce maximum row counts on all queries to prevent accidental data dumps
  • Schema exposure: Expose table and column names as MCP resources so the model understands the data structure
  • Audit logging: Log every query executed through the MCP server with the user context
Database integration requires a security-first approach. Start by creating a dedicated database user with read-only permissions; this is your most important safeguard. Always use parameterized queries, never string concatenation, to prevent SQL injection attacks. Enforce row limits on every query so a broad query does not accidentally return millions of rows and overwhelm the response. Expose the database schema as MCP resources so the model can discover available tables and columns. Finally, log every query with the requesting user's identity for audit purposes. These patterns ensure that database access through MCP is both useful and safe.

Database Query Tool

server.tool(
  "query_sales_data",
  "Query the sales database by region and date range. Read-only access.",
  {
    region: z.string().describe("Sales region code, e.g. 'US-WEST'"),
    start_date: z.string().describe("Start date in YYYY-MM-DD format"),
    end_date: z.string().describe("End date in YYYY-MM-DD format"),
    limit: z.number().max(100).default(25)
      .describe("Maximum rows to return, up to 100"),
  },
  async ({ region, start_date, end_date, limit }) => {
    const result = await db.query(
      `SELECT order_id, product, quantity, revenue
       FROM sales
       WHERE region = $1 AND order_date BETWEEN $2 AND $3
       ORDER BY order_date DESC
       LIMIT $4`,
      [region, start_date, end_date, limit]
    );
    return { content: [{ type: "text", text: JSON.stringify(result.rows) }] };
  }
);
This example demonstrates a safe database query tool. Notice several key safety features. The schema constrains inputs to specific, well-defined parameters; the model cannot write arbitrary SQL. The query uses parameterized placeholders ($1, $2, $3, $4) instead of string interpolation, preventing injection attacks. A hard limit of 100 rows is enforced at the schema level, with a default of 25. The database connection itself should use a read-only user, providing defense in depth. This pattern lets the model query real business data while maintaining strict security boundaries.
Part 6

Authentication & Authorization

Authentication is often the most complex part of enterprise MCP server development. Let's cover the patterns and options available to you.

Authentication Strategies

API Keys

Simplest approach. Store keys in environment variables or a secret manager. Best for internal services with static credentials. Rotate keys regularly.

OAuth 2.1 Flows

Full OAuth 2.1 support with authorization code and client credentials flows. OAuth 2.1 mandates PKCE and removes the implicit flow for stronger security. MCP supports OAuth 2.1 natively in the protocol.

Managed Identities

Azure Managed Identity and workload identity for cloud-hosted servers. No credentials in code or config. The platform handles token acquisition and rotation automatically.

Service-to-Service Tokens

Use client credentials flow for backend-to-backend communication. The MCP server authenticates as itself to downstream APIs using certificates or shared secrets.

There are four primary authentication strategies for MCP servers, and you will often combine more than one in a single server. API keys are the simplest approach and work well for internal services where a static credential is acceptable. OAuth 2.1 is necessary when your MCP tools need to access resources scoped to the current user; the MCP protocol includes native OAuth 2.1 support with authorization code flows. Azure Managed Identities eliminate credentials entirely by letting the platform handle token acquisition and rotation. For backend-to-backend calls where no user context is needed, use the OAuth client credentials flow. Choose the strategy that matches your security requirements and the APIs you are connecting to.

OAuth 2.1 in MCP

  • MCP defines a built-in OAuth flow for remote servers using Streamable HTTP transport
  • The server advertises its OAuth metadata at a well-known endpoint
  • The client handles the browser redirect, authorization code exchange, and token storage
  • Tokens are passed automatically with subsequent MCP requests
  • PKCE (Proof Key for Code Exchange) is mandatory under OAuth 2.1, not optional
  • Enables user-scoped access: each developer sees only what they are authorized to see
The MCP protocol includes first-class support for OAuth 2.1 when using the Streamable HTTP transport. The flow works as follows: the remote MCP server exposes its OAuth metadata at a well-known URL. When a client connects and the server requires authentication, the client opens a browser window for the user to authorize. After authorization, the client exchanges the code for tokens and stores them securely. All subsequent MCP requests include the access token automatically. OAuth 2.1 mandates PKCE, which is especially important when the client is a desktop application like VS Code. The result is user-scoped access, where each developer's MCP requests carry their own identity and permissions.
Part 7

Remote MCP Servers

Local stdio servers are great for development, but production workloads typically require remote servers that can be shared across teams, scaled dynamically, and managed centrally.

Deployment Strategies

Azure Container Apps

Containerize your MCP server and deploy to Azure Container Apps. Supports auto-scaling, managed identity, custom domains, and built-in HTTPS. Ideal for team-shared servers.

Azure App Service

Deploy as a standard web application. Supports Node.js, Python, and .NET natively. Use deployment slots for zero-downtime updates and built-in auth via Easy Auth.

Docker Compose

Run multiple MCP servers behind a reverse proxy for development and staging environments. Simple to configure and test locally before cloud deployment.

Azure Functions

Serverless option for lightweight MCP servers. Note: MCP's Streamable HTTP transport requires persistent HTTP endpoints with SSE streaming, which may conflict with Consumption plan timeouts and cold starts. A Premium or Dedicated plan is more appropriate for production MCP servers.

There are several viable deployment strategies for remote MCP servers. Azure Container Apps is often the best starting point because it provides auto-scaling, managed identity, and HTTPS out of the box. Azure App Service is another strong option, especially for teams already using it for web applications. For development and staging, Docker Compose lets you run multiple MCP servers locally behind a reverse proxy to simulate a production topology. Azure Functions can host MCP servers, but the Streamable HTTP transport's need for persistent connections and SSE streaming may conflict with Consumption plan timeouts and cold starts; a Premium or Dedicated Functions plan is a better fit for production workloads. The right choice depends on your traffic patterns, existing infrastructure, and operational preferences.

MCP in GitHub Copilot Extensions

  • Copilot Extensions use their own tool-registration mechanism, separate from MCP. MCP servers are configured at the editor level via .vscode/mcp.json or user settings. These are complementary integration surfaces
  • Extensions register tools that Copilot can discover and invoke across surfaces (VS Code, GitHub.com, CLI)
  • MCP provides the tool layer, while the extension framework handles distribution, authentication, and user management
  • Organizations can build private extensions with MCP tools available only to their developers
  • Public extensions on the GitHub Marketplace can also use MCP under the hood
GitHub Copilot Extensions and MCP servers are complementary but distinct integration surfaces. Copilot Extensions use their own tool-registration mechanism and are distributed through the GitHub Marketplace with built-in authentication and user management. MCP servers are configured at the editor level via .vscode/mcp.json or user settings and use the Model Context Protocol for communication. While both provide tools to Copilot, they operate through different channels. Organizations can create private extensions that are only available to their developers, making this a viable path for enterprise-internal tool distribution.
Part 8

Enterprise Workflows & Governance

Now we move into the organizational and operational aspects of running MCP at enterprise scale: chaining tools, governance, monitoring, and performance optimization.

Chaining MCP Tools

  • The AI model can call multiple tools in sequence to accomplish complex tasks
  • Example workflow: search_tickets → get_ticket_details → create_branch → update_ticket_status
  • Each tool should return enough context for the model to decide the next step
  • Design tools as composable building blocks, not monolithic operations
  • The model handles orchestration; you provide the primitives
  • Use clear descriptions so the model understands the logical order of operations
One of MCP's most powerful capabilities is tool chaining, where the model calls multiple tools in sequence to accomplish a complex workflow. For example, a developer might ask Copilot to "find the highest priority bug and start working on it." Copilot would call search_tickets to find bugs, get_ticket_details to read the top result, create_branch to set up a working branch, and update_ticket_status to mark it as in progress. You do not need to build this orchestration logic yourself. Instead, design each tool as a focused, composable building block and let the model figure out the sequence. The key is returning rich enough context from each tool so the model can make informed decisions about what to do next.

Enterprise Governance

  • Tool approval process: Establish a review workflow before any MCP tool reaches production
  • Allowlisting: Use VS Code settings and GitHub org policies to control which MCP servers developers can use
  • Security reviews: Evaluate data exposure, authentication strength, and blast radius for each tool
  • Role-based access: Restrict sensitive tools to specific teams or roles using server-side authorization
  • Change management: Version your MCP servers and use staged rollouts for updates
  • Documentation: Maintain a tool catalog so teams know what is available and how to use it
Enterprise governance is what separates a hobby project from a production-grade deployment. Start by establishing a tool approval process; every MCP tool should go through a security review before it reaches production, evaluating data exposure, authentication requirements, and the blast radius of potential misuse. Use allowlisting at the organization level to control which MCP servers developers can connect to. Implement role-based access on the server side so that tools like "delete_deployment" are only available to users with appropriate permissions. Version your servers and use staged rollouts, deploying to a canary group before rolling out organization-wide. Finally, maintain a tool catalog so developers can discover approved tools and understand their capabilities.

Monitoring & Logging

  • Structured logging: Log every tool invocation with tool name, parameters (sanitized), duration, and result status
  • Health endpoints: Expose /health and /ready endpoints for orchestrators and load balancers
  • Metrics: Track invocation counts, error rates, latency percentiles, and active connections
  • Alerting: Set thresholds for error rate spikes, latency degradation, and connection failures
  • Distributed tracing: Propagate trace IDs from the MCP client through to downstream API calls
  • Usage analytics: Understand which tools are most used, by whom, and for what workflows
Monitoring is essential for operating MCP servers reliably. Implement structured logging that captures every tool invocation with the tool name, sanitized input parameters, execution duration, and result status. Make sure to sanitize any sensitive data like tokens or personal information before it reaches your logs. Expose health and readiness endpoints so container orchestrators and load balancers can route traffic correctly. Track metrics such as invocation counts, error rates, and latency percentiles using your existing observability stack. Set up alerts for error rate spikes and latency degradation so your team can respond before users are impacted. Usage analytics help you understand adoption, identify popular tools, and plan capacity.

Performance Optimization

  • Response caching: Cache frequently requested, slowly changing data (e.g., org charts, wiki pages) with appropriate TTLs
  • Connection pooling: Reuse database and HTTP connections instead of creating new ones per request
  • Rate limiting: Protect downstream services from being overwhelmed by MCP tool calls
  • Pagination: Return bounded result sets and provide tools for fetching additional pages
  • Async operations: For long-running tasks, return immediately with a task ID and provide a separate status-check tool
  • Response shaping: Return only the fields the model needs, not the full API response
Performance optimization prevents your MCP server from becoming a bottleneck. Start with response caching for data that changes infrequently, such as organization charts, team rosters, or documentation pages. Set appropriate TTLs based on how often the data actually changes. Use connection pooling for databases and HTTP clients so you are not paying connection setup costs on every request. Implement rate limiting to protect downstream services; an AI model can generate many tool calls in quick succession, and your backend APIs may not be designed for that access pattern. Return bounded result sets with pagination support, and shape your responses to include only the fields the model needs. For long-running operations, use an async pattern where the tool returns a task ID immediately, and a separate tool lets the model check status.

Real-World Integration Scenarios

Let's look at concrete, real-world scenarios where organizations are using MCP to connect Copilot to their enterprise systems.

Scenario: Issue Tracking Integration

Jira Integration

Tools: search_issues, get_issue, create_issue, transition_issue, add_comment. Developers query, create, and update Jira issues directly from Copilot Chat without switching context.

ServiceNow Integration

Tools: search_incidents, get_incident, create_change_request, check_approval_status. On-call engineers use Copilot to triage incidents and initiate change requests during incident response.

Issue tracking integrations are among the most valuable MCP use cases. A Jira MCP server typically exposes five to seven tools covering search, read, create, transition, and comment operations. Developers can ask Copilot to find their assigned bugs, read the details, create a branch named after the issue, and update the issue status, all without leaving VS Code. A ServiceNow MCP server serves a different workflow: on-call engineers can search incidents, read details, and create change requests through natural language during an incident response when speed matters and context-switching is costly. Both integrations follow the same pattern of wrapping REST APIs as focused, composable MCP tools.

Scenario: CI/CD Pipeline Integration

  • check_pipeline_status: Query the status of the latest CI/CD run for a branch or commit
  • get_test_results: Retrieve test output, failure messages, and coverage metrics
  • trigger_deployment: Initiate a deployment to a specified environment (with confirmation)
  • get_deployment_history: List recent deployments with status, timestamp, and deployer
  • rollback_deployment: Trigger a rollback to a previous known-good version

Enables developers to monitor, diagnose, and act on CI/CD status without leaving their editor.

CI/CD pipeline integration brings your deployment workflow into Copilot Chat. A developer can ask "what's the status of my build?" and Copilot calls check_pipeline_status. If a test failed, the developer asks "show me the failures" and Copilot calls get_test_results with the relevant run ID. After fixing the issue and pushing, the developer can ask Copilot to trigger a staging deployment. For mutation tools like trigger_deployment and rollback_deployment, implement confirmation prompts in your tool descriptions and consider requiring explicit approval flags in the tool schema. This integration is especially valuable for teams with complex multi-stage pipelines where navigating the CI/CD web interface is time-consuming.

Scenario: Internal Knowledge Base

  • search_wiki: Full-text search across Confluence, SharePoint, or custom wiki platforms
  • get_page: Retrieve the full content of a specific wiki page
  • search_runbooks: Find operational runbooks by service name or incident type
  • get_api_docs: Retrieve API documentation for internal services
  • Transforms tribal knowledge into instantly accessible context for every developer
  • Pairs well with MCP resources for providing static reference data
Internal knowledge base integration is one of the highest-impact MCP use cases because it addresses a universal problem: tribal knowledge is hard to find. A wiki search tool lets developers ask Copilot questions like "how do we configure the payment gateway in staging?" and get answers drawn from your actual internal documentation. Runbook search helps on-call engineers find the right procedure during an incident. API documentation tools let developers discover and understand internal service interfaces without hunting through wikis. The key to making this work well is good search relevance; consider using a search backend that supports semantic or hybrid search to go beyond simple keyword matching.

Building a Complete Integration

  • 1. Identify the workflow: What does the developer need to accomplish? What systems are involved?
  • 2. Map the tools: Break the workflow into discrete, composable tool operations
  • 3. Design schemas: Define precise input types with descriptions and constraints
  • 4. Implement handlers: Write the API calls, error handling, and response formatting
  • 5. Add authentication: Choose and implement the right auth strategy for each downstream service
  • 6. Test with Copilot: Verify that the model selects the right tools and chains them correctly
