Copilot Billing - From PRUs to Usage-Based Billing

Prepare your team for GitHub AI Credits, budget guardrails, and lower-cost workflows that keep Copilot productive without surprise spend

Welcome to this course on GitHub Copilot's transition from premium request units, or PRUs, to usage-based billing. The goal is not just to understand the billing model. The goal is to help your organization keep Copilot useful, fast, and financially predictable.

Part 1

Why GitHub Changed Billing

The right way to explain the transition is to start with why GitHub made it in the first place. If your team only hears, "billing changed," they will assume it is a pure finance story. It is not. It is a product shape story, because Copilot has become much more agentic, multi-step, and compute-intensive than it was when PRUs were introduced.

Why Move Away From PRUs?

  • Copilot has evolved from a quick in-editor assistant into an agentic platform that can work across whole repositories
  • A short chat question and a long autonomous coding session should not cost the same
  • PRUs masked real differences in compute intensity, model choice, and session length
  • Usage-based billing aligns price to actual token consumption and the model used
  • GitHub positions the change as a way to improve long-term reliability and reduce the need for blunt gating
The official blog is explicit about the motivation. Copilot is not the same product it was a year earlier. Agentic usage is becoming the default, and that means longer runs, more tool use, more context, and higher inference cost. PRUs were too coarse for that reality. Usage-based billing moves the system closer to actual cost drivers: tokens and model selection. For your organization, that means the economics now reward disciplined task shaping instead of treating all advanced usage as equivalent.

PRUs vs. AI Credits

Dimension PRUs Usage-Based Billing
Billing unit Premium request units GitHub AI Credits
Primary cost signal Request count with multipliers Input, output, and cached tokens by model
Best fit Simple request metering Mixed workloads from quick chat to long agent runs
Admin control Limited allowance management Pooled credits, budgets, alerts, and hard stops
After allowance is exhausted Fallback could apply in some cases Usage is charged or blocked based on budget policy
This is the simplest comparison slide in the deck, and it is one your audience will likely photograph. Under PRUs, the mental model was request counting plus model multipliers. Under usage-based billing, the mental model is credits, tokens, and model rates. That change matters because it gives admins much better budget control, but it also means engineers need a better instinct for scope, verbosity, and model selection. The metering is now closer to how the work is actually done.

What Stays the Same, What Changes

Stays the Same

  • Base seat prices do not change
  • Code completions stay included
  • Next Edit Suggestions stay included
  • Teams still manage model access through policy

Changes

  • Usage is priced from token consumption and model rates
  • Budgets can be set at enterprise, org, cost center, and user levels
  • No automatic fallback to a cheaper model when budget is exhausted
  • Copilot code review also consumes GitHub Actions minutes
This slide helps calm the room. GitHub did not turn every Copilot interaction into a surprise bill, and they did not start charging for completions. The core seat pricing remains the same. The important changes are where the variable spend now shows up, how precisely it can be managed, and the fact that code review now has a second meter in GitHub Actions minutes. That last point is easy to miss and worth repeating to platform and finance stakeholders.

Migration Timing and Transitional Details

  • June 1, 2026: monthly Copilot plans move to usage-based billing
  • Copilot Pro and Pro+ annual subscribers stay on request-based billing until renewal, then transition
  • Existing Business and Enterprise customers receive promotional included usage for June, July, and August
  • GitHub launched a preview bill experience ahead of the cutover so customers can estimate spend
  • Now is the time to publish budget policy, model guidance, and team expectations before the first full billing cycle closes
The timing matters because many organizations are in a transition window right now. Monthly plans are straightforward. Annual plans create a staggered edge case, especially for individuals. For businesses and enterprises, GitHub intentionally added promotional included usage to soften the landing for the first three months. That gives teams a short runway to learn what their normal usage pattern looks like before they tighten or expand budgets.
Part 2

How Usage-Based Billing Works

Now that we know why the model changed, we can talk about mechanics. This section gives the mental model your teams need in order to reason about spend before they open the bill page. The key terms are AI Credits, tokens, model rates, and included allowances.

What Is a GitHub AI Credit?

  • GitHub AI Credits are the new billing unit for Copilot usage
  • 1 AI credit = $0.01 USD
  • Each interaction consumes input, output, and sometimes cached tokens
  • The total token usage is converted into AI Credits using the published rate for the model used
  • A fast lightweight task may cost a fraction of a credit; a long agent run over many files costs more because it does more work
If your audience remembers one line from this slide, make it this one: one AI credit equals one cent. That is the bridge from engineering behavior to finance language. Everything else follows from that. More context, more output, and heavier models cost more. Less context, tighter prompts, and cheaper models cost less. It sounds obvious, but PRUs trained many teams not to think this way, so it is worth stating directly.

What Is Billed and What Is Not?

Surface Uses AI Credits? Operational note
Copilot Chat Yes Scope and model choice matter immediately
Copilot CLI Yes Long transcripts and agentic loops can add up quickly
Copilot cloud agent Yes Multi-step agent sessions are high-value and high-consumption
Copilot code review Yes Also consumes GitHub Actions minutes
Code completions No Still included in paid plans
Next Edit Suggestions No Still included in paid plans
This slide is the practical pivot point for behavior change. If the work can be accomplished with completions or Next Edit Suggestions, do that first because those surfaces remain included. Chat, CLI, cloud agent, and code review are where the metered usage lives. That does not mean avoid them. It means use them intentionally and reserve them for tasks where their extra capability creates real value.

Even in an enterprise-focused session, it helps to show the full landscape because some attendees are comparing personal usage to team usage. The most important enterprise concept on this slide is pooling. Business and Enterprise do not create one tiny bucket per person. They create a shared pool at the billing entity level. That is a much better fit for real organizations, where a few power users often do much more high-value agentic work than everyone else.

How Spend Accumulates

graph LR
  A[Prompt or agent task] --> B[Context gathered]
  B --> C[Model selected]
  C --> D[Input + output + cached tokens]
  D --> E[AI credits consumed]
  E --> F{Pool or budget available?}
  F -->|Yes| G[Usage continues]
  F -->|No, overage allowed| H[Additional billed usage]
  F -->|No, overage blocked| I[Usage stops]
This flow is simple on purpose. Every user action becomes context, then a model choice, then tokens, then credits. The organization's controls only show up at the end of that path. That means cost management is partly an admin problem and partly a workflow design problem. If you only tune the budgets and ignore the prompt patterns, you will always be reacting instead of steering.

The Four Spend Levers You Control

Scope

Narrow prompts, smaller selections, and fewer irrelevant files reduce input tokens immediately.

Surface

Use completions or Next Edit Suggestions when they can solve the task instead of starting a billed chat or agent session.

Model

Lower-cost, faster models are enough for simple edits. Save deeper reasoning models for real complexity.

Session Length

Long conversations accumulate cost and noise. New task, new chat is often both cheaper and clearer.

These four levers are the backbone of the rest of the course. Most spend spikes can be explained by one of them. Either the scope was too broad, the wrong surface was used, the model was overpowered for the task, or the session kept dragging stale context forward. If teams learn to inspect usage through these four lenses, the billing page becomes much easier to explain and improve.
Part 3

Admin Controls and Governance

Now we move from mechanics to governance. GitHub added much better controls for organizations and enterprises, but controls only help if you use them in a deliberate policy structure. This section gives you the operating model, not just the feature list.

Pooled Credits Change the Economics

  • Each Business or Enterprise seat adds monthly included AI Credits to a shared pool
  • A company with 100 Business seats gets 190,000 pooled credits, not 100 isolated buckets
  • Power users can consume more when they are creating disproportionate value
  • Lighter users offset that consumption naturally without manual transfer or reallocation
  • Adding licenses mid-cycle increases the pool immediately; removing licenses reduces the pool at the next billing cycle
Pooling is one of the best parts of the new model for enterprise buyers. It recognizes that useful AI usage is unevenly distributed across a team. Your staff engineer doing multi-repo remediation work is going to consume more than a developer who only asks for occasional summaries. Under pooling, that is fine. The job of governance is no longer to flatten usage. It is to ensure high usage is intentional, justified, and visible.

Budget Hierarchy

graph TD
  A[Enterprise budget] --> B[Organization budget]
  B --> C[Cost center budget]
  C --> D[User budget]
  • Budgets can be used for visibility, alerts, or hard stops
  • User-level budgets are the sharpest control and can be set to $0 if needed
  • Cost centers help separate experiments, platform teams, and business units
  • Good governance starts broad, then narrows only where behavior data shows a real need
GitHub gives you four levels of budget control, but that does not mean you should use all four aggressively on day one. Start by deciding what you are trying to achieve. Do you want guardrails, budgeting visibility, or strict enforcement? Usually the right path is to set enterprise and organization visibility first, then use cost centers for pilot programs, and finally apply user-level hard stops to specific high-risk or high-cost cohorts if needed.

What Happens When the Pool Runs Out?

Additional Usage Allowed

Work continues and overage is billed at published rates. This is best when velocity matters more than a hard cap.

Additional Usage Blocked

Usage stops until the next cycle. This is best when finance requires a firm ceiling.

Important: if a user-level budget is exhausted, that user stops even if the broader pool still has headroom. There is no automatic fallback to a cheaper model when budget is exhausted.

This is a subtle but important rule. Organizations often assume pooled credits mean everyone can continue until the pool itself is empty. That is not true if you set user budgets. User-level caps override broader capacity. Also note that GitHub explicitly removed the old notion of falling back to a cheaper model when a limit is hit. Under the new system, you either keep paying or you stop. That makes your pre-set policy far more important.

A Sane Starting Policy

Layer Starting point Why
Enterprise Set alert thresholds, not hard stops Learn usage patterns before enforcing ceilings
Organization Track spend by org and flag outliers Find teams with unusual workflows early
Cost center Use for pilots or special initiatives Separates experiments from normal engineering spend
User Apply only to exception groups at first Avoid punishing legitimate heavy users before you understand ROI
Many teams will be tempted to lock everything down immediately. That is usually a mistake. The first month of a new billing model is a measurement exercise. Start with alerting and observation, not fear. Then identify where credits are going, which workflows are productive, and where poor habits are creating unnecessary cost. After that, budget controls become a scalpel instead of a hammer.

What to Review Every Week

  • Total AI Credits consumed vs. included pool remaining
  • Top users, teams, and cost centers by spend
  • Model mix: fast help vs. deep reasoning vs. agentic usage
  • Copilot code review usage and corresponding GitHub Actions minutes
  • Budget alerts, blocked users, and unexpected workflow spikes
Your weekly review should not be a finance-only artifact. It should be an engineering operations ritual. Look for changes in model mix, not just raw totals. If spend rises because a reliability team used cloud agent heavily during a critical remediation effort, that may be excellent value. If spend rises because everyone kept one mega-chat open for five days, that is coaching material. The billing overview gives you the raw signal. Your job is to interpret it in context.
Part 4

Developer Habits That Lower Spend

This is the part most developers care about. They do not want a lecture about billing. They want to know how to keep using Copilot effectively without becoming the reason the budget review gets tense. The answer is not to stop using AI. The answer is to choose the right surface, the right model, and the right context size for the job.

Choose the Cheapest Surface That Still Solves the Task

Need Lower-cost default Why it helps
One line, one block, one file Autocomplete or Next Edit Suggestions These remain included and avoid chat overhead
Question about a specific function Inline chat on a selection Smaller context means fewer billed tokens
Typical coding or explanation task Copilot Chat with explicit file references You keep scope tighter than a broad repo scan
Large, multi-step autonomous work Cloud agent or CLI agent Use the expensive path only when the autonomy pays for itself
This is probably the most useful habit slide in the deck. Developers often go straight to chat because it feels universal. Under usage-based billing, that is not always the smartest first move. If completions can draft the change, use them. If the question is about one function, use inline chat on that selection. Save fully agentic workflows for work that genuinely benefits from multi-step reasoning and tool use.

Example: Use Inline Chat on a Single Function

function calculateInvoiceTotal(lines, taxRate, discountPct) {
  const subtotal = lines.reduce((sum, line) => sum + line.qty * line.price, 0);
  const discount = subtotal * discountPct;
  return subtotal - discount + subtotal * taxRate;
}
  • Select only the function you want to change
  • Ask a precise prompt such as Fix the rounding bug and add a guard for negative discounts. Stay inside this function.
  • Avoid broad prompts like @workspace fix invoice math for a local issue
  • Start narrow, verify the fix, then widen scope only if needed
Inline chat is a great middle ground between free completions and broad chat sessions. The selection itself carries most of the context, so you do not need to drag the rest of the repository into the prompt. That reduces input tokens and usually improves answer quality because the model is not guessing which file or function you mean. It also naturally encourages developers to state the expected fix more clearly.

Example: Architect with Comments, Then Let Completions Fill It In

// Parse a CSV row into columns.
// Reject rows with missing sku or price.
// Normalize sku to uppercase and trim whitespace.
// Return a Product object with price converted to cents.

function parseProductRow(row) {
  
}
  • Use comments to define structure, validation rules, and output shape
  • Let autocomplete or Next Edit Suggestions expand each block
  • This shifts lightweight drafting work to unbilled completion surfaces
  • Use billed chat only if the implementation or edge cases become genuinely tricky
This is one of the highest-leverage spend habits because it moves work from a metered surface to an included surface. The comments do the architectural thinking in plain English, and completions draft the mechanical code. For a huge percentage of CRUD, parsing, mapping, and validation tasks, that is enough. Developers still get acceleration, but the organization does not pay chat or agent costs for work that completions can handle well.

Use Auto for the Default Path

  • Copilot Auto model selection is available across GitHub Copilot plans
  • In VS Code chat, task-optimized Auto combines real-time system health with task complexity routing
  • Auto is designed to reserve higher-cost reasoning models for problems that truly need them
  • Straightforward tasks are routed to faster, lower-cost models that still solve the problem efficiently
  • Policies still apply: Auto only chooses from models allowed by your plan and admin settings
If your audience wants one simple model policy, it is this: default to Auto. GitHub's current Auto routing is explicitly designed to improve cost efficiency by matching task complexity to model choice. That means developers do not need to manually think about frontier reasoning models for every prompt. Auto also keeps reliability and availability in the loop, which makes it a strong default for day-to-day work. Manual model picks should be the exception, not the default.

Override Auto Only When the Task Justifies It

Task shape Best fit Reason
Quick syntax, repetitive edits, tiny utilities Fast help / lightweight models Low latency and lower cost are enough
Normal coding, summaries, documentation General-purpose models Good balance of quality, speed, and cost
Complex debugging, refactors, architecture Deep reasoning models The task needs stronger reasoning across more context
Autonomous multi-step implementation Agentic coding models The workflow benefits from planning, tools, and iteration
The official model comparison guide is useful here because it frames model choice by task, not vendor preference. The mistake teams make is treating the most powerful model as the safest default. In practice, that is often slower and more expensive than needed. If the task is basically a small edit or explanation, use a lightweight or general-purpose path. Escalate to deep reasoning only when the problem is truly multi-step, ambiguous, or architecture-heavy.

Keep Prompts Concise and Batch Related Asks

Wasteful Pattern

Five follow-up messages for one task: "make it blue," "darker," "add padding," "fix the border," "one more thing."

Efficient Pattern

One prompt with all changes, desired output format, and acceptance criteria in a single round-trip.

  • Remove filler, repetition, and narrative backstory that does not help the model solve the task
  • Ask for a constrained output such as a checklist, diff summary, or five bullets max
  • Include success criteria once instead of re-explaining them across multiple messages
Multiple external prompt-engineering sources converge on the same advice: verbosity is expensive twice. It costs more because it adds tokens, and it costs more because it often causes clarifying back-and-forth. Batching related asks into a single clean prompt usually reduces both cost and cycle time. Also note that concise does not mean vague. A short but specific prompt is ideal. A short ambiguous prompt is a rework generator.

Reset or Compact Long Sessions

In Copilot

Use a new chat when the task changes. One task, one conversation is cheaper and usually more accurate.

Cross-Tool Pattern

If your coding agent supports transcript compaction, summarize the session instead of carrying the full history forever.

/compact Focus on the accepted API changes, remaining test failures, and files already modified.

Note: /compact is an example from other coding agents, not a Copilot chat command. The principle still applies: summarize stale context and carry only the decisions that matter.

The core idea here is larger than any one product. Long transcripts get expensive and noisy. In Copilot, the practical answer is often to start a fresh chat for a fresh task. In other tools that support compaction or summarization commands, the right move is to compress the transcript into a shorter working memory. The reason is the same in both cases: carry decisions forward, not the entire conversation history.

Other Internet-Tested Tactics That Translate Well

  • Share only the files you need. Local bug? Reference the file or selection, not the entire workspace.
  • Do trivial edits yourself. Renames, imports, and tiny formatting fixes are often faster than opening a billed session.
  • Keep instruction files lean. Shared context should be current and useful, not a 500-line manifesto loaded every time.
  • Chunk complex work semantically. Break a large task into meaningful parts instead of asking for everything in one giant prompt.
  • Prefer structured outputs. Tables, checklists, and concise summaries reduce unnecessary response length.
These tactics come from broader LLM and coding-assistant guidance across the web, and they hold up well in a Copilot environment. They all push in the same direction: less irrelevant context, fewer unnecessary round-trips, and better bounded outputs. I would not present them as magic tricks. They are workflow hygiene. But good workflow hygiene becomes much more visible when your AI platform starts billing by actual usage.
Part 5

Rollout and Real Scenarios

We now have the concepts and the habits. The last step is turning them into rollout guidance that a real organization can use. These scenario slides help explain when cheaper patterns are enough and when higher-cost AI workflows are absolutely justified.

Scenario: Same Bug, Cheaper Workflow

Approach What happens Likely spend pattern
Expensive default Open a broad chat, describe the bug vaguely, let the model explore multiple files, then refine through several follow-ups Higher input tokens, more round-trips, higher chance of rework
Efficient default Select the function, use inline chat for the bug, or write a comment scaffold and accept completions for the fix Lower context cost and better first-pass accuracy
This scenario is useful because it does not ask developers to stop using AI. It simply asks them to pick a smarter entry point. The expensive path is not wrong. It is just wasteful for a small, local issue. The cheaper path narrows context, reduces prompt churn, and often produces a more precise answer because the model is not wandering through the codebase looking for the problem.

Scenario: When Higher Spend Is the Right Call

Good Use of Higher-Cost AI

  • Multi-file refactor with tests
  • Architecture trade-off analysis
  • Repo-wide remediation campaign
  • Agentic workflow with clear acceptance criteria

Poor Use of Higher-Cost AI

  • Single-function bug fix
  • Simple boilerplate the editor can complete
  • Repeated formatting or rename tasks
  • Vague prompts that force unnecessary exploration
A spend-aware culture should not become a fear culture. There are absolutely situations where deeper reasoning models, chat, CLI, or cloud agent are worth every credit. The key test is leverage. If the workflow replaces meaningful human effort, lowers risk, or accelerates a multi-step task that would otherwise be slow and error-prone, higher spend can be excellent value. The anti-pattern is using that same power for tiny jobs that cheaper surfaces solve just as well.

A Practical 30-Day Rollout Plan

Window What to do
Week 0 Review official plan allowances, enable preview billing visibility, and align finance with engineering on overage policy
Week 1 Publish model guidance: default to Auto, use inline chat for local work, and prefer completions for lightweight drafting
Week 2 Create cost centers for pilots or special programs, then start weekly usage reviews
Week 3 Identify expensive but low-value patterns and coach teams on better workflows
Week 4 Adjust budgets, document proven practices, and decide where user-level caps are actually needed
A 30-day plan matters because this transition is behavioral as much as technical. Week 1 is communication. Week 2 is observation. Week 3 is coaching. Week 4 is policy refinement. If you reverse that order and start with restrictive caps before your teams understand the new model, you will create frustration without actually learning which workflows deserve investment.

Team Policy Checklist

  • Default to Auto unless there is a clear reason to override it
  • Prefer completions and Next Edit Suggestions for drafting and local edits
  • Use inline chat on a selection before escalating to broad chat or agentic workflows
  • One task, one chat. Start fresh when the task changes
  • Batch related asks and ask for concise outputs
  • Monitor both AI Credits and GitHub Actions minutes for code review
  • Review expensive workflows weekly and document the ones that are worth the spend
This checklist is intentionally short. If you hand teams a ten-page guidance document, they will ignore it. These seven points are enough to establish a spend-aware default culture. Over time, you can refine them with team-specific examples, but keep the top-level message simple: use the cheapest tool that still solves the task, and reserve high-cost AI for high-leverage work.
Summary

Wrap-Up and Next Steps

We have covered the official shift, the pricing model, the governance controls, and the developer behaviors that matter most. Let us close with the takeaways you want people to remember after the meeting ends and the links they will actually need when they go implement this.

Key Takeaways

  • PRUs are gone; Copilot usage now maps more directly to tokens, models, and AI Credits
  • Seat prices stay the same, but variable usage becomes much more visible and controllable
  • For enterprises, pooling and budgets are the two biggest governance improvements
  • Code completions and Next Edit Suggestions remain your best low-cost drafting surfaces
  • Inline chat, concise prompts, and fresh sessions reduce both cost and noise
  • Auto model selection should be the default for most teams
  • The right goal is not minimal usage. The right goal is high-value usage
If the room leaves with only these points, the course has done its job. The biggest mindset shift is that optimization is no longer about avoiding advanced features. It is about matching the feature, model, and scope to the value of the task. Good teams will still spend credits. They will just spend them on work that deserves them.
