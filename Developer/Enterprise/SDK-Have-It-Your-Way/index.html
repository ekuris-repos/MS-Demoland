Part 1

Why Build Outside the Editor?

Before we look at the SDK itself, we need to understand why you would want to take Copilot outside of VS Code and into your own applications.

Use Cases for Standalone Copilot Apps

Customer-Facing

  • Custom web UIs: Embed Copilot intelligence into dashboards and portals
  • Chat widgets: Add AI-powered support to your product
  • Mobile apps: Integrate Copilot into native experiences

Internal Tooling

  • Slack/Teams bots: AI assistants in your team's chat platform
  • CLI tools: Command-line utilities powered by Copilot
  • Internal portals: Knowledge retrieval and workflow automation
The Copilot SDK opens up a wide range of scenarios. On the customer-facing side, you can embed Copilot intelligence into your own web applications, build chat widgets that sit inside your product, or integrate AI into mobile experiences. On the internal side, think Slack bots that answer engineering questions, CLI tools that help developers automate repetitive tasks, or internal portals that surface knowledge from your organization's documentation. The key idea is that Copilot is no longer confined to the editor; it becomes a service you can call from anywhere.

Why the SDK Instead of Raw API Calls?

  • Session management: The SDK handles conversation state, message history, and context windows automatically
  • Tool orchestration: Built-in support for function calling, tool execution, and result injection
  • Streaming: First-class support for streaming responses with proper backpressure handling
  • Authentication: Integrates with GitHub's auth system; no separate API keys to manage
  • Model abstraction: Switch between GitHub Models and Azure OpenAI without changing application code
  • Production hardened: The same runtime that powers GitHub Copilot in production
You might wonder why you need an SDK at all when you could just call model APIs directly. The answer is that the SDK does a lot of heavy lifting for you. It manages sessions and conversation state, orchestrates tool calls automatically, handles streaming with proper backpressure, and integrates with GitHub's authentication system so you do not need separate API keys. Perhaps most importantly, it gives you model abstraction: you can switch between GitHub Models and Azure OpenAI without rewriting your application. This is the same runtime that powers Copilot in production, so you get battle-tested reliability out of the box.
Part 2

SDK Architecture

Now that we know why the SDK exists, let's look under the hood at how it is structured.

Core Components

  • CopilotClient: The main entry point. Creates sessions, sends messages, and manages the connection to the Copilot service
  • Session: Represents a conversation. Tracks message history, context, and model configuration
  • Message: A single user or assistant turn, including content, role, and optional tool calls
  • Tool definitions: JSON Schema descriptions of functions the model can invoke during a conversation
  • Response stream: An async iterable that yields tokens, tool-call events, and completion signals
The SDK is built around a handful of core components. The CopilotClient is your main entry point; you create one, authenticate it, and use it to open sessions. A Session represents a single conversation with its own message history and model configuration. Messages are individual turns in the conversation, each tagged with a role like "user" or "assistant." Tool definitions let you describe functions the model can call, and the response stream gives you an async iterable you can consume token by token for real-time output. These components compose together cleanly, and we will see each one in action as we build our first app.

Message Flow

  1. Your app creates a CopilotClient and calls createSession()
  2. Your app sends a user message via sendAndWait() or send()
  3. The SDK forwards the message to the Copilot service along with session context
  4. The model generates a response (or requests a tool call)
  5. If a tool call is requested, the SDK invokes your registered handler and returns the result to the model
  6. The final response is returned to your application as a complete message or a token stream
Let's walk through the message flow step by step. Your application creates a CopilotClient and opens a session. When the user types something, you send that message through the SDK. The SDK packages it up with session context and forwards it to the Copilot service. The model processes the message and either generates a direct response or requests a tool call. If a tool call is requested, the SDK automatically invokes your registered handler, feeds the result back to the model, and the model produces its final response. Your application receives either a complete message or a streaming token sequence. The important takeaway is that tool-call orchestration is fully automatic; you just register your handlers and the SDK does the rest.

Getting Started

Scaffolding with azd

The fastest way to start is with the copilot-sdk-service Azure Developer CLI template:

azd init --template copilot-sdk-service
cd my-copilot-app
npm install

This creates a ready-to-deploy project with:

  • A working Express server with Copilot SDK integration
  • Azure Bicep files for App Service or Container Apps deployment
  • An azure.yaml manifest for azd up deployment
  • Example tool definitions and handlers
The quickest way to get started is with the azd template. Run "azd init --template copilot-sdk-service" and you get a project that is ready to deploy from day one. The template gives you an Express server pre-wired with the Copilot SDK, Bicep infrastructure files for Azure deployment, an azure.yaml manifest so you can deploy with a single "azd up" command, and example tool definitions you can customize. You will not need to configure authentication, set up build pipelines, or wire up infrastructure from scratch. This is the recommended starting point for any production project.

Project Structure

my-copilot-app/
├── src/
│   ├── index.ts          # Express server entry point
│   ├── copilot.ts        # CopilotClient setup
│   ├── tools/

│   │   └── handlers.ts     # Tool implementation

└── tsconfig.json
Here is the typical project layout you get from the template. The src directory contains your application code: index.ts is the Express server entry point, copilot.ts sets up the CopilotClient, and the tools directory holds your tool definitions and handlers. The routes directory contains your HTTP endpoints, typically a chat route. The infra directory has Bicep files for Azure infrastructure, and azure.yaml ties everything together for azd deployments. This structure keeps concerns separated cleanly. Your SDK integration lives in copilot.ts, your business logic lives in tool handlers, and your infrastructure is fully codified.
Part 4

Building Your First App

Let's write some code. We will build a simple Copilot-powered chat endpoint from scratch.

Creating the CopilotClient

import { CopilotClient } from "@github/copilot-sdk";

const client = new CopilotClient({
  token: process.env.GITHUB_TOKEN,
});

// Create a session
const session = await client.createSession({
  instructions: "You are a helpful assistant for our support team.",
});

// Send a message and wait for the response
const response = await session.sendAndWait(
  "How do I reset a user's password?"
);

console.log(response.message.content);
This is the simplest possible Copilot SDK application. You import CopilotClient from the package, create an instance with your GitHub token, and open a session. The instructions parameter sets the system prompt that guides the model's behavior. Then you call sendAndWait with a user message and get back a complete response. That's it. Five lines of meaningful code to get a working Copilot integration. Notice that you do not specify a model, manage context windows, or handle retries; the SDK takes care of all of that. In production you would not hardcode the token, of course, but this shows just how simple the happy path is.

Integrating with Express

import express from "express";
import { CopilotClient } from "@github/copilot-sdk";

const app = express();
app.use(express.json());

const client = new CopilotClient({
  token: process.env.GITHUB_TOKEN,
});

app.post("/chat", async (req, res) => {
  const { message, sessionId } = req.body;
  const session = await client.getOrCreateSession(sessionId);

  const response = await session.sendAndWait(message);
  res.json({
    reply: response.message.content,
    sessionId: session.id,
  });
});

app.listen(3000);
Here we wire the SDK into an Express server. The /chat endpoint accepts a message and an optional session ID. If a session ID is provided, we resume that conversation; otherwise, we create a new one. The response includes both the assistant's reply and the session ID so the client can continue the conversation in future requests. This is the pattern you will use for web applications, chat widgets, and API services. The key thing to notice is getOrCreateSession: it lets you maintain conversation continuity across HTTP requests without managing state yourself.
Part 5

Streaming & Tool Integration

Now let's look at two features that take your application from a basic demo to something production-worthy: streaming responses and tool integration.

Streaming Responses

app.post("/chat/stream", async (req, res) => {
  const { message, sessionId } = req.body;
  const session = await client.getOrCreateSession(sessionId);

  res.setHeader("Content-Type", "text/event-stream");

  const stream = session.send(message);

  for await (const event of stream) {
    if (event.type === "content") {
      res.write(`data: ${JSON.stringify({
        token: event.content
      })}\n\n`);
    }
  }

  res.write("data: [DONE]\n\n");
  res.end();
});
Streaming is essential for a good user experience. Instead of waiting for the entire response to generate, you send tokens to the client as they arrive. The SDK's send method returns an async iterable that yields events. Each event has a type; "content" events carry a token of text. We use Server-Sent Events here, but you could also use WebSockets or any other streaming transport. The key difference from sendAndWait is that send does not block. You consume the stream in a for-await loop and forward each token to your client. This gives users near-instant feedback and makes long responses feel fast.

Defining and Registering Tools

const tools = [
  {
    name: "lookup_order",
    description: "Look up an order by its ID",
    parameters: {
      type: "object",
      properties: {
        orderId: {
          type: "string",
          description: "The order ID to look up",
        },
      },
      required: ["orderId"],
    },
  },
];

const session = await client.createSession({
  instructions: "You are a support agent.",
  tools,
  onToolCall: async (toolCall) => {
    if (toolCall.name === "lookup_order") {
      const order = await db.orders.find(
        toolCall.arguments.orderId
      );
      return JSON.stringify(order);
    }
  },
});
Tool integration is where the SDK really shines. You define tools as JSON Schema objects with a name, description, and parameter schema. Then you register them when you create a session, along with an onToolCall handler that implements the actual logic. When the model decides it needs to look up an order, it emits a tool-call event. The SDK automatically invokes your handler, feeds the result back to the model, and the model incorporates that data into its response. You do not need to manage the back-and-forth yourself. This pattern works for database lookups, API calls, file operations, or any other server-side action you want the model to invoke.
Part 6

Multi-Turn Conversations

Real applications need more than single-shot interactions. Let's look at how the SDK manages context across multiple turns.

Context Management

Automatic Context

  • Session tracks full message history
  • Context window is managed automatically
  • Older messages are summarized when the window fills
  • Tool-call results are included in context

Manual Control

  • Set maxHistory to limit retained turns
  • Inject context with session.addContext()
  • Clear history with session.reset()
  • Persist sessions for cross-request continuity
The SDK gives you two levels of context management. By default, it handles everything automatically: message history is tracked in the session, context windows are managed so you do not exceed token limits, and tool-call results are retained for the model to reference later. When you need more control, the SDK exposes manual options. You can set maxHistory to limit how many turns are retained, inject additional context with addContext for things like user profiles or retrieved documents, clear history with reset when a conversation topic changes, and persist sessions to a store for continuity across HTTP requests. This balance of automatic and manual control means most applications work out of the box, but you can fine-tune when you need to.

Multi-Turn in Practice

// Turn 1: User asks a question
const r1 = await session.sendAndWait(
  "What's the status of order ORD-1234?"
);
// Model calls lookup_order tool, returns details

// Turn 2: Follow-up referencing earlier context
const r2 = await session.sendAndWait(
  "Can you cancel it?"
);
// Model knows "it" = ORD-1234 from session context

// Turn 3: Inject additional context
session.addContext(
  "Company policy: orders can only be cancelled within 24h."
);
const r3 = await session.sendAndWait(
  "Actually, is it too late to cancel?"
);
// Model checks order date against policy
Here is a practical example of multi-turn in action. In turn one, the user asks about an order, and the model automatically calls the lookup_order tool. In turn two, the user says "Can you cancel it?" and the model resolves "it" to ORD-1234 from the session history. In turn three, we inject a company policy using addContext, and the model now factors that policy into its response. This is the experience your users expect from a conversational interface: natural follow-ups, pronoun resolution, and context-aware answers. The SDK handles the complexity of maintaining all of this state.
Part 7

Bring Your Own Model

Not every organization can use GitHub-hosted models. Let's look at how the SDK supports custom model endpoints.

BYOM: Azure OpenAI

import { CopilotClient } from "@github/copilot-sdk";
import { DefaultAzureCredential }
  from "@azure/identity";

const client = new CopilotClient({
  model: {
    endpoint: process.env.AZURE_OPENAI_ENDPOINT,
    deployment: "gpt-4o",
    credential: new DefaultAzureCredential(),
  },
});
  • No API keys: Use DefaultAzureCredential for managed identity authentication
  • Same SDK surface: Sessions, tools, and streaming work identically
  • Data residency: Keep data within your Azure region and subscription
  • Custom models: Point to any OpenAI-compatible endpoint
Bring Your Own Model, or BYOM, is critical for enterprise adoption. Many organizations require data residency guarantees or have existing Azure OpenAI deployments they want to leverage. With the SDK, you simply pass a model configuration object instead of a GitHub token. Here we are using DefaultAzureCredential from the Azure Identity SDK, which means you get managed identity support with no API keys to rotate. The rest of your code stays exactly the same: sessions, tools, streaming, multi-turn conversations, all of it works identically regardless of the model backend. You can also point to any OpenAI-compatible endpoint, so self-hosted models are supported too.

When to Use BYOM vs. GitHub Models

GitHub Models

  • Fastest path to production
  • No Azure subscription required
  • Automatic model updates
  • Built-in rate limits and quotas
  • Best for: prototypes, dev tools, internal apps

BYOM (Azure OpenAI)

  • Data residency and compliance
  • Custom fine-tuned models
  • Managed identity authentication
  • Dedicated capacity (PTU)
  • Best for: regulated industries, high-volume production
The decision between GitHub Models and BYOM depends on your requirements. GitHub Models is the fastest path: no Azure subscription, automatic model updates, and built-in rate limits. It is perfect for prototypes, developer tools, and internal applications where simplicity matters. BYOM with Azure OpenAI is the choice when you need data residency guarantees, custom fine-tuned models, managed identity authentication, or dedicated capacity via Provisioned Throughput Units. Regulated industries, healthcare, finance, and government workloads will almost always need BYOM. The good news is that you can start with GitHub Models and switch to BYOM later without changing your application code.

Production Deployment

Your app works locally. Now let's get it running in Azure with proper monitoring and scaling.

Deploying to Azure

# One-command deployment with azd
azd up

# Or deploy to specific Azure services:
# App Service (simplest)
azd up --environment production

# Container Apps (scalable)
azd up --template copilot-sdk-service-container
  • App Service: Simplest option. Managed platform, auto-scaling, easy SSL
  • Container Apps: Kubernetes-based. Scale to zero, event-driven scaling, Dapr integration
  • Infrastructure as code: Bicep files in the infra/ directory define all resources
  • CI/CD: The template includes a GitHub Actions workflow for automated deployments
Deployment is straightforward because the azd template includes everything you need. Run "azd up" and the CLI provisions your Azure resources, builds your application, and deploys it. You have two primary hosting options. App Service is the simplest: it is a managed platform with auto-scaling, built-in SSL, and easy configuration. Container Apps gives you more control with Kubernetes-based infrastructure, scale-to-zero capability, and Dapr integration for microservices. Both options are defined in the Bicep files under the infra directory, so your infrastructure is fully codified and reproducible. The template also includes a GitHub Actions workflow, so you can set up automated deployments from day one.

Monitoring & Observability

  • Application Insights: Automatic request tracing, dependency tracking, and exception logging
  • Token usage metrics: Track prompt and completion token consumption per session
  • Latency monitoring: Measure time-to-first-token and total response time
  • Tool-call analytics: Monitor which tools are invoked, how often, and their success rate
  • Alerts: Set thresholds for error rates, latency spikes, and quota consumption
// Custom telemetry example
const session = await client.createSession({
  instructions: "You are a helpful assistant.",
  onMetrics: (metrics) => {
    appInsights.trackEvent({
      name: "CopilotResponse",
      properties: {
        sessionId: metrics.sessionId,
        promptTokens: metrics.promptTokens,
        completionTokens: metrics.completionTokens,
        latencyMs: metrics.latencyMs,
      },
    });
  },
});
Once your app is in production, observability is essential. The azd template configures Application Insights automatically, giving you request tracing, dependency tracking, and exception logging out of the box. Beyond the basics, you should track Copilot-specific metrics: token usage per session so you can forecast costs, time-to-first-token for user experience monitoring, and tool-call analytics to understand which tools are being used and how often they fail. The SDK provides an onMetrics callback that fires after each response, making it easy to push custom telemetry to Application Insights, Datadog, or whatever monitoring platform you use. Set alerts on error rates and latency spikes so you catch issues before your users do.
Part 9

Security Considerations

Security is not optional for production AI applications. Let's cover the key areas you need to get right.

Authentication & Token Management

GitHub Token Flow

  • Use GitHub App or OAuth App tokens
  • Store tokens in Azure Key Vault
  • Rotate tokens on a schedule
  • Never commit tokens to source control

Azure Managed Identity

  • No secrets to manage for BYOM
  • System-assigned or user-assigned identities
  • Role-based access control for model endpoints
  • Automatic credential rotation
Authentication is the most critical security consideration for any Copilot SDK application. When using GitHub Models, you authenticate with a GitHub token. In production, this should be a GitHub App installation token or an OAuth App token stored in Azure Key Vault, never in environment variables or source code. Implement token rotation on a schedule and monitor for unauthorized usage. When using BYOM with Azure OpenAI, the picture is simpler: use managed identities and you have no secrets to manage at all. DefaultAzureCredential handles credential rotation automatically. Whichever path you choose, apply the principle of least privilege. Your application should only have access to the models and tools it needs, nothing more.

Security Best Practices

  • Input validation: Sanitize user input before passing it to the model. Validate tool-call arguments against your schemas
  • Output filtering: Review model responses for sensitive data before returning them to users
  • Rate limiting: Implement per-user rate limits to prevent abuse and control costs
  • Audit logging: Log all tool calls, model interactions, and authentication events
  • Network security: Run your service in a virtual network. Restrict outbound traffic to known endpoints
  • Prompt injection defense: Use system instructions to constrain model behavior. Validate tool outputs before acting on them
Beyond authentication, there are several security practices you should follow. Validate all user input before sending it to the model, and validate tool-call arguments against your JSON schemas to prevent injection attacks. Filter model responses for sensitive data; the model might inadvertently surface information from its context that should not reach the end user. Implement per-user rate limits to prevent both abuse and runaway costs. Log everything: every tool call, every model interaction, every authentication event. These logs are essential for incident response and compliance audits. Run your service in a virtual network and restrict outbound traffic to only the endpoints it needs. Finally, design your system instructions to constrain the model's behavior and treat tool outputs as untrusted input that needs validation before acting on it.
