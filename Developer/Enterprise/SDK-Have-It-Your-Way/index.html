Use Cases for Standalone Copilot Apps

Customer-Facing

  • Custom web UIs: Embed Copilot intelligence into dashboards and portals
  • Chat widgets: Add AI-powered support to your product
  • Mobile apps: Integrate Copilot into native experiences

Internal Tooling

  • Slack/Teams bots: AI assistants in your team's chat platform
  • CLI tools: Command-line utilities powered by Copilot

The Copilot SDK opens up a wide range of scenarios. On the customer-facing side, you can embed Copilot intelligence into your own web applications, build chat widgets that sit inside your product, or integrate AI into mobile experiences. Why the SDK Instead of Raw API Calls?

  • Session management: The SDK handles conversation state, message history, and context windows automatically
  • Tool orchestration: Built-in support for function calling, tool execution, and result injection
  • Streaming: First-class support for streaming responses with proper backpressure handling
  • Authentication: Integrates with GitHub's auth system; no separate API keys to manage
  • Model abstraction: Switch between GitHub Models and Azure OpenAI with minimal configuration changes
  • Production hardened: A client SDK for building applications powered by the Copilot platform
You might wonder why you need an SDK at all when you could just call model APIs directly. The answer is that the SDK does a lot of heavy lifting for you. It manages sessions and conversation state, orchestrates tool calls automatically, handles streaming with proper backpressure, and integrates with GitHub's authentication system so you do not need separate API keys. It also gives you model abstraction: you can switch between GitHub Models and Azure OpenAI with minimal configuration changes. The SDK provides a client library for building applications powered by the Copilot platform, so you get a well-supported foundation out of the box.
Part 2

SDK Architecture

Now that we know why the SDK exists, let's look under the hood at how it is structured.

Core Components

  • CopilotClient: The main entry point. Creates sessions, sends messages, and manages the connection to the Copilot service
  • Session: Represents a conversation. Tracks message history, context, and model configuration
  • Message: A single user or assistant turn, including content, role, and optional tool calls
  • Tool definitions: JSON Schema descriptions of functions the model can invoke during a conversation
  • Response stream: An async iterable that yields tokens, tool-call events, and completion signals
The SDK is built around a handful of core components. The CopilotClient is your main entry point; you create one, authenticate it, and use it to open sessions. A Session represents a single conversation with its own message history and model configuration. Messages are individual turns in the conversation, each tagged with a role like "user" or "assistant." Tool definitions let you describe functions the model can call, and the response stream gives you an async iterable you can consume token by token for real-time output. These components compose together cleanly, and we will see each one in action as we build our first app.

Message Flow

  1. Your app creates a CopilotClient and calls createSession()
  2. Your app sends a user message via sendAndWait() or send()
  3. The SDK forwards the message to the Copilot service along with session context
  4. The model generates a response (or requests a tool call)
  5. If a tool call is requested, the SDK invokes your registered handler and returns the result to the model
  6. The final response is returned to your application as a complete message or a token stream
Let's walk through the message flow step by step. Your application creates a CopilotClient and opens a session. When the user types something, you send that message through the SDK. The SDK packages it up with session context and forwards it to the Copilot service. The model processes the message and either generates a direct response or requests a tool call. If a tool call is requested, the SDK automatically invokes your registered handler, feeds the result back to the model, and the model produces its final response. Your application receives either a complete message or a streaming token sequence. The important takeaway is that tool-call orchestration is fully automatic; you just register your handlers and the SDK does the rest.

Building Your First App

Let's write some code. We will build a simple Copilot-powered chat endpoint from scratch.

Creating the CopilotClient

import { CopilotClient } from "@github/copilot-sdk";const client = new CopilotClient({  token: process.env.GITHUB_TOKEN,});// Create a sessionconst session = await client.createSession();// Send a message and wait for the responseconst response = await session.sendAndWait(  "Explain this code");console.log(response);
sequenceDiagram
  participant User
  participant SDK as SDK Agent
  participant Tool as Tool Handler
  User->>SDK: Send message
  SDK->>SDK: Model requests tool call
  SDK->>Tool: Invoke handler
  Tool-->>SDK: Return result
  SDK->>SDK: Feed result to model
  SDK-->>User: Final response
This is the simplest possible Copilot SDK application. You import CopilotClient from the package, create an instance with your GitHub token, and open a session. Then you call sendAndWait with a user message and get back a response. That is it: a few lines of meaningful code to get a working Copilot integration. Notice that you do not specify a model, manage context windows, or handle retries. The SDK takes care of all of that. In production you would not hardcode the token, of course, but this shows just how simple the happy path is.

Integrating with Express

import express from "express";import { CopilotClient } from "@github/copilot-sdk";const app = express();app.use(express.json());const client = new CopilotClient({  token: process.env.GITHUB_TOKEN,});app.post("/chat", async (req, res) => {  const { message } = req.body;  const session = await client.createSession();  const response = await session.sendAndWait(message);  res.json({    reply: response,    sessionId: session.id,  });});app.listen(3000);
Here we wire the SDK into an Express server. The /chat endpoint accepts a message and creates a new session. The response includes both the assistant's reply and the session ID so the client can reference it in future requests. This is the pattern you will use for web applications, chat widgets, and API services. The key thing to notice is createSession: it initializes a new conversation that you can then interact with through the session object.
Part 5

Streaming & Tool Integration

Now let's look at two features that take your application from a basic demo to something production-worthy: streaming responses and tool integration.

Streaming Responses

app.post("/chat/stream", async (req, res) => {  const { message } = req.body;  const session = await client.createSession();  res.setHeader("Content-Type", "text/event-stream");  const stream = session.send(message);  for await (const chunk of stream) {    res.write(`data: ${JSON.stringify({ chunk })}\n\n`);  }  res.write("data: [DONE]\n\n");  res.end();});
Streaming is essential for a good user experience. Instead of waiting for the entire response to generate, you send chunks to the client as they arrive. The SDK's send method returns an async iterable that you consume in a for-await loop. We use Server-Sent Events here, but you could also use WebSockets or any other streaming transport. The key difference from sendAndWait is that send does not block. You iterate over the stream and forward each chunk to your client. This gives users near-instant feedback and makes long responses feel fast.

Defining and Registering Tools

const tools = [  {    name: "lookup_order",    description: "Look up an order by its ID",    parameters: {      type: "object",      properties: {        orderId: {          type: "string",          description: "The order ID to look up",        },      },      required: ["orderId"],    },  },];// Register tools with the sessionconst session = await client.createSession({ tools });

The SDK handles tool orchestration automatically. When the model decides to invoke a tool, the SDK manages the call and feeds the result back into the conversation.

Tool integration is where the SDK really shines. You define tools as JSON Schema objects with a name, description, and parameter schema. Then you pass them when you create a session. When the model decides it needs to invoke a tool, the SDK handles the orchestration automatically and feeds the result back into the conversation. This pattern works for database lookups, API calls, file operations, or any other server-side action you want the model to invoke.
Part 6

Multi-Turn Conversations

Real applications need more than single-shot interactions. Let's look at how the SDK manages context across multiple turns.

Context Management

Automatic Context

  • Session tracks full message history
  • Context window is managed automatically
  • Tool-call results are included in context

Providing Context

  • Conversation history is managed through the session object
  • Additional context can be provided through system messages in the conversation
  • Persist sessions for cross-request continuity
The SDK gives you two levels of context management. By default, it handles everything automatically: message history is tracked in the session, context windows are managed so you do not exceed token limits, and tool-call results are retained for the model to reference later. When you need more control, you can provide additional context through system messages in the conversation. You can also persist sessions for continuity across HTTP requests. This balance of automatic and explicit control means most applications work out of the box, but you can fine-tune when you need to.

Multi-Turn in Practice

// Turn 1: User asks a questionconst r1 = await session.sendAndWait(  "What's the status of order ORD-1234?");// Model calls lookup_order tool, returns details// Turn 2: Follow-up referencing earlier contextconst r2 = await session.sendAndWait(  "Can you cancel it?");// Model knows "it" = ORD-1234 from session context// Turn 3: Provide context through conversation messagesconst r3 = await session.sendAndWait(  "Our company policy says orders can only be cancelled "  + "within 24 hours. Is it too late to cancel?");// Model checks order date against the stated policy
Here is a practical example of multi-turn in action. In turn one, the user asks about an order, and the model automatically calls the lookup_order tool. In turn two, the user says "Can you cancel it?" and the model resolves "it" to ORD-1234 from the session history. In turn three, we provide additional context directly in the user message, and the model factors that context into its response. This is the experience your users expect from a conversational interface: natural follow-ups, pronoun resolution, and context-aware answers. The SDK handles the complexity of maintaining conversation state.
Part 7

Bring Your Own Model

Not every organization can use GitHub-hosted models. Let's look at how the SDK supports custom model endpoints.

BYOM: Azure OpenAI

import { CopilotClient } from "@github/copilot-sdk";const client = new CopilotClient({  token: process.env.GITHUB_TOKEN,});// Use Azure OpenAI at session levelconst session = await client.createSession({  model: "o4-mini",  provider: {    type: "azure",    baseUrl: process.env.AZURE_OPENAI_ENDPOINT,    bearerToken: token,    wireApi: "completions",    azure: { apiVersion: "2025-04-01-preview" },  },});
  • Session-level provider: Configure the Azure OpenAI provider on each session with a pre-fetched bearer token
  • Same SDK surface: Sessions, tools, and streaming work identically
  • Data residency: Keep data within your Azure region and subscription
  • Custom models: Point to any OpenAI-compatible endpoint
Bring Your Own Model, or BYOM, is critical for enterprise adoption. Many organizations require data residency guarantees or have existing Azure OpenAI deployments they want to leverage. With the SDK, you configure the provider at the session level, passing a pre-fetched bearer token and your Azure OpenAI endpoint. The rest of your code stays the same: tools, streaming, and multi-turn conversations all work identically regardless of the model backend. You can also point to any OpenAI-compatible endpoint, so self-hosted models are supported too.

When to Use BYOM vs. GitHub Models

GitHub Models

  • Fastest path to production
  • No Azure subscription required
  • Automatic model updates
  • Built-in rate limits and quotas
  • Best for: prototypes, dev tools, internal apps

BYOM (Azure OpenAI)

  • Data residency and compliance
  • Custom fine-tuned models
  • Managed identity authentication
  • Dedicated capacity (PTU)
  • Best for: regulated industries, high-volume production
The decision between GitHub Models and BYOM depends on your requirements. GitHub Models is the fastest path: no Azure subscription, automatic model updates, and built-in rate limits. It is perfect for prototypes, developer tools, and internal applications where simplicity matters. BYOM with Azure OpenAI is the choice when you need data residency guarantees, custom fine-tuned models, managed identity authentication, or dedicated capacity via Provisioned Throughput Units. Regulated industries, healthcare, finance, and government workloads will almost always need BYOM. The good news is that you can start with GitHub Models and switch to BYOM later with minimal configuration changes.

Security Considerations

Security is not optional for production AI applications. Let's cover the key areas you need to get right.

Authentication & Token Management

GitHub Token Flow

  • Use GitHub App or OAuth App tokens
  • Store tokens in Azure Key Vault
  • Rotate tokens on a schedule
  • Never commit tokens to source control

Azure Managed Identity

  • No secrets to manage for BYOM
  • System-assigned or user-assigned identities
  • Role-based access control for model endpoints
  • Automatic credential rotation
Authentication is the most critical security consideration for any Copilot SDK application. When using GitHub Models, you authenticate with a GitHub token. In production, this should be a GitHub App installation token or an OAuth App token stored in Azure Key Vault, never in environment variables or source code. Implement token rotation on a schedule and monitor for unauthorized usage. When using BYOM with Azure OpenAI, the picture is simpler: use managed identities and you have no secrets to manage at all. DefaultAzureCredential handles credential rotation automatically. Whichever path you choose, apply the principle of least privilege. Your application should only have access to the models and tools it needs, nothing more.

Security Best Practices

  • Input validation: Sanitize user input before passing it to the model. Validate tool-call arguments against your schemas
  • Output filtering: Review model responses for sensitive data before returning them to users
  • Rate limiting: Implement per-user rate limits to prevent abuse and control costs
  • Audit logging: Log all tool calls, model interactions, and authentication events
  • Network security: Run your service in a virtual network. Restrict outbound traffic to known endpoints
  • Prompt injection defense: Use system instructions to constrain model behavior. Validate tool outputs before acting on them
Beyond authentication, there are several security practices you should follow. Validate all user input before sending it to the model, and validate tool-call arguments against your JSON schemas to prevent injection attacks. Filter model responses for sensitive data; the model might inadvertently surface information from its context that should not reach the end user. Implement per-user rate limits to prevent both abuse and runaway costs. Log everything: every tool call, every model interaction, every authentication event. These logs are essential for incident response and compliance audits. Run your service in a virtual network and restrict outbound traffic to only the endpoints it needs. Finally, design your system instructions to constrain the model's behavior and treat tool outputs as untrusted input that needs validation before acting on it.
