Part 1

What Is Copilot CLI?

Let's start at the very beginning. What exactly is Copilot CLI, and where does it fit in the GitHub ecosystem?

Copilot CLI at a Glance

  • An extension for the GitHub CLI (gh) that adds AI-powered assistance directly in your terminal
  • Two core commands: gh copilot suggest for generating commands, and gh copilot explain for understanding them
  • Works across bash, PowerShell, and cmd on macOS, Linux, and Windows
  • No editor required. Just your terminal, an authenticated GitHub account, and a Copilot license
Copilot CLI is an extension for the GitHub CLI, which is that gh command you may already use for pull requests, issues, and repo management. Once installed, it gives you two new subcommands: suggest and explain. Suggest takes a natural-language description and returns a shell command you can run. Explain takes a command you already have and breaks it down in plain English. It works across all major shells and operating systems. The key insight here is that you do not need VS Code open at all. If you are SSHed into a server, working in a CI runner, or just prefer your terminal, Copilot CLI meets you where you are.

Prerequisites

  • A recent version of the GitHub CLI (2.40 or later) installed (gh --version to verify)
  • Authenticated session: run gh auth login if you have not already
  • A GitHub Copilot license (Free, Pro, Pro+, Business, or Enterprise)
  • An internet connection (commands are processed in the cloud)
Before we install anything, let's make sure the prerequisites are in place. You need a recent version of the GitHub CLI, 2.40 or later. Run gh --version to check. You also need to be authenticated; if you have not done that yet, gh auth login will walk you through it. And of course you need a Copilot license. Any tier works: Free, Pro, Pro+, Business, or Enterprise. One thing to note is that Copilot CLI requires an internet connection. The AI processing happens on GitHub's servers, not locally. We will talk more about what goes to the cloud later in the security section.

Installing Copilot CLI

# Install the Copilot CLI extension
gh extension install github/gh-copilot

# Verify the installation
gh copilot --version

# Upgrade to the latest version
gh extension upgrade gh-copilot
Installation is a single command: gh extension install github/gh-copilot. That is it. The GitHub CLI handles downloading and configuring the extension automatically. Once installed, you can run gh copilot --version to confirm everything is in place. If you have already installed it and want the latest features, gh extension upgrade gh-copilot will pull down the newest release. This is a lightweight install; it does not modify your shell profile, add login items, or install any background services. It just extends the gh command with those two new subcommands.
Part 2

Core Commands

Now that we are installed, let's explore the two commands that make up the heart of Copilot CLI.

gh copilot suggest

# Ask Copilot to suggest a command
gh copilot suggest "find all .log files larger than 100MB"

# Specify a target type explicitly
gh copilot suggest -t shell "list all running services"

# Target type: gh (GitHub CLI commands)
gh copilot suggest -t gh "list my open pull requests"
  • Describe what you want to do in plain English
  • Copilot returns a shell command tailored to your environment
  • Target types: shell (default), gh (GitHub CLI), git
The suggest command is the one you will use most often. You describe what you want to accomplish in natural language, and Copilot returns a command you can run. By default, it generates a shell command for your current environment. You can override this with the -t flag to specify a target type: shell for general commands, gh for GitHub CLI commands, or git for Git operations. After the suggestion appears, you get three options: run the command immediately, copy it to your clipboard, or revise your request. That revision option is key. If the first suggestion is close but not quite right, you can refine your description and get a better result without starting over.

gh copilot explain

# Explain a complex command
gh copilot explain "find . -name '*.ts' -exec grep -l 'TODO' {} +"

# Explain a git command
gh copilot explain "git rebase -i --autosquash HEAD~5"

# Explain a Docker command
gh copilot explain "docker run --rm -v $(pwd):/app -w /app node:18 npm test"
  • Pass any command as a string, and Copilot breaks it down piece by piece
  • Great for auditing scripts, reviewing unfamiliar commands, or learning new tools
The explain command is the mirror of suggest. Instead of generating a new command, it takes a command you already have and explains what it does. This is incredibly useful in several situations: when you find a command in a Stack Overflow answer and want to understand it before running it, when you are reading a colleague's script and encounter unfamiliar flags, or when you are learning a new tool like kubectl or terraform. The explanation breaks the command into its individual parts, explaining each flag, argument, and operator. It is like having a patient mentor who never gets tired of questions.

The Suggest Workflow

  • Step 1: Describe your goal in plain English
  • Step 2: Copilot returns a suggested command
  • Step 3: Choose an action:
    • ✅ Execute the command directly in your shell
    • 📋 Copy to clipboard for use elsewhere
    • 🔄 Revise your request and get a new suggestion
  • Iterate: The Revise option lets you refine your description while preserving context from previous rounds
Let's walk through the full suggest workflow. You start by typing your goal in plain English. Copilot processes that and returns a command. At this point, you have three choices. You can execute the command right there in your terminal. You can copy it to your clipboard if you need to paste it into a script or a different terminal. Or you can revise your description, which sends a follow-up request with additional context. The revise option is very powerful for iterative refinement. For example, you might start with "compress this folder" and then revise to "compress this folder excluding node_modules and output as tar.gz." Each revision builds on the previous context, so Copilot understands what you have already tried.

Interactive vs. One-Shot Mode

Interactive Mode

Run gh copilot suggest without arguments. Copilot prompts you to choose a target type and enter your description step by step.

Best for: exploration, learning new tools, and iterative refinement.

One-Shot Mode

Pass your description inline: gh copilot suggest "your request". Copilot accepts the initial prompt directly, reducing interactive steps, but may still offer follow-up options like Revise, Execute, or Copy.

Best for: speed and when you know exactly what you need.

Copilot CLI supports two interaction patterns. In interactive mode, you run the command with no arguments, and Copilot walks you through a guided flow: it asks you to pick a target type, then enter your description. This is great when you are exploring or not sure exactly what you need. In one-shot mode, you pass your description inline as a quoted string, which skips the initial prompts. You may still see follow-up options like Execute, Copy, or Revise, so it is not fully non-interactive. One-shot mode is faster and most experienced users start in interactive mode when learning a new tool, then graduate to one-shot mode once they know the patterns.

Target Types and Shell Awareness

# Shell command (Copilot auto-detects your active shell)
gh copilot suggest -t shell "get disk usage for the C drive"
# PowerShell result: Get-Volume -DriveLetter C | Select-Object DriveLetter, SizeRemaining, Size
# Bash result:       df -h /

# Git-specific suggestion
gh copilot suggest -t git "undo my last commit but keep the changes"
# Result: git reset --soft HEAD~1

# GitHub CLI-specific suggestion
gh copilot suggest -t gh "list my open pull requests"
# Result: gh pr list --author @me --state open
The -t flag selects the command category: shell, git, or gh. It does not select a shell flavor. When you use -t shell, Copilot auto-detects your active shell and generates the right syntax. If you are in PowerShell, you get Get-Volume; if you are in bash, you get df -h. This is not just syntax differences; it is genuinely different tools and approaches optimized for each environment. The -t git flag restricts output to Git commands, and -t gh restricts it to GitHub CLI commands. These categories help Copilot narrow its suggestions to the right domain.
Part 3

CLI vs. VS Code Chat

Now let's tackle one of the most common questions: how does Copilot CLI relate to Copilot Chat in VS Code?

Key Differences

Copilot CLI

  • Terminal-first experience
  • No file or project context
  • Focused on commands
  • Works over SSH, in CI, anywhere with a shell
  • Lightweight; no editor required

VS Code Chat

  • Editor-first experience
  • Full workspace context via #file, @workspace
  • Focused on code
  • Integrated with VS Code features (diff, apply, etc.)
  • Richer interaction model
This is a question that comes up in every session, so let's be clear about it. Copilot CLI and VS Code Chat are complementary tools, not competitors. CLI lives in your terminal. It has no awareness of your files, your project structure, or your open tabs. It is laser-focused on commands: generating them, explaining them, and helping you run them. VS Code Chat, on the other hand, is deeply integrated with your editor. It can see your files, understand your project, apply code changes, and leverage agents like @workspace. The rule of thumb is simple: if your task is about running a command, use CLI. If your task is about writing or understanding code, use Chat.

When to Use Which

  • Use CLI when: you need a git command, a Docker command, a find/grep/awk pipeline, kubectl operations, or any shell one-liner
  • Use Chat when: you need to generate code, refactor a function, write tests, understand a file, or interact with your project context
  • Use both when: you are working in VS Code's integrated terminal. Chat handles your code; CLI handles your commands
Let me make this concrete with examples. Reach for CLI when you need to remember that obscure git rebase flag, figure out the right docker compose syntax, or build a complex find pipeline. These are command-centric tasks where file context does not help. Reach for Chat when you need to refactor a function, generate unit tests, or understand what a piece of code does. These are code-centric tasks where project context is essential. And here is the real power move: use both together. When you are in VS Code's integrated terminal, you can use CLI for your terminal commands and switch to the chat panel for code questions. They cover each other's blind spots perfectly.

Combining CLI and Chat

  • Scenario: You need to set up a new Docker container and write the application code
  • Step 1: Use CLI to generate the docker run command with the right flags and volume mounts
  • Step 2: Use Chat to scaffold the application code that will run inside the container
  • Step 3: Use CLI to generate the docker build and docker push commands for deployment
  • Each tool handles what it does best; you stay in the flow
Let's walk through a real scenario. Say you are setting up a new containerized application. You need the right Docker commands and the application code itself. Start in your terminal with CLI: gh copilot suggest "run a node 20 container with the current directory mounted and port 3000 exposed". CLI gives you the exact docker run command. Now switch to Chat to scaffold your Express server, routes, and middleware. Chat can see your project files and generate code that fits. When you are ready to deploy, go back to CLI for the docker build and docker push commands. Each tool handles what it does best, and you never leave your development environment.

Practical Workflows

Theory is great, but let's get practical. Here are real workflows where Copilot CLI saves significant time.

Git Operations

# Complex git scenarios made simple
gh copilot suggest -t git "squash the last 3 commits into one"
gh copilot suggest -t git "cherry-pick a range of commits from the feature branch"
gh copilot suggest -t git "find which commit introduced a bug between v1.0 and v2.0"
gh copilot suggest -t git "show all files changed in the last 7 days by a specific author"
Git is probably the number-one use case for Copilot CLI. We all know the basics, but git has hundreds of commands and thousands of flags. How often do you find yourself searching the web for the right git rebase syntax or the exact flags for git log? With CLI, you just describe what you want. "Squash the last 3 commits." "Cherry-pick a range." "Find which commit introduced this bug." These are all real requests that would normally require a trip to Stack Overflow or the git documentation. CLI gives you the answer instantly, in context, and you can execute it right there. The -t git flag ensures the suggestions are git-specific rather than generic shell commands.

Docker & Infrastructure

# Docker workflows
gh copilot suggest "remove all stopped containers and dangling images"
gh copilot suggest "run postgres 16 with a persistent volume and custom password"

# Kubernetes
gh copilot suggest "scale the api deployment to 5 replicas in the production namespace"
gh copilot suggest "get logs from all pods matching label app=worker in the last hour"

# Azure / Cloud
gh copilot suggest "create a resource group in eastus2 named my-project-rg"
Docker and infrastructure commands are another area where CLI really shines. Docker syntax can be verbose and easy to get wrong, especially with volume mounts, environment variables, and network settings. Instead of memorizing flags, just describe your intent: "run Postgres 16 with a persistent volume and a custom password." CLI generates the full command with all the right flags. The same applies to Kubernetes. Kubectl commands can get very long, especially when you are filtering by labels, namespaces, and output formats. And for cloud CLI tools like az, aws, or gcloud, CLI can generate the right invocation without you having to remember which subcommand group the operation belongs to.

File System & Text Processing

# Find and filter
gh copilot suggest "find all TypeScript files modified in the last 24 hours"
gh copilot suggest "recursively search for TODO comments excluding node_modules"

# Text processing
gh copilot suggest "extract unique email addresses from a CSV file"
gh copilot suggest "count lines of code by file extension in this directory"

# System tasks
gh copilot suggest "check which process is using port 3000"
gh copilot suggest "monitor CPU and memory usage of a specific process"
File-system and text-processing tasks are where many developers hit a wall with traditional commands. How do you write a find command that excludes certain directories? What is the right awk syntax for extracting a column from a CSV? These are commands most of us use infrequently enough that we forget the syntax between uses. With CLI, you describe the outcome you want, and you get a working command. "Extract unique email addresses from a CSV" might produce an awk or grep pipeline that would take several minutes to construct manually. System administration tasks like checking port usage or monitoring processes are equally accessible.

GitHub CLI Integration

# Use -t gh for GitHub CLI-specific suggestions
gh copilot suggest -t gh "list all issues assigned to me across all repos"
gh copilot suggest -t gh "create a PR from my current branch with a generated title"
gh copilot suggest -t gh "list all failed workflow runs in the last week"
gh copilot suggest -t gh "add a label to all open issues matching a search query"
  • The -t gh flag restricts suggestions to gh commands specifically
  • Perfect for discovering gh subcommands you did not know existed
Here is a feature that often surprises people: Copilot CLI can generate GitHub CLI commands too. Using the -t gh flag, you restrict the suggestions to gh subcommands. This is incredibly useful because the GitHub CLI is extensive. It covers repos, issues, PRs, Actions, Codespaces, releases, and more. Most developers only use a fraction of these. With CLI, you can discover commands you did not know existed. "List all failed workflow runs in the last week" might generate a gh run list command with flags you have never used before. It is like having an expert guide to the GitHub CLI built right into your terminal.

Session Context & Model Selection

Let's look at some of the deeper features: how context works across commands and how to select models.

Session Persistence

  • Each gh copilot suggest call is a standalone request by default
  • When you revise a suggestion (choose the "Revise" option), context carries forward within that session
  • There is no persistent memory between separate invocations of the command
  • This is by design: CLI commands should be predictable and stateless
A common question is whether Copilot CLI remembers previous interactions. The answer is nuanced. Within a single suggest session, if you choose the "Revise" option, your previous request and the suggestion are included as context for the next round. So iterative refinement does work. However, once you exit that session and run a new gh copilot suggest command, there is no memory of previous sessions. Each invocation starts fresh. This is actually a feature, not a limitation. Terminal commands should be predictable and repeatable. If the same input always produces the same output, you can trust CLI in scripts and automation. Compare this to VS Code Chat, which maintains conversation history within a thread.

Model Selection

  • Copilot CLI uses a fixed model assigned by GitHub; users cannot select or change the model
  • Enterprise administrators can configure organizational policies that influence model availability across Copilot surfaces
  • The CLI does not expose a model picker or model configuration option
  • The model may change over time as GitHub updates its backend, but this is not user-controlled
Copilot CLI uses a fixed model assigned by GitHub's backend. Unlike VS Code Chat, which offers a model picker, the CLI does not let you choose or configure which model you use. Organizations on Business or Enterprise plans can set policies that influence model availability across Copilot surfaces, but there is no per-user model selection in the CLI. The model may change over time as GitHub updates its infrastructure, but that is not something you control. The key takeaway is that CLI is part of the unified Copilot platform, but its model selection is fully managed by GitHub.

Enterprise & Security

For those of you in enterprise environments, this section covers authentication, data privacy, and governance.

Enterprise SSO Authentication

  • Copilot CLI honors your organization's SAML SSO configuration
  • Authentication flows through gh auth login, the same as all GitHub CLI operations
  • Tokens are scoped to your organization and respect IP allow-lists
  • No separate login or API key required for Copilot specifically
In enterprise environments, authentication is always a top concern. The good news is that Copilot CLI uses the same authentication as the rest of the GitHub CLI. If your organization uses SAML SSO, the gh auth login flow handles it. There is no separate API key, no additional configuration file, and no second authentication step. Your personal access token or OAuth token is scoped to your organization and respects IP allow-lists and other security policies. This means your security team does not need to approve a new authentication mechanism; they just need to confirm that Copilot is enabled for your organization.

Security: What Goes to the Cloud

  • Sent to the cloud: your natural-language description and the selected shell type
  • Not sent: your file contents, environment variables, shell history, or directory listings
  • Copilot Business and Enterprise: no prompts or suggestions are retained for model training
  • All communication uses TLS encryption in transit
  • Responses are not cached on GitHub's servers
Let's be precise about what data flows to the cloud when you use Copilot CLI. When you run a suggest or explain command, your natural-language description and the target shell type are sent to GitHub's servers for processing. That is it. Copilot CLI does not send your environment variables, your shell history, your file contents, or your directory listings. On Business and Enterprise plans, GitHub provides the additional guarantee that prompts and suggestions are not retained for model training. All communication happens over TLS, so the data is encrypted in transit. This is an important distinction from VS Code Chat, which can include file content as context. CLI is inherently more private because its scope is narrower.

What Stays Local

  • Your shell session and command history remain on your machine
  • The suggested command is displayed locally; you choose whether to run it
  • GitHub's telemetry focuses on suggestion quality metrics; refer to GitHub's privacy documentation for complete details
  • Extension data is stored locally in the gh configuration directory
Equally important is understanding what stays on your machine. Your shell session, your command history, and any command output are all local. When Copilot returns a suggestion, it is displayed in your terminal. Whether you execute it, modify it, or ignore it is up to you. GitHub's telemetry focuses on suggestion quality metrics; refer to GitHub's privacy documentation for the full picture. The extension itself stores configuration data locally, in the same directory as the rest of your GitHub CLI configuration. There are no background processes phoning home. This gives you a clear mental model: your description goes up, a suggestion comes back, and everything else stays local.

Power User Techniques

Now let's level up. These techniques will help you integrate Copilot CLI into your workflow more deeply.

Aliases & Shell Functions

# Bash/Zsh aliases
alias cs='gh copilot suggest'
alias ce='gh copilot explain'

# PowerShell functions
function cs { gh copilot suggest @args }
function ce { gh copilot explain @args }

# Usage becomes:
cs "find all large files in this repo"
ce "tar -czf backup.tar.gz --exclude=node_modules ."
The first power-user technique is setting up aliases. Typing gh copilot suggest every time is verbose. Create a short alias like cs for suggest and ce for explain. In bash or zsh, add these to your shell profile. In PowerShell, add them to your $PROFILE script. Now you can type cs "find all large files" and get your suggestion instantly. This reduces the friction to near zero. Some teams standardize on aliases so everyone uses the same shortcuts, which makes pair programming and screen sharing more consistent. You can also create more specific aliases that include the target type, like alias csg='gh copilot suggest -t git' for git-specific suggestions.

Advanced Shell Functions

# Bash function with pre-configured target types
gitsuggest() {
  gh copilot suggest -t git "$*"
}

ghsuggest() {
  gh copilot suggest -t gh "$*"
}

# Usage:
gitsuggest "revert the last merge commit"
ghsuggest "close all stale issues older than 90 days"
  • Wrap CLI commands in functions that encode your team's preferred target types
  • Add to your dotfiles repository for consistent setup across machines
You can take aliases further with shell functions that encode specific workflows. A gitsuggest function that always passes -t git means you never have to remember the flag. A ghsuggest function does the same for GitHub CLI commands. Store these in your dotfiles repository and they follow you across every machine. Some teams go even further and create wrapper functions that log suggestions to a local file for auditing, or that pipe the output into a specific format. The key principle is that since Copilot CLI is just a regular command-line tool, you can compose it with all the standard Unix tools and patterns you already know.

Tips for Better Suggestions

  • Be specific: "list files" is vague; "list all .ts files modified in the last 7 days sorted by size" is precise
  • Include constraints: mention the OS, shell, or tool version if it matters
  • Use the right target type: -t git, -t gh, or -t shell to narrow the scope
  • Iterate with Revise: refine rather than starting over when the first result is close
  • Combine explain with suggest: if a suggestion looks unfamiliar, copy it and pass it to gh copilot explain
The quality of suggestions scales directly with the quality of your descriptions. Vague requests produce vague results. Instead of "list files," try "list all TypeScript files modified in the last 7 days, sorted by size descending." Include constraints when they matter: "on Ubuntu 22.04" or "using PowerShell 7." Always use the right target type flag to narrow the scope. When a suggestion is close but not perfect, use the Revise option instead of starting a new session. And here is a great combined workflow: if a suggestion contains flags you do not recognize, copy it and pass it to gh copilot explain before executing. The suggest command is interactive and does not output raw text for piping, so you will need to copy the command manually. This way you learn while you work, and you never run a command you do not fully understand.

Common Pitfalls

  • Running suggestions blindly: always review the command before executing, especially destructive operations like rm, drop, or force-push
  • Expecting file context: CLI does not see your files; if your task requires project knowledge, use Chat instead
  • Ignoring shell differences: a bash command will not work in PowerShell; use the -t flag to match your environment
  • Overloading a single request: break complex multi-step tasks into individual suggestions
Let's talk about common mistakes. The biggest one is running suggested commands without reading them first. Copilot is very good, but it is not perfect. A suggestion involving rm -rf, git push --force, or DROP TABLE deserves careful review. Always read the command, understand what it does, and consider the consequences before executing. The second pitfall is expecting CLI to know about your project. It does not see your files. If you ask "refactor the server module to use async/await," CLI will not produce useful results because it has no code context. Use Chat for that. Third, pay attention to shell differences. A command generated for bash will often fail in PowerShell and vice versa. Finally, if your task is complex, break it into smaller pieces rather than trying to get one enormous command.
