The Who

What Is Copilot CLI?

Let's start with the fundamentals. What exactly is Copilot CLI, and how is it different from what you may have used before?

Copilot CLI at a Glance

  • A standalone agentic AI that runs directly in your terminal
  • Installed via npm install -g @github/copilot; launched with the copilot command
  • Can read, modify, and execute files in your project, not just suggest commands
  • Three interaction modes: standard, plan, and autopilot; plus a programmatic interface (-p)
  • Comes pre-configured with the GitHub MCP server for direct GitHub.com integration
  • Supported on Linux, macOS, and Windows (PowerShell and WSL)
Copilot CLI is a standalone tool you install globally with npm. It is not the older gh copilot suggest and gh copilot explain extension for the GitHub CLI. This is a fundamentally different product. While the old tool could only suggest or explain commands, Copilot CLI is a fully agentic AI. That means it can read your project files, write new code, run shell commands, interact with Git, and even create pull requests on GitHub. You talk to it in natural language, and it figures out the tools, commands, and file changes needed to accomplish your task. It always asks for your approval before modifying or executing anything, so you stay in control.

Who Is Copilot CLI For?

  • Terminal-first developers who want AI assistance without leaving the command line
  • DevOps and platform engineers managing infrastructure, CI/CD pipelines, and cloud environments
  • SSH users working on remote machines where an IDE is not available
  • Automation builders who want to integrate AI into shell scripts and pipelines
  • Anyone who wants to avoid context switching between the terminal, browser, and editor
If you already love working in the terminal, Copilot CLI is built for you. It puts a powerful AI agent right where you already work. You do not need VS Code, you do not need a browser, and you do not need to switch contexts. This is especially valuable for developers who SSH into servers, work in containers, or spend significant time in CI/CD environments. Platform engineers will find it useful for infrastructure management, and automation builders can integrate it directly into scripts using the programmatic mode. The core promise is simple: stay in the terminal, let AI handle the complexity.

How Copilot CLI Works

Copilot CLI follows an agentic loop: you describe a task, Copilot proposes actions, you approve or redirect, and it executes.

flowchart LR
  A["You describe a task"] --> B["Copilot analyzes"]
  B --> C{"Tool needed?"}
  C -- Yes --> D["Request approval"]
  D --> E["Execute tool"]
  E --> F["Show result"]
  C -- No --> F
  F --> A
The architecture is an agentic loop. You type a prompt describing what you want. Copilot analyzes your request and determines the right approach. If it needs to use a tool (for example, reading a file, running a shell command, or modifying code) it asks for your approval first. Once approved, it executes the tool, shows you the result, and continues the conversation. This loop repeats until your task is complete. The key difference from the old CLI is that Copilot takes multiple actions in sequence, building on previous results, rather than returning a single command suggestion.

Getting Started

Now let's get hands-on. We will walk through prerequisites, installation, and your first launch.

Prerequisites

  • Node.js 22 or later (node --version to verify)
  • A GitHub Copilot subscription: Pro, Pro+, Business, or Enterprise
  • If your organization provides Copilot, the Copilot CLI policy must be enabled in the organization's settings
  • Windows users: PowerShell v6 or higher is required ($PSVersionTable.PSVersion to verify)
  • An internet connection (prompts are processed in the cloud)
Before we install, let's check the prerequisites. You need Node.js version 22 or later if you plan to install via npm. Run node --version to check. You also need a Copilot license. Copilot CLI is available on Pro, Pro+, Business, and Enterprise plans. If you get Copilot through an organization, the administrator must enable the Copilot CLI policy in the organization settings. Finally, you need an internet connection because prompts are processed on GitHub's servers. Important note for Windows users: Copilot CLI requires PowerShell version 6 or higher. The legacy Command Prompt is not supported. Run $PSVersionTable.PSVersion in PowerShell to verify your version.

Installing Copilot CLI

# npm (all platforms, requires Node.js 22+)
npm install -g @github/copilot

# WinGet (Windows)
winget install GitHub.Copilot

# Homebrew (macOS and Linux)
brew install copilot-cli

# Verify the installation
copilot --version
  • Multiple install methods: npm, WinGet, Homebrew, install script, or direct download
  • To update later, run the same install command again or use copilot update
You have several ways to install Copilot CLI. npm works on all platforms and requires Node.js 22 or later: npm install -g @github/copilot. On Windows, you can also use WinGet: winget install GitHub.Copilot. On macOS and Linux, Homebrew works: brew install copilot-cli. There is also an install script for macOS and Linux (curl -fsSL https://gh.io/copilot-install | bash) and you can download executables directly from the copilot-cli repository on GitHub. Once installed, the copilot command is available in any terminal session. Run copilot --version to confirm the installation succeeded. To update later, run the same install command or use copilot update.

First Launch and Trust

  • Navigate to your project directory and type copilot to begin
  • On first launch, Copilot asks you to trust the current directory
  • Three options:
    • Yes, proceed: trust for this session only
    • Yes, and remember this folder for future sessions: permanent trust
    • No, exit (Esc): cancel and exit
  • If you are not authenticated, use the /login slash command and follow the prompts
  • For CI/CD and automation, authenticate with a fine-grained personal access token (with the "Copilot Requests" permission) via the COPILOT_GITHUB_TOKEN, GH_TOKEN, or GITHUB_TOKEN environment variable
When you first type copilot, the tool asks you to confirm that you trust the directory you are in. This is a critical security step. During your session, Copilot may read, modify, and execute files in and below that directory. You have three choices. "Yes, proceed" trusts the folder for this session only. "Yes, and remember" makes it permanent so you will not be asked again from that folder. "No, exit" cancels the session entirely. Choose carefully: only use Copilot CLI in directories where you trust the contents. Do not launch it from your home directory or from folders containing sensitive data you do not want modified. If you are not yet logged in to GitHub, Copilot will prompt you. Use the /login slash command and follow the on-screen instructions. For headless or CI/CD environments, you can authenticate with a fine-grained personal access token with the "Copilot Requests" permission enabled. Export the token as the COPILOT_GITHUB_TOKEN, GH_TOKEN, or GITHUB_TOKEN environment variable (checked in that order of precedence).
The How

Interactive Mode

Now that we are installed and trusted, let's explore the interactive interface. This is the primary way most people use Copilot CLI.

The Interactive Interface

  • Type copilot to start a session; Copilot greets you and waits for input
  • Describe tasks in natural language: "refactor the error handler to use async/await"
  • Include specific files with @ followed by a relative path: Explain @src/app.js (Tab completion supported)
  • Copilot responds by reading files, proposing changes, and executing commands
  • You approve or redirect at every step that modifies or executes files
  • Send follow-up messages while Copilot is thinking to steer the conversation
  • Resume a previous session with /resume or copilot --continue for the most recent
  • Continue until you exit with Ctrl+C or /exit
The interactive interface is the default mode. Type copilot and you are in a conversation. Describe what you want in natural language and Copilot starts working. Unlike the old gh copilot suggest which returned a single command, this tool takes multiple actions in sequence. It might read a file, analyze the code, modify it, run your test suite, and then fix any failures. At every step that involves modifying or executing files, Copilot asks for your approval. You can include specific files in your prompt using @ followed by the relative path, for example Explain @config/ci/ci-required-checks.yml. Tab completion helps you find the right file. You can also steer the conversation while Copilot is thinking by sending follow-up messages or queuing additional instructions. When you reject a tool approval, you can provide inline feedback so Copilot adapts its approach without stopping entirely. Sessions can be resumed later with /resume or from the command line with copilot --resume. To quickly resume the most recent local session, use copilot --continue. You can even bring a coding agent session from GitHub.com into your local CLI.

Interaction Modes

  • Press Shift+Tab to cycle between three modes: standard, plan, and autopilot
  • Standard (ask/execute): Copilot acts on your prompt immediately, requesting tool approval at each step
  • Plan mode: Copilot analyzes your request, asks clarifying questions, and builds a structured implementation plan before writing any code
  • Autopilot mode: Copilot completes multi-step tasks autonomously without requiring approval at each step (use --autopilot on the command line)
  • Review and approve plans before execution; use --max-autopilot-continues to limit autonomous runs
  • Best practice: use plan mode for complex tasks, autopilot for trusted repetitive workflows
The interactive interface has three modes, cycled with Shift+Tab. Standard mode is the default: Copilot acts immediately on your prompt and asks for tool approval at every step. Plan mode pauses before execution. Copilot analyzes your request, asks clarifying questions to understand the scope, and builds a structured plan. You review the plan, adjust it if needed, and then give the go-ahead. This is invaluable for complex tasks where you want to verify intent before any code is written. The third mode is autopilot. In autopilot, Copilot works through multi-step tasks with greater autonomy, continuing to the next step without waiting for your approval each time. You can enable autopilot from the command line with --autopilot and limit the number of autonomous continuation steps with --max-autopilot-continues. Use autopilot in trusted workflows where you are confident in the scope, and always combine it with --deny-tool flags to block destructive operations.

Tool Approval

When Copilot needs to read, modify, or execute a file, it requests your approval:

  • Yes: allow this tool once; Copilot asks again next time
  • Yes, and approve TOOL for the rest of the session: blanket approval for this tool type during the current session
  • No, and tell Copilot what to do differently (Esc): reject the action and redirect Copilot

Be cautious with session-wide approvals. Approving rm for the session allows Copilot to delete any file in the trusted directory without further prompts.

Tool approval is the safety mechanism at the heart of Copilot CLI. Every time Copilot wants to do something that could affect your environment (running touch, chmod, node, sed, or any other shell command) it asks you first. You have three choices. "Yes" approves the single action. "Yes, and approve for the session" gives blanket approval for that tool type for the rest of your session. This is convenient for repetitive operations, but be aware of the implications. If you approve rm for the session, Copilot can delete any file without asking again. The third option lets you reject the action and tell Copilot to try a different approach. Always think carefully before approving destructive tools for an entire session.

Slash Commands

Command Purpose
/login Authenticate with GitHub
/model Switch the AI model (default: Claude Sonnet 4.5)
/compact Manually compress conversation context
/context View token usage breakdown
/mcp View and manage connected MCP servers
/allow-all Approve all permissions: tools, paths, and URLs (alias: /yolo)
/resume Resume a previous session (or use copilot --continue for the most recent)
/feedback Submit feedback, bug reports, or feature requests
/exit End the session
Slash commands give you direct control over the session. /login handles authentication. /model lets you switch the AI model. The default is Claude Sonnet 4.5, but you can choose from other available models. Each model has a different premium request multiplier, so higher-end models cost more per prompt. /compact manually compresses your conversation history, which is useful if you are in a long session. Copilot also auto-compacts when you approach 95% of the token limit, enabling virtually unlimited sessions. /context shows a detailed token usage breakdown. /mcp lets you view and manage your connected MCP servers. /allow-all (or its alias /yolo) approves all permissions: tools, paths, and URLs. Note that this is broader than the --allow-all-tools command-line flag, which only approves tools. Use with caution. /resume lets you pick up a previous session where you left off. You can also prefix any input with ! to run a shell command directly without going through the model, for example !git status. And /feedback lets you submit feedback directly to GitHub.
The How

Programmatic Mode

Interactive mode is great for exploration, but sometimes you need a single-shot command. That is where programmatic mode comes in.

Programmatic Mode

# Pass a single prompt; Copilot completes the task and exits
copilot -p "Show me this week's commits and summarize them"

# Allow a specific tool so Copilot can run git without prompting
copilot -p "Show me this week's commits" --allow-tool='shell(git)'

# Allow all tools for fully unattended operation
copilot -p "Revert the last commit" --allow-all-tools
  • Use -p or --prompt to pass a one-off request
  • Copilot completes the task and exits; no persistent session
  • Ideal for scripts, CI/CD pipelines, and automation workflows
Programmatic mode lets you pass a single prompt directly on the command line using the -p or --prompt flag. Copilot processes the request, completes the task, and exits. There is no ongoing conversation. This is the mode you use for automation: shell scripts, CI/CD pipelines, scheduled tasks, and any scenario where you need AI assistance without human interaction. For programmatic mode to work with tasks that modify or execute files, you need to include approval flags since there is no interactive prompt to answer.

Approval Flags

Flag Effect
--allow-all-tools Approve all tools without prompting
--allow-tool='shell(git)' Approve a specific command (e.g., git)
--allow-tool='write' Approve file modification tools
--allow-tool='MCP_SERVER' Approve tools from a specific MCP server
--deny-tool='shell(rm)' Block a specific tool (overrides allow flags) 
# Allow everything except rm and force-push
copilot -p "Clean up the repo" --allow-all-tools \
  --deny-tool='shell(rm)' --deny-tool='shell(git push)'
The approval flags give you fine-grained control over what Copilot can do without asking. --allow-all-tools is the broadest option: everything is approved. --allow-tool lets you approve specific capabilities. Use shell(git) to allow git commands, write to allow file modifications, or an MCP server name to allow tools from that server. You can even specify sub-commands: shell(git push) allows push but not other git operations. The --deny-tool flag blocks specific tools and takes precedence over allow flags. You can combine them for precise control: allow everything but block rm and git push. This combination is the recommended pattern for automation since it provides capability while protecting against specific destructive operations.

Customization

One of the biggest differences between Copilot CLI and the old gh copilot tool is how deeply it can be customized. Let's explore the options.

Custom Instructions and Memory

  • Custom instructions give Copilot context about your project: how to build, test, and validate changes
  • All instruction files in your project combine automatically; no priority-based fallback
  • Copilot Memory builds persistent understanding of your repository over time
  • Memories capture coding conventions, patterns, and preferences that Copilot deduces as it works
  • Reduces the need to repeat context in every session
Custom instructions let you tell Copilot how your project works. You can describe your build process, testing conventions, coding standards, and any other context that helps Copilot deliver better results. All instruction files in your project are combined automatically, so you do not need to worry about priority or overrides. Copilot Memory takes this further. It builds a persistent understanding of your repository by storing observations it makes while working. Over time, Copilot remembers your coding conventions, preferred patterns, and project-specific knowledge. Each session gets more productive because Copilot already knows how you like things done.

MCP Servers

  • Copilot CLI comes with built-in MCP servers: GitHub (issues, PRs, Actions), Playwright (browser automation), fetch (HTTP requests), and time (utilities)
  • Connect additional MCP servers to extend Copilot's capabilities
  • Use cases:
    • Integrate domain-specific tools and workflows
    • Add contextual data from your environment
    • Customize automation to fit your team's practices
  • Manage servers with the /mcp slash command during a session
  • Control server access with --allow-tool='MCP_SERVER' and --deny-tool='MCP_SERVER(tool_name)'
MCP, the Model Context Protocol, lets you extend Copilot CLI with additional data sources and tools. The CLI ships with several built-in MCP servers: the GitHub MCP server for interacting with issues, pull requests, and Actions; Playwright for browser automation; fetch for HTTP requests; and time for time utilities. You can disable any built-in server with --disable-builtin-mcps or --disable-mcp-server. You can also add your own servers. For example, you could connect a Microsoft Learn MCP server to query official documentation, or a custom server that interfaces with your team's internal tools. During a session, the /mcp slash command lets you view connected servers and their available tools. For programmatic mode, you control MCP access with the --allow-tool and --deny-tool flags using the server name. This extensibility is what makes Copilot CLI more than a chat bot; it becomes a connected agent with access to your entire development ecosystem.

Agents, Hooks, and Model Selection

  • Custom agents: create specialized versions of Copilot tailored to specific tasks (e.g., a frontend engineer agent that follows your team's guidelines)
  • Hooks: execute custom shell commands at key points during agent execution for validation, logging, or security scanning
  • Skills: enhance Copilot's ability to perform specialized tasks with bundled instructions, scripts, and resources
  • Model selection: use /model or --model to choose from available models (default: Claude Sonnet 4.5)
  • Each prompt consumes premium requests based on the model's multiplier
Customization goes deeper than instructions. Custom agents let you create specialized versions of Copilot for specific tasks. You could have a frontend agent that knows your component library, or a database agent that understands your schema conventions. Copilot CLI also includes built-in specialized agents that it delegates common tasks to automatically. Hooks let you run custom shell commands at key points during execution, perfect for adding validation steps, logging, or security scans. Skills package up instructions, scripts, and resources to enhance specific capabilities. For model selection, the default is Claude Sonnet 4.5, but you can switch with the /model command or the --model flag. Each prompt costs premium requests, multiplied by the model's cost factor shown in the model list.

CLI vs. VS Code Chat

A common question: how does Copilot CLI compare to Copilot Chat in VS Code? They are both agentic, both powerful, and serve different contexts. Let's look at when to use each.

Key Differences

Copilot CLI

  • Terminal-native experience
  • Works over SSH, in CI, and in containers
  • Agentic: reads and modifies files via tools
  • Pre-configured GitHub MCP server
  • No editor required
  • Programmatic mode for automation

VS Code Copilot Chat

  • Editor-native experience
  • Deep VS Code integration (diff, apply, debug)
  • Visual file context via @workspace, #file
  • Extensions marketplace for additional agents
  • Rich UI with inline suggestions
  • Built-in code review UX
Both tools are agentic. Both can read files, write code, and execute commands. The core difference is the interface and where they excel. Copilot CLI is terminal-native: it works anywhere you have a shell, including SSH sessions, CI runners, and containers. It ships with the GitHub MCP server for direct GitHub.com integration and has a programmatic mode for automation. VS Code Chat is editor-native: it has deep integration with VS Code features like diffs, inline suggestions, and the extensions marketplace. It provides a visual experience with rich UI elements. The tools are complementary, not competing.

When to Use Which

  • Use CLI when: you are in the terminal, working over SSH, running automation scripts, or managing GitHub operations from the command line
  • Use VS Code Chat when: you want visual diffs, inline code suggestions, or tight integration with editor features like debugging and testing
  • Use both when: you are working in VS Code's integrated terminal; CLI handles terminal tasks while Chat handles code exploration and editor features
Here is the decision framework. If you are in the terminal, use CLI. If you are in the editor and want visual tools, use Chat. If you are in VS Code's integrated terminal, you have both available at once. Use CLI for your terminal tasks and switch to the Chat panel for visual code exploration. The pattern most experienced developers adopt is using CLI for project-wide operations, automation, and GitHub workflows, while using Chat for focused code editing and debugging within the editor. They cover different strengths perfectly.

Trust & Security

Security is not an afterthought. Let's look at the safety mechanisms built into Copilot CLI.

Security Considerations

  • Trusted directories: Copilot only operates within directories you explicitly trust; do not launch from your home directory
  • Tool approval: every file modification and command execution requires your approval by default
  • Denial flags: use --deny-tool to block specific dangerous commands even when using --allow-all-tools
  • Scope awareness: permission scoping is heuristic; GitHub does not guarantee protection of files outside the trusted directory tree
  • Risk mitigation: for automation with broad permissions, run Copilot CLI in a restricted environment such as a VM, container, or dedicated system
  • Known limitation: organization-level MCP server policies (MCP usage toggle and registry URL) are not yet enforced in Copilot CLI
Security with Copilot CLI starts with trusted directories. Only launch it from directories where you trust all the files. This means not just your code, but any executables, scripts, or configuration files Copilot might access. The tool approval system is your second line of defense: by default, every action that modifies or executes files requires explicit approval. Even when you use --allow-all-tools for automation, you can use --deny-tool to block specific dangerous commands. It is important to know that GitHub states permission scoping is heuristic. They do not guarantee that files outside trusted directories will always be protected. For high-stakes automation with broad permissions, the recommended practice is to run Copilot CLI inside a container, VM, or system with restricted permissions and network access. One current limitation: some organization-level MCP server policies are not yet enforced in the CLI, so check with your security team if MCP governance is a concern.

Productivity Techniques

Now let's level up with techniques that make Copilot CLI even more efficient.

Aliases & Shell Integration

# Bash/Zsh: quick aliases for programmatic mode
alias cpcli='copilot -p'
alias cpall='copilot --allow-all-tools -p'

# PowerShell equivalents
function cpcli { copilot -p @args }
function cpall { copilot --allow-all-tools -p @args }

# Usage
cpcli "Explain each of these scripts and offer improvements"
cpall "Run npm audit and apply fixes automatically"
Aliases reduce friction. Instead of typing copilot --allow-all-tools -p every time, create a short alias like cpall. In bash or zsh, use the alias syntax. In PowerShell, create function wrappers. Add these to your shell profile so they persist across sessions: ~/.bashrc for bash, ~/.zshrc for zsh, or $PROFILE for PowerShell. Some teams standardize aliases so everyone uses the same shortcuts, which makes pair programming and knowledge sharing easier. You can also create specialized aliases that include denial flags for safety, like an alias that allows all tools but always blocks rm and git push --force.

Tips for Effective Prompts

  • Be specific: "update the README" is vague; "rewrite the README to make it more accessible to newcomers" is actionable
  • Include constraints: mention file names, frameworks, or testing requirements when relevant
  • Use @filepath for context: include specific files in your prompt with @src/app.js
  • Use plan mode for complex tasks: press Shift+Tab before submitting multi-step requests
  • Toggle reasoning visibility: press Ctrl+T to show or hide the model's reasoning process (persists across sessions)
  • Steer the conversation: send follow-up messages while Copilot is thinking to adjust direction
  • Give feedback on rejections: when you reject a tool approval, explain why so Copilot adapts its approach
  • Reference MCP servers by name: if you know a specific server can handle a task, mention it in your prompt
The quality of results scales with the quality of your prompts. Be specific about what you want. Instead of "fix the bug," try "the login form throws a TypeError when the email field is empty; fix the validation in auth.js." Include constraints when they matter: "use vitest, not jest" or "keep the changes backward compatible with Node 18." Use the @filepath syntax to attach specific file contents to your prompt. For complex tasks, switch to plan mode so you can review the approach before execution starts. Press Ctrl+T to toggle reasoning visibility, which shows how Copilot works through complex problems. This setting persists across sessions. One of the best features is conversation steering: you can send messages while Copilot is still processing to adjust direction without waiting. When you reject a tool request, tell Copilot why so it can try a different strategy. If you have MCP servers configured, mentioning them by name in your prompt helps Copilot route the task to the right tool.
