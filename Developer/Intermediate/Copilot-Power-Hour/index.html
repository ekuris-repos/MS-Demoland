Part 1 · 5 min

GitHub Copilot Today

Let's start with the landscape. GitHub Copilot is moving fast, and the product people are using today looks very different from what shipped two years ago. This section is five minutes of grounding so we all share the same vocabulary before we go deeper.

Where Copilot Lives Today

IDE

Inline completions, Chat, Agent mode, and Next Edit Suggestions inside VS Code, JetBrains, Neovim, and Visual Studio.

CLI

A standalone agentic AI in your terminal. Reads files, writes code, runs commands, and connects to GitHub without opening an editor.

github.com

Copilot Chat on github.com, the GitHub Coding Agent that works in Issues and Pull Requests, and automated code review.

SDK

Build AI-powered applications and workflows outside the editor using the Copilot SDK and agent runtime.

Four surfaces, four different jobs. The IDE is the most familiar: inline ghost-text completions, Chat for conversation and code generation, Agent mode for multi-step autonomous tasks. The CLI is newer and more powerful than most people realize. It is a fully agentic tool that can read your project files, write code, run shell commands, and file pull requests directly from your terminal. GitHub.com has its own Copilot surfaces: Chat on the site itself, the Coding Agent that can pick up a task from an Issue and open a PR when it is done, and automated code review. And then there is the SDK for teams building AI capabilities into their own products and pipelines. This session focuses on the IDE and CLI surfaces.

One Product, Many Models

  • Copilot is backed by a catalog of large language models from OpenAI, Anthropic, Google, and more
  • Switch models mid-conversation using the model selector in the Chat panel or CLI
  • Different models have different strengths: speed, reasoning depth, context window, and credit cost
  • The model selector shows the credit cost per message so you can make an informed choice
  • Rule of thumb: fast model for simple questions, reasoning model for complex debugging or architecture

The model selector is one of the most underused levers in Copilot. Every paid plan gives you access to models from OpenAI, Anthropic, and Google. They are not all the same, and the differences matter. A fast model like GPT-4o will answer a simple "what does this function do" question in a second. A reasoning model like o3 will spend more time and more credits thinking through a complex refactoring problem but produce a significantly better result. Premium models cost more credits per message, and the selector shows you the cost before you send. Building the habit of choosing the right model for the task at hand is one of the simplest ways to improve both output quality and credit efficiency.

Part 2 · 15 min

Context Engineering

Context engineering is the skill of deliberately shaping what information Copilot has access to when it responds. Most developers think of Copilot as a chat interface where the quality of the output depends on the quality of the question. That is true, but only a fraction of the picture. The files you have open, the instructions in your repo, the plugins you have connected, the agent you are using: all of these are part of the context that shapes every single response. Mastering context engineering is how you go from "Copilot is pretty good" to "Copilot feels like it actually knows our codebase."

What Is Context Engineering?

  • Every Copilot response is generated from a context window: a fixed-size payload of text sent to the model
  • Context engineering is the practice of deliberately filling that window with the right information
  • Inputs to the context window include: open files, selected code, chat history, instruction files, skills, MCP tool results, and your prompt
  • More relevant context = better output; irrelevant or missing context = vague, generic, or wrong output
  • You do not need to type everything into the chat. Files and instructions do the work for you, automatically.

  C["Instruction files (.instructions.md)"] --> Z
  D["Skills (SKILL.md)"] --> Z
  E["MCP tool outputs"] --> Z
  F["Your prompt"] --> Z

The context window is the fundamental constraint. Every model has a maximum number of tokens it can process at once. That window is filled by everything Copilot knows about your current situation: the files you have open, the code you have selected, your chat history, any instruction files in your repo, and your actual prompt. The art of context engineering is making sure that window contains the most useful signal and the least noise. The good news: most of the heavy lifting can be automated. You write instruction files and skills once, commit them to your repo, and they are automatically injected into every relevant interaction. You do not have to remember to include context; the files do it for you.

The Four Customization Primitives

Prompts

.prompt.md files that capture reusable tasks. Run them from chat like saved commands. Share them with your team by committing to .github/prompts/.

Instructions

.instructions.md files that automatically shape every Copilot response in your workspace. Write once, applies everywhere. No effort per-prompt.

Skills

SKILL.md files that package domain knowledge agents can invoke on demand. Only loaded when relevant, so complex knowledge does not add cost to unrelated tasks.

Agents

.agent.md files that define specialized modes combining instructions, skills, and tools. A virtual team member with a specific role and persona.

Four primitives, all plain Markdown files, all committed to your repo. Prompts are things you actively run, like saved chat commands for recurring tasks. Instructions are passive and automatic, they shape every response without you doing anything per-session. Skills are domain knowledge packages that agents pull in only when they recognize the task as relevant. Agents are the orchestrators: they have their own persona, tool access, and can invoke skills. Here is the key insight about why these are called primitives: they compose. An agent can reference instruction files and declare which skills it is allowed to use. A skill can include examples and schemas. An instruction can be scoped to a specific file pattern. They are building blocks, and the interesting stuff happens when you combine them.

MCP: Giving Copilot More to Work With

  • Model Context Protocol (MCP) is an open standard for connecting AI agents to external tools and data sources
  • MCP servers expose tools, resources, and prompts that Copilot can call during a conversation
  • Copilot ships pre-configured with the GitHub MCP server (Issues, PRs, repos, code search)
  • Add third-party MCP servers in VS Code settings or the Copilot CLI config file to connect databases, Jira, Slack, internal APIs, and more
  • Each tool call returns real data into the context window; Copilot can reason over live information, not just static files
MCP is arguably the biggest force multiplier in the current Copilot ecosystem. The core idea: instead of pasting data into your prompt, you connect an MCP server that Copilot can query in real time. Ask Copilot to review a PR and it can fetch the actual diff, the CI status, and the related issue thread, all without you copying anything. Connect a database MCP server and Copilot can query your schema directly when you ask it to write a migration. Connect a Jira server and it can read the requirements ticket while it writes the implementation. Every tool call adds fresh, specific data to the context window. VS Code and Copilot CLI both support MCP configuration. In VS Code you add servers in your settings JSON. In the CLI you edit a configuration file. The process takes about five minutes per server.

Custom Instructions: Copilot That Knows Your Standards

# .github/copilot-instructions.md

## Language and Framework
- Always use TypeScript with strict mode enabled
- Prefer async/await over raw Promises or callbacks
- Use functional components with hooks (no class components)

## Testing
- Write tests using Vitest, not Jest
- Every public function must have at least one test

## Style
- Use single quotes for strings
- Maximum line length: 100 characters
  • Copilot reads this file and applies the rules to every interaction in the repo, automatically
  • Use applyTo in scoped .instructions.md files to activate rules only for specific file patterns
  • Enterprise orgs can define organization-wide instructions that apply across all repositories
This is the highest-ROI change most teams can make today. Create a file called copilot-instructions.md in your .github directory and Copilot reads it before every response in that repo. You are encoding your team's coding standards into a file the AI actually respects. Every developer gets consistent guidance from day one without having to know what to ask for. The scoped variant, an .instructions.md file with an applyTo glob pattern, lets you be more targeted. Testing rules only activate for test files. React component rules only activate for component files. This keeps instructions focused and avoids loading irrelevant content. For Enterprise customers, organization-wide instructions let admins set baseline expectations across every repo in the org. A practical tip: start with five to ten rules, commit them, and add more as you discover places where Copilot's default behavior does not match your team's preferences.

Context Engineering: Decision Framework

Situation Reach for Why
You type the same detailed prompt every morning Prompt file Save it once, share it with your team, run it from the picker
Copilot keeps generating code that breaks your team's conventions Instruction file Always-on, zero per-session effort, applies to every response
Your deployment procedure is a 20-step wiki nobody reads Skill Agents pull it in on demand; it is only loaded when relevant
You want a security reviewer who uses your specific standards Agent A virtual team member with its own persona, tools, and methodology
Copilot needs live data from an external system MCP server Tool calls fetch real-time information into the context window
The framework in one slide. Repeating a task? Prompt file. Enforcing a standard? Instruction file. Documenting expertise? Skill. Need a specialist? Agent. Need live data? MCP server. The most common mistake I see is people trying to solve all five of these with longer and longer prompts typed directly into chat. That works once. It does not scale. These five primitives are how you turn a personal productivity tool into a team productivity platform.
Part 3 · 15 min

Token Optimization

Token optimization is the practice of getting the most value out of every credit you spend. The goal is not to use Copilot less. The goal is to be deliberate. A well-scoped prompt on a well-chosen model with clean context produces a better result than a vague prompt on a premium model with a cluttered context window, and it costs less.

What Drives Credit Spend

  • 1 AI credit = $0.01 USD. Every billed interaction consumes input, output, and sometimes cached tokens.
  • Token usage is converted to credits at the model's published rate. Premium models cost more per token.
  • What is billed: Chat messages, CLI sessions, cloud agent tasks, code review
  • What is not billed from credits: Inline completions and Next Edit Suggestions (included in paid plans)
  • Business and Enterprise seats contribute to a shared, pooled credit balance
graph LR
  A["Input tokens (your context)"] --> C["Credit charge"]
  B["Output tokens (Copilot's response)"] --> C
  D["Model tier"] --> C
One credit is one cent. Every interaction that consumes tokens draws from that pool. Input tokens are what you send: your prompt, your open files, your instruction files, your chat history. Output tokens are what the model returns. Both cost credits, and the rate varies by model. This means your context engineering decisions from Part 2 directly affect your spending. More context is not always better. The right context is better. If your instruction file is three thousand words of guidance that only fifteen percent of it applies to the current task, you are burning tokens on noise. That is the central insight of token optimization: precision is cheaper than volume.

Context Hygiene

  • Close irrelevant tabs. Copilot reads neighboring open files as context. Every unnecessary tab adds noise and costs tokens.
  • Select before you ask. Highlight the specific function or block you are asking about instead of leaving the entire file in context.
  • Start a new chat for unrelated tasks. Chat history accumulates. An old debugging conversation pollutes context for a documentation request.
  • Keep instruction files focused. Scoped applyTo patterns prevent testing instructions from loading during an architecture discussion.
  • Avoid pasting large blobs. Use @workspace or #file: references so Copilot can fetch precisely what it needs.
Context hygiene is not glamorous but it is effective. Think of your context window as a whiteboard in a meeting room. If the whiteboard is covered with notes from twelve previous meetings, the current meeting suffers. The simplest habit: before you ask a complex question, close the tabs that are not relevant, select the specific code you care about, and start a fresh chat thread. Thirty seconds of setup saves tokens and usually gets you a better answer on the first try rather than two or three follow-ups. The tab closure point is frequently underestimated. In a busy VS Code session with twenty files open, a significant fraction of your context window is consumed by files you opened an hour ago and forgot about.

Prompt Discipline

Be specific about scope

Tell Copilot exactly what it should and should not touch. "Refactor the error handler in api/routes.ts, do not change the interface" beats "clean up my code."

State your constraints upfront

Tell Copilot what framework, version, or pattern you are using before it makes assumptions. Assumptions generate output tokens that you have to correct.

Request the format you need

Ask for a function, not an explanation of a function. Ask for a diff, not a description of what changed. Output tokens cost just as much as input tokens.

One task per prompt

Multi-goal prompts lead to long, diffuse responses. Split complex work into sequential focused prompts. Each one is easier to verify and cheaper to correct if wrong.

Prompt discipline is about respecting both your time and your credit balance. Vague prompts produce long, hedging responses that require follow-up, which means you are paying for the correction round-trip on top of the original vague answer. The four disciplines on this slide are worth turning into a checklist until they become habit. Scope: what exactly, in which file. Constraints: what versions, patterns, and frameworks are in play. Format: ask for the artifact you actually need. Atomicity: one task, one prompt. I have seen teams cut their credit spend by thirty percent with nothing other than prompt discipline. The output quality went up at the same time.

Model Selection as a Cost Lever

Task type Model tier Rationale
Quick syntax question, code lookup Fast (GPT-4o, Claude Haiku) Speed matters more than depth; premium models add no value here
Refactoring, feature implementation Balanced (Claude Sonnet, GPT-4o) Good reasoning at moderate cost; right for most daily tasks
Architecture review, complex debugging Reasoning (o3, Claude Opus) Multi-step reasoning justifies the higher cost; faster to the right answer
Large file analysis, codebase search Large context (Gemini) Big context windows reduce the need to summarize and re-prompt
Model selection is the most direct cost lever you have. The credit cost difference between a fast model and a reasoning model for the same task can be an order of magnitude. Paying that premium is absolutely worth it when you are debugging a race condition across a distributed system. Paying it to find out what the Array.map method returns is not. Build the habit of asking yourself: "Does this task actually need reasoning, or do I just want the best model because it feels safer?" Often, the fast model is more than sufficient and returns the answer in a third of the time.

Token Anti-Patterns to Avoid

  • Conversational sprawl: Using a single chat thread all day accumulates history that is irrelevant to your current task. Start fresh threads for new tasks.
  • Overkill model selection: Running every request through a premium reasoning model. Match the model to the complexity of the task.
  • Over-loaded instruction files: A five-thousand-word copilot-instructions.md injected into every single interaction. Scope long guides to skills or agents where they are only loaded when relevant.
  • Undirected agent mode: Asking an agent to "improve my code" without a defined scope. Agents loop, generating significant token spend without a clear stopping condition.
  • Redundant context: Pasting code into chat that is already in an open file Copilot can already see. Use #file: references instead.
These are the five anti-patterns I see most often in team rollouts. Conversational sprawl is the sneakiest: a developer starts a session at 9am and is still in the same chat thread at 5pm. By afternoon, the context is full of morning work that has nothing to do with the current task. The model is spending tokens processing irrelevant history. The fix is trivially easy: Ctrl+L to start a new thread. Over-loaded instruction files are also common. Teams write a comprehensive guide for Copilot and put it all in copilot-instructions.md. That file is injected into every single interaction, including quick inline completions. The right structure is a lean copilot-instructions.md with the fundamentals, and Skills for the detailed domain knowledge that only needs to load when it is actually relevant.

Quick Optimization Checklist

  • Before asking: close irrelevant tabs, select the relevant code block, start a new thread if context is stale
  • When writing a prompt: scope it to a specific file and task, state your constraints, specify the output format
  • When choosing a model: default to a balanced tier; upgrade to reasoning only when the task needs it
  • For team scale: keep copilot-instructions.md lean; move deep expertise into Skills; use scoped applyTo patterns
  • For agent tasks: describe a clear stopping condition; review plans in plan mode before execution
Five habits. Before asking: clean your context. When writing: be specific. When choosing a model: match complexity to cost. For the team: keep instructions lean and use skills for depth. For agent tasks: define what done looks like before you start. Print this slide, put it next to your monitor, and follow it for two weeks. Your output quality will go up and your credit consumption will go down. Those two outcomes are not in tension; they are the same outcome. Better context produces better results in fewer tokens.
Part 4 · 15 min

Live Demo: Copilot CLI

Demo time. We are going to use Copilot CLI to show three things in fifteen minutes: terminal-native development without opening an editor, parallel workstreams using the Open in Editor technique, and a GitHub integration task. Everything I do here you can replicate on your own machine tonight.

Demo: Install and Launch Copilot CLI

# Install (requires Node.js 22+)
npm install -g @github/copilot

# Or use WinGet on Windows
winget install GitHub.Copilot

# Verify
copilot --version

# Navigate to your project and launch
cd my-project
copilot
  • On first launch, Copilot asks you to trust the current directory. Confirm if you want it to read and write files there.
  • Sign in with /login if prompted
  • Press Shift+Tab to cycle between standard, plan, and autopilot modes
Install takes about thirty seconds. npm install -g @github/copilot, then copilot to launch. You will be asked to trust the directory the first time. This is a meaningful security prompt: in the trusted directory, Copilot can read, write, and execute files. Only trust directories you are comfortable with. The three interaction modes are important. Standard mode is like chat: you prompt, it acts, it asks for approval before each tool use. Plan mode is deliberate: Copilot analyzes the task, asks clarifying questions, and builds a step-by-step plan for you to review before touching a single file. Autopilot completes multi-step tasks without stopping for approval at each step. Use plan mode when you are not certain about scope. Use autopilot only in trusted, well-understood workflows.

Demo: Terminal-Native Development

  • Reference specific files with @: Explain the error handling in @src/api/routes.ts
  • Ask Copilot to make changes: Add input validation to the createUser handler in @src/api/users.ts
  • Copilot shows the proposed edit, asks for approval, then writes the file
  • Run your tests from the same session: Run the test suite and fix any failures
  • Copilot executes the test command, reads the output, diagnoses failures, and proposes fixes
sequenceDiagram
  participant Dev as You
  participant CLI as Copilot CLI
  participant FS as File System
  Dev->>CLI: "Add input validation to @src/api/users.ts"
  CLI->>FS: Read users.ts
  CLI->>Dev: Propose edit (approval required)
  Dev->>CLI: Approve

  Dev->>CLI: "Run the test suite and fix any failures"
  CLI->>FS: Execute npm test
  CLI->>Dev: Diagnose failures, propose fixes
This is the core workflow. You describe what you want in plain English, reference files with the @ prefix, and Copilot handles the rest. The critical difference from the old CLI is that it can iterate. If the test suite fails after it writes the code, it reads the failure output and keeps going. You do not have to copy error messages into a chat window and ask what they mean; it already knows. Tab completion works for file paths in the @ references, which makes navigation fast. One important note: Copilot requests approval before each tool use in standard mode. You are always in control of what actually runs. Only approve tool calls that look correct.

Demo: Parallel Workstreams

  • LLM inference has inherent latency: queue time plus token generation time
  • A single Chat panel means idle time while the model works
  • In VS Code: click "Open in Editor" (pencil icon at top of Chat) to pop the current session into a tab
  • Open Chat again: you now have a second, independent session running in parallel
  • In the CLI: open a second terminal window and launch copilot from a different directory or with a focused task
gantt
  title Parallel vs Sequential Copilot Sessions
  dateFormat X
  axisFormat %s

  section Sequential
  Write tests          :a1, 0, 3
  Scaffold component   :a2, after a1, 3

  section Parallel

  Draft docs           :b3, 0, 3
This is one of the most practical productivity techniques in this entire session. Copilot is fast, but it is not instant. While it is writing your unit tests, you can open a second session and ask it to scaffold the component those tests will cover. While that one is running, open a third and draft the documentation. Three tasks run in roughly the same time as one. The pattern works in VS Code using the Open in Editor button to create independent editor-tab sessions. It works in Copilot CLI by opening multiple terminal windows. The only cost is managing the output from all three streams. Start small: two parallel sessions. Once that feels natural, add a third. This is particularly powerful for the "write code, write tests, write docs" cycle that usually runs sequentially.

Demo: GitHub Integration via MCP

  • Copilot CLI ships with the GitHub MCP server pre-configured
  • From the terminal, ask it to interact with your GitHub repositories directly:
# List open PRs needing review
What pull requests are open in my-org/my-repo that are ready for review?

# Investigate a failing CI run
The latest CI run on PR #142 failed. Diagnose the failure and suggest a fix.

# Create a PR from local changes
I've made the changes we discussed. Create a PR with a descriptive title and body
summarizing what changed and why.
  • No browser switching required: everything stays in the terminal
  • Copilot fetches live GitHub data via the MCP server and reasons over it alongside your local files
The GitHub MCP server is configured out of the box. That means Copilot CLI can query Issues, Pull Requests, your repo, and your CI results the moment you install it. Ask it to find open PRs that need review and it will list them with summaries. Ask it to diagnose a failed CI run and it will fetch the log, identify the root cause, and suggest a fix. Ask it to create a PR from your local changes and it will write a title and body based on the diff. This is the "stay in the terminal" promise made concrete: the feedback loop between local development and GitHub never requires you to open a browser tab.

Key Takeaways

  • Copilot is not just an IDE plugin anymore. The CLI, SDK, and github.com surfaces each unlock different workflows; know which one fits your task.
  • Context engineering is the core skill. Instruction files, skills, agents, and MCP servers let you automate the context that previously lived in your head.
  • Precision beats volume. A focused prompt on the right model with clean context outperforms a vague prompt on a premium model every time.
  • Parallelize deliberately. Multiple independent Copilot sessions running concurrently collapse the time for tasks that used to be sequential.
  • Start small, iterate fast. Create one instruction file this week. Add one prompt file. Try the CLI on one task. Build the habit before adding complexity.
Five takeaways to close. Copilot is a platform now, not a plugin. Context engineering is the skill that separates effective users from exceptional ones. Precision beats volume, both in output quality and credit spend. Parallelism is free if you build the habit. And start small: one instruction file, one prompt file, one CLI session. Those three changes are achievable this week and will make a measurable difference by Friday.
