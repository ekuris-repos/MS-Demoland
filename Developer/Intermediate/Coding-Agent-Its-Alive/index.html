Part 1

Meet the Coding Agent

Let's start with the fundamentals. What exactly is the Coding Agent, and where does it fit in the Copilot ecosystem?

What Is the GitHub Coding Agent?

  • A cloud-based autonomous agent that runs entirely on GitHub's infrastructure
  • Receives a GitHub Issue as input, then plans, codes, and opens a pull request
  • Operates in its own secure, ephemeral environment (a fresh VM per session)
  • Uses your repository's context: code, docs, instructions, and setup files
  • Available on Copilot Pro, Business, and Enterprise plans
The GitHub Coding Agent is a cloud-based autonomous agent that runs entirely on GitHub's infrastructure. You do not need VS Code open; you do not need your laptop running. You assign an issue to the agent, and it spins up a fresh virtual machine, clones your repository, reads the issue description, plans its approach, writes code, and opens a pull request. Each session runs in an isolated, ephemeral environment. That means the agent starts clean every time and cannot persist state between sessions. It has access to your repository's code, documentation, and instruction files, but nothing outside that boundary. The Coding Agent is available on Copilot Pro, Business, and Enterprise plans.

Coding Agent vs. VS Code Agent Mode

Coding Agent (Cloud)

  • Runs on GitHub's infrastructure
  • Triggered by issue assignment
  • Works asynchronously; you can close your laptop
  • Opens a PR when finished
  • No access to local files or tools

Agent Mode (VS Code)

  • Runs locally inside your editor
  • Triggered by chat prompts
  • Works interactively; you guide it in real time
  • Edits files in your workspace directly
  • Full access to local tools, MCP servers, and terminal
This is where people often get confused, so let's be explicit. The Coding Agent and VS Code agent mode are both "agentic" experiences, but they run in completely different environments. The Coding Agent is cloud-based. It runs on GitHub, is triggered by assigning an issue, and works asynchronously. You can literally close your laptop and come back later to find a pull request waiting for you. VS Code agent mode is local. It runs inside your editor, is triggered by chat prompts, and you interact with it in real time. It can access your local files, run terminal commands, and use MCP servers. Think of it this way: VS Code agent mode is your hands-on workshop; the Coding Agent is your off-hours assistant. They complement each other, and we will cover how to hand work between them later in this session.

How the Coding Agent Works

  • 1. Issue assignment - You assign an issue to @github-copilot or use the Copilot button
  • 2. Environment setup - The agent spins up a VM and runs your copilot-setup-steps.yml
  • 3. Planning - It reads the issue, your instructions, and the codebase to form a plan
  • 4. Implementation - The agent writes code, runs tests, and iterates on failures
  • 5. Pull request - It opens a PR with a description, links to the issue, and a summary of changes
  • 6. Review loop - You review, leave comments, and the agent can address feedback
Let's walk through the lifecycle step by step. It starts when you assign an issue to the Coding Agent, either by tagging @github-copilot in a comment or using the "Assign to Copilot" button on the issue. The agent then spins up a fresh virtual machine and runs your copilot-setup-steps.yml file to install dependencies and build the project. Next, it enters a planning phase where it reads the issue description, your custom instructions, and the relevant parts of your codebase. Then it implements the changes: writing code, running tests, and iterating if tests fail. When it is satisfied, it opens a pull request that links back to the original issue and includes a summary of what it did and why. Finally, you review the PR. If you leave comments requesting changes, the agent can pick those up and push additional commits. It is a full review loop, not just a one-shot code dump.
Part 2

Triggering the Coding Agent

Now that we understand what the Coding Agent is, let's look at how to actually trigger it and set it up for success.

Assigning Issues to Copilot

  • Option 1: On the issue page, click the "Assign to Copilot" button in the sidebar
  • Option 2: Comment @github-copilot on the issue with instructions
  • Option 3: Assign the issue to Copilot from the issue assignee dropdown
  • The agent starts working within seconds and posts a status comment on the issue
  • You can track progress directly in the issue thread
There are a few ways to trigger the Coding Agent. The simplest is clicking the "Assign to Copilot" button that appears in the issue sidebar. You can also mention @github-copilot in a comment on the issue, which is useful if you want to add specific instructions beyond the issue description. A third option is selecting Copilot from the assignee dropdown, just like you would assign an issue to a teammate. Whichever method you choose, the agent starts working within seconds and posts a status comment on the issue so you can track its progress. It will update that comment as it moves through planning, implementation, and PR creation.

Writing Effective Issue Descriptions

✅ Good Issue

  • Clear, specific title
  • Describes the desired outcome
  • Lists acceptance criteria
  • References relevant files or APIs
  • Scoped to a single, well-defined task

❌ Weak Issue

  • Vague title like "Fix the bug"
  • No context about expected behavior
  • Multiple unrelated tasks in one issue
  • No mention of affected files or modules
  • Assumes knowledge the agent does not have
The quality of the issue description directly determines the quality of the agent's output. Think of the issue as your spec. A good issue has a clear, specific title and describes the desired outcome, not just the problem. It lists acceptance criteria so the agent knows what "done" looks like. It references specific files, functions, or APIs when relevant. And critically, it is scoped to a single task. The agent does much better with focused changes than with sweeping refactors. On the other hand, vague issues like "Fix the bug" or "Update the UI" give the agent almost nothing to work with. Multi-part issues that bundle unrelated tasks together will also produce poor results. The agent works best when it can focus on one coherent piece of work.

Watching the Agent Work

  • The agent posts real-time updates as comments on the issue
  • You can see its plan before it starts coding
  • If the agent gets stuck, it will ask clarifying questions in the issue thread
  • You can reply to those questions and the agent will incorporate your answers
  • The session also appears in Copilot's activity feed on github.com
One of the great things about the Coding Agent is transparency. It does not just disappear into a black box. As it works, it posts real-time updates as comments on the issue. You can see the plan it formulates before it starts writing code. If it encounters ambiguity or gets stuck, it will post a clarifying question right in the issue thread. You can reply to that question, and the agent will incorporate your answer into its approach. This makes it a collaborative process, not a fire-and-forget one. You can also monitor all active sessions through Copilot's activity feed on github.com, which gives you an overview of everything the agent is working on across your organization.
Part 3

Custom Instructions

Now we get into the core configuration topic: custom instructions. This is how you shape the Coding Agent's behavior to match your organization's standards and conventions.

Organization-Level Instructions

  • Create a .github repository at the organization level (the special org-wide config repo)
  • Add a copilot-instructions.md file inside its .github/ directory
  • These instructions apply to every repository in the organization
  • Great for: coding standards, language preferences, required patterns, security policies
  • The Coding Agent reads these instructions before it starts planning
Organization-level instructions are the broadest layer of configuration. Every GitHub organization can have a special repository called ".github" which holds org-wide configuration files. Inside that repository, you create a file at .github/copilot-instructions.md. The Coding Agent reads this file before it begins planning on any repository in your organization. This makes it the right place for universal standards: coding conventions, language preferences, required patterns, and security policies. For example, if your organization mandates that all new code must include unit tests, or that SQL queries must use parameterized statements, that belongs in the org-level instructions. Think of it as the baseline that applies everywhere unless a repository overrides it.

Example: Org-Level copilot-instructions.md

# Organization Coding Standards for Copilot

## General Rules
- Write all code in TypeScript unless the repository specifies otherwise.
- Use ESLint and Prettier configurations from the repository root.
- Always include unit tests for new functions and methods.

## Security
- Never hardcode secrets, API keys, or credentials.
- Use parameterized queries for all database access.
- Validate all user input at system boundaries.

## Pull Request Standards
- Keep PRs focused on a single concern.
- Include a clear description of what changed and why.
- Reference the originating issue number.
Here is what an organization-level copilot-instructions.md file might look like. It starts with general rules: default to TypeScript, use the repo's ESLint and Prettier configs, and always include unit tests. Then it has a security section covering secrets, parameterized queries, and input validation. Finally, there are pull request standards for how the agent should structure its PRs. Notice the tone: these are clear, direct instructions. The agent follows them literally, so avoid vague guidance like "write good code." Be specific about what "good" means in your organization. This file lives in the .github repository at the org level, inside the .github directory of that repo.

Repository-Level Instructions

  • Add a .github/copilot-instructions.md file inside your repository
  • These instructions apply only to that repository
  • Use them for: framework-specific patterns, project architecture, naming conventions
  • The agent reads both org-level and repo-level instructions together
  • Repo-level instructions can specialize (not override) org-level ones
In addition to organization-level instructions, each repository can have its own copilot-instructions.md file in its .github directory. These instructions are scoped to that single repository. This is where you put project-specific guidance: the framework you use, architectural patterns, naming conventions, and any rules that are unique to this codebase. The agent reads both layers together. Org-level instructions set the baseline; repo-level instructions add specificity. For example, your org instructions might say "write TypeScript," and your repo instructions might add "use the NestJS framework with the repository pattern." They compose together rather than one replacing the other.

Example: Repo-Level copilot-instructions.md

# Project-Specific Instructions

## Architecture
- This is a Next.js 14 application using the App Router.
- API routes live in `app/api/` and use Route Handlers.
- Database access goes through Prisma; never write raw SQL.

## Testing
- Use Vitest for unit tests and Playwright for E2E tests.
- Place unit tests next to the source file: `foo.ts` → `foo.test.ts`.

## Styling
- Use Tailwind CSS utility classes. Do not create custom CSS files.
- Follow the design tokens defined in `tailwind.config.ts`.
Here is a repository-level instructions file for a Next.js project. It tells the Coding Agent exactly what framework is in use, where files should go, which testing tools to use, and how to handle styling. Notice how specific this is: "API routes live in app/api and use Route Handlers," "Place unit tests next to the source file." This level of detail helps the agent produce code that fits naturally into your existing codebase. Without these instructions, the agent would have to infer the architecture from the code alone, which works sometimes but not always. Being explicit eliminates guesswork and produces better results on the first try.

Layering: Org + Repo Instructions

  • The Coding Agent merges instructions from three sources:
  1. Organization-level .github repo → .github/copilot-instructions.md
  2. Repository-level → .github/copilot-instructions.md
  3. Issue description → task-specific context and requirements
  • Instructions compose additively; the agent sees all layers combined
  • If instructions conflict, the more specific layer (repo or issue) takes precedence
  • Keep each layer focused: org = universal, repo = project, issue = task
Understanding how these layers compose is key to configuring the Coding Agent effectively. The agent reads instructions from three sources: organization-level, repository-level, and the issue description itself. All three layers are combined into a single context for the agent to follow. If there is a conflict, the more specific layer takes precedence. An issue description can override a repo instruction, and a repo instruction can override an org instruction. The best practice is to keep each layer focused on its scope. Organization instructions set universal standards. Repository instructions define project-specific patterns. Issue descriptions provide task-specific context. When you maintain this discipline, the layers work together cleanly and the agent produces consistent, predictable output.
Part 4

Environment Setup

Custom instructions tell the agent what to do. The setup steps file tells it how to prepare the environment so it can actually build and test your code.

The copilot-setup-steps.yml File

  • Lives at .github/copilot-setup-steps.yml in your repository
  • Defines the setup commands the agent runs before it starts coding
  • Uses the same syntax as GitHub Actions steps
  • Installs dependencies, builds the project, and prepares the environment
  • Without this file, the agent cannot verify its own work by running tests or builds
The copilot-setup-steps.yml file is one of the most important configuration files for the Coding Agent. It lives at .github/copilot-setup-steps.yml in your repository. This file tells the agent what commands to run to set up the development environment before it starts writing code. It uses the same step syntax as GitHub Actions, so if you have experience with Actions workflows, this will look very familiar. The setup steps typically install dependencies, build the project, and verify that the test suite passes. This is critical because it lets the agent validate its own work. If you skip this file, the agent has no way to run your tests or build your project, which means it is coding blind. Always provide setup steps.

Example: copilot-setup-steps.yml

steps:
  - uses: actions/setup-node@v4
    with:
      node-version: "20"

  - name: Install dependencies
    run: npm ci

  - name: Build the project
    run: npm run build

  - name: Verify tests pass
    run: npm test
  • Keep setup steps fast; every second adds to the agent's turnaround time
  • Use npm ci instead of npm install for deterministic, faster installs
Here is a straightforward example for a Node.js project. It uses the setup-node action to configure Node 20, runs npm ci to install dependencies deterministically, builds the project, and then runs the test suite. The agent executes these steps in order before it starts its work, and it will also re-run the tests after making changes to verify everything still passes. One important tip: keep your setup steps fast. Every second your setup takes is added to the agent's total turnaround time. Use npm ci instead of npm install because it is faster and deterministic. Avoid heavy build steps that are not necessary for the agent's work. If your project has a lengthy compilation step, consider whether a lighter alternative exists for the agent's environment.
Part 5

Reviewing & Handoff

The Coding Agent does the initial work. Your job is to review it effectively and, when needed, hand tasks between your IDE and the cloud agent. Let's look at both sides of that workflow.

Reviewing Coding Agent Pull Requests

  • The agent opens a draft PR linked to the originating issue
  • The PR description includes a summary of the approach and changes made
  • Review it like any other PR: check correctness, style, test coverage, and edge cases
  • Leave review comments; the agent can push follow-up commits to address them
  • If the changes are fundamentally wrong, close the PR and refine the issue
When the Coding Agent finishes, it opens a draft pull request linked to the originating issue. The PR includes a description summarizing what the agent did, which files it changed, and its reasoning. Review this PR exactly as you would review a PR from a teammate. Check correctness: does the code actually solve the problem? Check style: does it follow your conventions? Check test coverage: did it add or update tests? Check edge cases: did it handle error scenarios? If you find issues, leave review comments. The agent can pick up those comments and push follow-up commits. This works just like a normal review cycle. If the approach is fundamentally wrong, the better option is to close the PR, refine the issue description with more detail, and reassign it.

Common Patterns in Agent Output

What the Agent Does Well

  • Boilerplate and scaffolding
  • CRUD endpoints and data models
  • Unit tests for existing functions
  • Documentation and README updates
  • Small, well-scoped refactors

Where to Watch Closely

  • Complex business logic with edge cases
  • Performance-sensitive code paths
  • Security-critical authentication flows
  • Cross-service integration points
  • Large-scale architectural changes
Over time, you will notice patterns in what the Coding Agent excels at and where it needs extra scrutiny. The agent is strong at generating boilerplate, CRUD endpoints, data models, unit tests for existing functions, documentation updates, and small, well-scoped refactors. These are tasks with clear patterns and limited ambiguity. Where you should review more carefully: complex business logic with lots of edge cases, performance-sensitive code, security-critical flows like authentication and authorization, cross-service integration points, and large-scale architectural changes. The agent can attempt all of these, but the risk of subtle mistakes is higher. The rule of thumb is simple: the more ambiguity in the task, the more carefully you should review the output.

Handoff: VS Code → Coding Agent

  • Start locally in VS Code agent mode to prototype and explore
  • When the approach is clear, write a detailed issue capturing your findings
  • Include file paths, function names, and acceptance criteria in the issue
  • Assign the issue to the Coding Agent to complete the implementation
  • This frees you to move on to other work while the agent finishes the task
One of the most powerful workflows is starting a task locally and then handing it off to the Coding Agent. Here is how it works. You begin in VS Code agent mode, using Copilot interactively to prototype, explore the codebase, and figure out the right approach. Once you have a clear picture of what needs to happen, you write a detailed GitHub Issue capturing everything you learned: which files need to change, which functions are involved, and what the acceptance criteria are. Then you assign that issue to the Coding Agent. Because you have already done the exploration, the issue description is detailed and specific, which means the agent is set up for success. Meanwhile, you are free to move on to other work. This pattern is especially effective for tasks that are conceptually straightforward but time-consuming to implement, like propagating a pattern across multiple files.

Handoff: Coding Agent → VS Code

  • The agent opens a PR; check out the branch locally in VS Code
  • Use VS Code agent mode or Copilot Chat to refine the agent's work
  • Run the app locally to verify behavior beyond what automated tests cover
  • Push your refinements to the same branch; the PR updates automatically
  • This "agent start, human finish" pattern combines speed with precision
The reverse handoff is just as valuable. The Coding Agent opens a PR, and you check out that branch locally in VS Code. Now you have the agent's work in your editor, and you can use VS Code agent mode or Copilot Chat to refine it. Maybe you want to adjust the styling, add a few more test cases, or tweak the error handling. You can also run the app locally to verify behavior that automated tests might not cover, like the look and feel of a UI change. Any commits you push to that branch automatically update the pull request. This "agent start, human finish" pattern is extremely productive. The agent handles the bulk of the implementation, and you add the polish and nuance that require human judgment.
Part 6

Governance & Best Practices

Let's close out the instructional content with governance topics: session management, security boundaries, and best practices for getting the most out of the Coding Agent.

Managing Sessions and Capacity

  • Each Coding Agent run consumes premium requests from your Copilot plan
  • Copilot Pro: included premium requests; Business/Enterprise: per-seat allocation
  • Sessions are ephemeral; the agent does not retain state between runs
  • Organization admins can enable or disable the Coding Agent at the org and repo level
  • Monitor usage through Copilot metrics in your organization settings
The Coding Agent is not unlimited. Each run consumes premium requests from your Copilot plan. On Pro plans, you have a set number of premium requests per month. On Business and Enterprise, the allocation is per seat. Organization admins should monitor usage through the Copilot metrics dashboard in GitHub's organization settings. Planning-wise, keep in mind that sessions are ephemeral. The agent starts fresh every time, which means it does not learn from previous runs. If you need to build on previous work, reference it explicitly in the issue description. Admins can also control access: the Coding Agent can be enabled or disabled at the organization level, and it can be turned off for specific repositories if needed. This gives you fine-grained control over where the agent is active.

Security and Trust

  • The Coding Agent has read access to the repository it is working on
  • It cannot access other repositories, secrets, environment variables, or external services
  • Each session runs in a sandboxed, ephemeral VM that is destroyed after use
  • On Business and Enterprise plans, no code is retained for model training
  • The agent's changes go through your normal review process: branch protection, CI, and code review
  • Organization policies and rulesets apply to agent-created branches and PRs
Security is understandably a top concern when giving an AI agent write access to your codebase. Here is what you need to know. The Coding Agent has read access to the repository it is assigned to, and it can push to branches in that repository. It cannot access other repositories, secrets stored in GitHub Actions, environment variables, or external services. Each session runs in a sandboxed VM that is destroyed when the session ends. On Business and Enterprise plans, no prompts or code are retained for model training. Importantly, the agent's output goes through all of your existing safeguards. Branch protection rules still apply. CI pipelines still run. Code review is still required before merging. The agent does not bypass any of your organization's policies or rulesets. It is subject to the same guardrails as any other contributor.

Best Practices for Success

  • Write detailed issues with acceptance criteria, file references, and examples
  • Scope tasks tightly; one issue per focused change, not sweeping refactors
  • Provide setup steps so the agent can build and test your project
  • Layer your instructions: org for universals, repo for project specifics
  • Review iteratively; leave comments and let the agent push follow-up commits
  • Use handoff workflows; explore locally, then delegate to the agent (and vice versa)
  • Monitor capacity; track premium request usage across your organization
Let's distill everything we have covered into actionable best practices. First, write detailed issues. The issue is the agent's spec; the more precise it is, the better the output. Second, scope tasks tightly. One issue should map to one focused change. Third, always provide copilot-setup-steps.yml so the agent can build and test your project. Fourth, layer your instructions thoughtfully: organization-level for universal standards, repository-level for project specifics. Fifth, review iteratively. Leave comments on the PR and let the agent address them rather than closing and starting over. Sixth, use handoff workflows. Start locally in VS Code to explore, then hand off to the agent for implementation, or vice versa. Seventh, monitor capacity. Track how many premium requests your organization is using so you can plan accordingly.
