Part 2

Agent Mode Deep Dive

Now that we've oriented ourselves in the codebase, let's go deeper on how agent mode actually works. Understanding the agentic loop will help you diagnose problems and steer the agent more effectively.

The Agentic Loop

  • Plan - The agent reads your prompt, gathers context, and forms a plan
  • Act - It executes tool calls: file edits, terminal commands, codebase searches
  • Observe - It reads the output of each action (compiler errors, test results, terminal output)
  • Iterate - If something failed, it adjusts and tries again automatically

The loop continues until the task is complete or the agent needs your input.

Agent mode follows a "plan, act, observe, iterate" loop. Here's what happens step by step. First, when you type a prompt, the agent reads it and gathers context. It searches for relevant files, reads your instruction files, and forms a plan. Then it starts acting. It might create a file, edit existing code, run a terminal command, or search the codebase for more information. After each action, it observes the result. Did the file save correctly? Did the terminal command produce an error? Did the tests pass? If something went wrong, it iterates. It reads the error message, adjusts its approach, and tries again. This loop keeps running until the task is done or the agent reaches a point where it needs your input. The key thing to understand: this is not a single request-response. It's a multi-step autonomous process. The agent might make ten or twenty tool calls for a single prompt. That's normal and expected. Your job is to write the prompt clearly, let the agent work, and then review the result.

Tools Available to Agent Mode

Built-in Tools

  • File operations - Create, edit, read, and delete files
  • Terminal - Run shell commands (with approval gates)
  • Codebase search - Semantic and text-based search across your workspace
  • Diagnostics - Read compiler errors and lint warnings
  • Fetch - Retrieve content from URLs

Extensible via MCP

  • MCP servers - Connect external tools and data sources
  • GitHub, Playwright, databases, custom APIs
  • Configured in .vscode/mcp.json
  • Team-shared via version control
Agent mode has access to several built-in tools. File operations let it create, edit, read, and delete files. Terminal access lets it run shell commands, though you'll be asked to approve them first. Codebase search lets it find relevant code semantically or by text matching. Diagnostics gives it access to compiler errors and lint warnings. Fetch lets it read web content. And it uses codebase search to locate related test files. Beyond the built-in tools, agent mode is extensible through MCP, the Model Context Protocol. MCP servers are external processes that expose additional tools to the agent. The GitHub MCP server gives it access to issues, pull requests, and code search. The Playwright MCP server lets it interact with web pages. You can also build your own MCP servers for custom workflows. MCP servers are configured in a .vscode/mcp.json file that you commit to your repo, so the whole team gets the same tool set. If you haven't taken the "MCP for You and Me" course yet, we'll just cover the basics here. The dedicated MCP courses go much deeper.

Approval Gates and Control

  • Terminal commands require explicit approval by default
  • File edits appear as diffs you can accept or reject
  • You can pause and redirect the agent mid-loop by typing a follow-up
  • The github.copilot.chat.agent.maxRequests setting caps the number of iterations
// Auto-approve safe commands (optional, use with care)
// Settings → github.copilot.chat.agent.autoApprove
//
// Only enable this after you understand what agent mode can do.
// Start with manual approval and relax controls as you build trust.
Agent mode is powerful, but you stay in control. Terminal commands require your explicit approval before they run. You'll see a notification showing the exact command, and you click to approve or reject it. File edits show up as diffs so you can review every change before accepting. If the agent goes in a direction you don't like, you can type a follow-up message at any time to redirect it. If you want to cap how many iterations the agent can take, the maxRequests setting lets you set a limit. There's also an auto-approve option for terminal commands. Some developers enable this for trusted workflows where they want the agent to run freely. My recommendation: start with manual approval. Learn what kinds of commands the agent runs, build confidence in the pattern, and only relax the controls once you're comfortable. This is the same approach you'd use with any automation tool. Trust but verify.
Part 3

The Mode Selector

Now let's talk about how agent mode fits into the broader picture. VS Code's Copilot Chat has three modes, and knowing when to use each one is a key skill.

Three Modes, Three Jobs

Ask

Read-only exploration. Ask questions, get explanations, explore ideas. No files are modified.

Edit

Targeted changes. Modify specific files with diffs you accept or reject individually.

Agent

Autonomous execution. Plans multi-step tasks, runs commands, iterates on errors.

The three modes are Ask, Edit, and Agent. You select them from the dropdown at the top of the chat panel. Ask mode is completely read-only. No files are touched, no commands are run. It's the right choice when you want to understand something, explore an idea, or get advice before acting. Edit mode makes targeted changes to files. You describe what you want changed, and Copilot proposes diffs that you can accept or reject individually. It's precise and controlled. Agent mode is the fully autonomous option. It plans multi-step tasks, creates and edits files, runs terminal commands, reads the output, and iterates. It's the most powerful mode but also the most hands-off. The question is: when do you use which? Here's a rule of thumb. If you're asking a question, use Ask. If you're making a specific change to a specific file, use Edit. If you're working on a task that spans multiple files and might need terminal commands, use Agent. In this course, we'll mostly be in Agent mode, but we'll switch to Ask and Edit when they're the better tool for the job.

Model Selection

  • Click the model selector next to the mode dropdown in the chat panel
  • Models differ in speed, reasoning depth, and premium request cost (shown as a multiplier)
  • You can switch models at any time, even mid-conversation
  • There is no single "best" model. Match the model to the task complexity.
The model selector is right next to the mode dropdown at the top of the chat panel. Each model has trade-offs: some are faster, some are better at deep reasoning, some cost more premium requests per message. The multiplier next to each model tells you how many premium requests each message consumes. For exploratory questions and simple tasks, a lighter model works well. For complex multi-file changes, a more capable model will save you iterations. You can switch models at any time, even in the middle of a conversation. The key thing is to experiment. Try different models on the same task and see which one gives you the best results. Over time you'll develop intuition for when to reach for which model.

Instructions for OctoCAT Supply

# .github/copilot-instructions.md

## Language & Framework
- TypeScript with strict mode for both frontend and backend
- React 18 with functional components and hooks (no class components)
- Express.js for the API layer

## Style
- Use Tailwind CSS utility classes for frontend styling
- Prefer named exports over default exports
- Use async/await, never callbacks

## Testing
- Use Vitest for all tests (not Jest)
- Co-locate test files next to source files: `foo.test.ts`
- Run tests with `make test`

## Database
- SQLite via better-sqlite3
- Always use parameterized queries (never string concatenation)
Here's what a good copilot-instructions.md looks like for this project. It covers language and framework choices, style conventions, testing standards, and database patterns. Notice how every rule is specific and actionable. "TypeScript with strict mode" is clear. "Use Vitest, not Jest" eliminates ambiguity. "Always use parameterized queries" prevents SQL injection. These instructions will apply to every Copilot interaction in this workspace. When agent mode creates a new API endpoint, it will use TypeScript. When it writes tests, it will use Vitest. When it queries the database, it will use parameterized queries. You write these rules once, and they shape every response going forward. The real skill is knowing what to put in this file. Start with the corrections you find yourself making over and over. If you keep telling Copilot "no, use Vitest, not Jest," that rule belongs in this file.
Part 5

Building the Feature

Time for the core of the mission. We're going to use agent mode to build a complete feature: a new API endpoint and the React component that consumes it. This is where agent mode really shines: multi-step tasks that span multiple files.

Writing a Good Agent Prompt

  • State the goal clearly in the first sentence
  • Specify constraints: frameworks, patterns, file locations
  • Define done: what does success look like?
  • Reference existing code: "follow the pattern in api/routes/products.ts"
// ❌ Vague
"Add a reports feature"

// ✅ Clear
"Create a new GET /api/reports/inventory endpoint that returns
current stock levels grouped by category. Follow the route pattern
in api/routes/products.ts. Include a Vitest test file.
Use the existing database connection from api/db/index.ts."
The quality of your agent mode output is directly proportional to the quality of your prompt. A vague prompt like "add a reports feature" leaves agent mode guessing about almost everything: file locations, naming conventions, data structure, testing approach. A good prompt has four parts. First, state the goal: "Create a GET endpoint that returns inventory data." Second, specify constraints: "Follow the route pattern in products.ts." Third, define done: "Include a Vitest test file." Fourth, reference existing code so the agent knows what conventions to follow. The reference is especially important. When you say "follow the pattern in products.ts," agent mode reads that file, extracts the conventions (middleware, error handling, response format), and applies them to the new code. This is how you get consistent, idiomatic output without writing a novel of instructions. Think of it this way: you're not programming a machine. You're briefing a new team member on a task. The more context you give in the briefing, the less back-and-forth you'll have.

Building the Backend

  • Step 1 - Ask agent mode to create the API route file
  • Step 2 - Watch it create the route, register it in the app, and add the database query
  • Step 3 - Let it run the tests and fix any failures automatically

Agent mode handles file creation, route registration, database queries, and test execution in a single loop. You review the output at the end.

Here's how the backend build works in practice. You write a clear prompt describing the endpoint you want. Agent mode starts its loop. It reads existing route files to understand the pattern. It creates a new route file with the endpoint. It opens the app entry point and registers the new route. It writes the database query using the existing connection module. Then it runs the tests. If a test fails, it reads the error, adjusts the code, and runs the tests again. This whole sequence might take ten or fifteen iterations. That's normal. The important thing is that each iteration builds on the previous one. You don't have to manually switch between files, remember import paths, or run test commands. Agent mode handles the mechanics; you handle the direction.

Building the Frontend

  • Step 1 - Ask agent mode to create a React component that calls the new API endpoint
  • Step 2 - Reference an existing component for styling and data fetching patterns
  • Step 3 - Let it add the route and navigation entry
"Create a React component at frontend/src/pages/InventoryReport.tsx
that fetches data from GET /api/reports/inventory and displays it
in a table. Use the same data fetching pattern as ProductList.tsx.
Add a route for /reports/inventory in the router config and a
navigation link in the sidebar."
The frontend build follows the same pattern. Write a clear prompt with a specific file path, a reference to existing code, and a definition of done. In this case, we're telling agent mode where to create the file, which API endpoint to call, which existing component to use as a pattern reference, and that it should wire up routing and navigation. Agent mode will read ProductList.tsx to understand the data fetching pattern, create the new component using the same conventions, add the route to the router config file, add a navigation link in the sidebar, and then check for any diagnostic errors. The reference to ProductList.tsx is doing a lot of heavy lifting here. Instead of describing the entire data fetching pattern in your prompt, you point to the example and let agent mode extract the pattern itself. This works because agent mode reads files, not just prompt text.
Part 6

Quality Assurance

The feature is built. Now comes the part that separates proof-of-concept from production-ready: quality assurance. Agent mode is surprisingly good at this phase.

Testing With Agent Mode

  • Agent mode can write tests, run them, and fix failures in a single loop
  • Point it at existing test files so it matches your team's testing patterns
  • Use specific language: "Write unit tests for the inventory route. Test the happy path, empty results, and invalid parameters."
  • Let the agent run make test and iterate on failures
Testing is one of agent mode's strongest workflows. Here's why: the feedback loop is tight. Agent mode writes a test, runs it, reads the output, and immediately knows whether it passed or failed. If it failed, it reads the error message, figures out the fix, and tries again. This iterate-until-green pattern is exactly how a developer would work, but agent mode does it faster because it doesn't need to switch contexts. The key to good test generation is specificity. Don't just say "write tests." Say "write unit tests for the inventory route. Test the happy path with valid data, the empty result case, and the invalid parameter case." Also reference existing test files. "Follow the pattern in products.test.ts" gives agent mode a concrete example of your team's testing style: how you set up fixtures, how you organize describe blocks, what assertions you prefer.

Iterative Refinement

  • Review the diff after each agent loop completes
  • Provide follow-up prompts for specific corrections: "The error handling should return a 400, not a 500, for invalid input"
  • Use Edit mode for surgical, single-file changes after the agent has done the heavy lifting
  • Use Ask mode to discuss trade-offs before committing to a direction
Agent mode rarely produces perfect output on the first pass, and that's fine. The real workflow is iterative. The agent builds the initial version, you review the diff, and then you refine. For broad adjustments, stay in agent mode: "The error handling should return a 400 for invalid input, not a 500." For surgical fixes to a specific file, switch to edit mode. It's more precise and doesn't trigger the full agentic loop. For design discussions, switch to ask mode: "What are the trade-offs between returning all results versus paginating?" This mode switching is a skill. Over time you'll develop a rhythm: agent mode for the initial build, ask mode for design questions, edit mode for targeted fixes. Don't be afraid to switch modes in the middle of a task. Each mode is a different tool in your toolbox.
Part 7

Multi-File Workflows

One of the reasons we chose a full-stack codebase for this course is that real features rarely live in a single file. Let's talk about how agent mode handles coordinated changes across multiple files.

Cross-Stack Changes

  • Agent mode coordinates changes across backend, frontend, tests, and configuration files
  • A single prompt can touch: API route, database query, React component, router config, test file
  • Agent mode reads the output from each change to inform the next one
// One prompt, multiple files
"Rename the 'stock' field to 'quantity' across the entire codebase.
Update the database schema, API responses, frontend components,
and all test assertions. Run the tests after to verify nothing broke."
Here's a concrete example. Suppose you need to rename a field from "stock" to "quantity" across the entire stack. In a traditional workflow, you'd manually update the database schema, every API response, every frontend component that reads the field, and every test assertion. With agent mode, you describe the change once, and the agent does the cross-file coordination. It searches for every occurrence, makes the changes, and runs the tests to verify. This is not a find-and-replace. Agent mode understands context. It knows that renaming a database field means updating the column definition, the SELECT queries, the API response mapping, the TypeScript types, and the React components that display the data. It handles each file differently based on what that file does.

When Agent Mode Struggles

  • Ambiguous goals - If you don't know what you want, the agent can't either
  • No reference code - Without examples to follow, output is generic
  • Very large changes - Break big features into smaller, testable increments
  • External service dependencies - The agent can't access services it doesn't have credentials for

When in doubt: break the task down, give the agent reference files, and iterate.

Let's be honest about where agent mode struggles. Ambiguous goals are the number one cause of poor output. If you say "make it better" or "refactor this," the agent has to guess what you mean. Lack of reference code is the second most common issue. If there are no existing patterns to follow, the agent falls back on generic conventions that may not match your project. Very large changes are risky because the agent may lose context partway through. Break big features into smaller pieces: build the API first, then the frontend, then the tests, then wire them together. External services are a hard limit. If your feature requires authentication against a third-party API and the agent doesn't have credentials, it can't test its work. In those cases, mock the external dependency and test the integration separately. These aren't failures of agent mode; they're constraints of any development workflow. The more you understand them, the better you'll be at structuring prompts that work within these limits.

Prompt Files for Common Tasks

mode: agent
description: Scaffold a new API endpoint with tests

Create a new Express route at api/routes/${input:routeName}.ts that
handles ${input:httpMethod} requests to /api/${input:routeName}.

**Requirements:**
- Follow the pattern in api/routes/products.ts
- Add the route to api/index.ts
- Create a test file at api/routes/${input:routeName}.test.ts
- Run make test to verify

Save common patterns as prompt files. Run them with a click from the / picker.

Once you've done something well in agent mode, save the prompt as a file so the whole team benefits. This example is a prompt file for scaffolding new API endpoints. Notice the input variables: routeName and httpMethod. When you run this prompt from the picker, Copilot asks you for those values and fills them in. The prompt body references existing code, specifies the file path convention, and includes a test run as the success criteria. This is the bridge between the PISA course and this one. In PISA, you learned the primitives. Here, you're using them in a real project with real conventions. A well-stocked .github/prompts folder means any developer on the team can scaffold a new endpoint, component, or test file in seconds, and the output will match the project's conventions.

Key Patterns to Remember

  • Reference existing code in every prompt. "Follow the pattern in X" outperforms lengthy descriptions.
  • Use the right mode. Agent for multi-step builds. Edit for surgical fixes. Ask for exploration.
  • Set up instructions first. Five minutes of preparation saves thirty minutes of correction.
  • Break big tasks into pieces. Backend first, then frontend, then integration.
  • Let the agent iterate on tests. The plan-act-observe loop works best with tight feedback.
  • Save good prompts as files. One developer's insight becomes the whole team's workflow.
These are the six patterns I want you to take away from today. First, always reference existing code. "Follow the pattern in products.ts" produces better output than a paragraph of instructions because agent mode can read the actual code and extract conventions. Second, use the right mode. Agent mode isn't always the answer. Sometimes Ask or Edit is more efficient. Third, set up instructions before you start building. Front-loading the preparation pays dividends throughout the session. Fourth, break big tasks into pieces. Build the backend, build the frontend, wire them together, run the tests. Each piece is a focused agent mode task. Fifth, let the agent iterate on tests. The tight feedback loop between running a test and reading the result is where agent mode excels. And sixth, save your good prompts. When you find a prompt that consistently produces great results, commit it as a prompt file. That turns individual expertise into team institutional knowledge.
