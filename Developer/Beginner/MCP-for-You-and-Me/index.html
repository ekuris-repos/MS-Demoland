Part 1

What Is MCP?

Let's start at the very beginning. What is the Model Context Protocol, and why should you care?

The Problem MCP Solves

  • AI assistants are powerful, but they're isolated - they can only see what's in their context window
  • Developers constantly switch between tools: editor, browser, terminal, issue tracker, CI dashboard
  • Before MCP, connecting an AI to an external tool required custom integrations for each tool and each AI
  • That's an N × M problem - every new tool needs a new integration for every AI host
Before MCP, AI assistants like Copilot were incredibly capable, but they could only work with information already in front of them - your open files, your chat history, and whatever you pasted in. If you wanted Copilot to look up a GitHub Issue, you had to copy the issue text and paste it into chat yourself. If you wanted it to check your CI status, same thing - manual copy and paste. And if a tool vendor wanted their platform to work with an AI assistant, they had to build a separate integration for each AI host. If you had ten tools and five AI assistants, that's fifty custom integrations. That's the N times M problem. MCP solves this with a single standard protocol.

The Model Context Protocol

  • MCP is an open standard for connecting AI assistants to external tools and data sources
  • Think of it as a USB-C port for AI - one protocol, many integrations
  • MCP servers expose tools (actions), resources (data), and prompts (templates) that any MCP-compatible host can use
  • Created by Anthropic, adopted by GitHub, VS Code, and a growing ecosystem
  • Any developer can build an MCP server - the specification is open and free
The Model Context Protocol is an open standard that creates a universal connection layer between AI assistants and the outside world. The USB-C analogy is the easiest way to think about it. Before USB-C, every device had its own proprietary charger. USB-C gave us one port that works with everything. MCP does the same thing for AI. An MCP server exposes three types of capabilities: tools, which are actions the AI can take like "create an issue" or "run a query"; resources, which are data the AI can read like "list of open issues" or "contents of a database table"; and prompts, which are reusable templates for common tasks. Any AI host that speaks MCP can use any MCP server, and any MCP server works with any compatible host. One protocol, universal connectivity. The spec was created by Anthropic and has been adopted by GitHub, VS Code, and a rapidly growing community of tool builders.
Part 2

How MCP Works

Now that you know what MCP is at a high level, let's look at the architecture. Understanding the moving parts will help you troubleshoot and eventually build your own servers.

The MCP Architecture

  • Host - The application that contains the AI (VS Code, a CLI tool, a web app)
  • Client - The component inside the host that speaks the MCP protocol
  • Server - An external process that exposes tools, resources, and prompts over MCP
+-----------------------------------------+
|  VS Code (Host)                         |
|                                         |
|   Copilot Chat  -->  MCP Client  -------+--->  GitHub MCP Server
|                                         |
|                      MCP Client  -------+--->  Database MCP Server
|                                         |
|                      MCP Client  -------+--->  Custom MCP Server
+-----------------------------------------+
Here's the architecture. There are three layers. The host is the application that contains the AI - in our case, that's VS Code. The client is the piece inside the host that actually speaks the MCP protocol - it handles the JSON-RPC messages, manages the connection lifecycle, and routes tool calls. The server is an external process that exposes capabilities. When Copilot needs to interact with GitHub, the MCP client inside VS Code sends a request to the GitHub MCP server. The server does the work - fetches issues, creates comments, whatever - and sends the result back through the client to Copilot. The beautiful thing is that VS Code can connect to multiple MCP servers simultaneously. You might have a GitHub server for issues and PRs, a database server for queries, and a custom server built by your team, all running at the same time. Each one is a separate process, and the MCP client manages them all.

What MCP Servers Expose

Tools

Actions the AI can invoke: create_issue, run_query, send_message. The AI decides when to call them based on the conversation.

Resources

Data the AI can read: file contents, database schemas, API documentation. Resources provide context without taking action.

Prompts

Reusable templates exposed by the server: "Summarize this PR," "Generate a migration." Similar to prompt files but provided by the server.

MCP servers can expose three types of capabilities. First, tools - these are actions. When the GitHub MCP server exposes a "create_issue" tool, Copilot can call it to create an issue on your behalf. The AI decides when to use tools based on the conversation context. Second, resources - these are read-only data. A database MCP server might expose your schema as a resource, so the AI can understand your data model without running any queries. Third, prompts - these are reusable templates that the server provides. A GitHub server might expose a "summarize this PR" prompt that's already optimized for working with pull request data. Not every server exposes all three. The GitHub MCP server focuses heavily on tools, while a documentation server might focus on resources. It depends on what the server is designed to do.

How Servers Connect

  • stdio - The server runs as a local process; VS Code communicates over standard input/output
  • Streamable HTTP - The server runs remotely; VS Code sends HTTP requests to a URL
  • GitHub's MCP server uses stdio - it runs locally and talks to the GitHub API on your behalf
  • Configuration lives in .vscode/mcp.json or VS Code settings
MCP servers connect to VS Code through one of two transport modes. The first is stdio, where the server runs as a local process on your machine and communicates through standard input and output. This is the most common mode and what GitHub's MCP server uses. VS Code spawns the process, sends JSON-RPC messages to its stdin, and reads responses from its stdout. The second mode is streamable HTTP, where the server runs remotely and VS Code sends HTTP requests to a URL. This is useful for servers that need to run on a shared infrastructure or that are too heavy to run locally. The configuration for MCP servers lives in a mcp.json file in your .vscode directory, or in your VS Code settings. We'll set this up hands-on in just a moment.
Part 3

Setting It Up

Enough theory - let's get the GitHub MCP server running. This is where things get fun.

Enabling the GitHub MCP Server

  • Open Settings (Ctrl+,) and search for MCP
  • Enable chat.mcp.enabled to turn on MCP support in Copilot Chat
  • Enable github.copilot.chat.codesearch.enabled for enhanced code search
  • The GitHub MCP server ships with the GitHub Copilot Chat extension - no separate install needed
  • Restart VS Code to activate the server
Setting up the GitHub MCP server is surprisingly simple. Open Settings with Ctrl+Comma and search for "MCP." You'll find a setting called "chat.mcp.enabled" - turn that on. This tells VS Code to allow MCP servers to connect to Copilot Chat. For the GitHub MCP server specifically, you also want to enable "github.copilot.chat.codesearch.enabled" which activates the enhanced code search capabilities. Here's the nice part: the GitHub MCP server is already bundled with the GitHub Copilot Chat extension. You don't need to install anything extra. Just flip the settings and restart VS Code. After the restart, the GitHub MCP server starts automatically in the background.

Verifying the Connection

  • Open Copilot Chat (Ctrl+Shift+I)
  • Click the tools icon (wrench) in the chat input area
  • You should see GitHub MCP tools listed: create_issue, search_code, list_issues, etc.
  • If tools aren't visible, check the Output panel (Ctrl+Shift+U) → select GitHub MCP
  • Common issues: missing authentication, proxy blocking the connection, or MCP not enabled
Once you've restarted VS Code, let's verify everything is working. Open Copilot Chat and look for the tools icon in the chat input area - it usually looks like a wrench or a puzzle piece. Click it, and you should see a list of available MCP tools from the GitHub server. You'll see things like "create_issue," "search_code," "list_issues," and more. If the tools aren't showing up, open the Output panel with Ctrl+Shift+U and select "GitHub MCP" from the dropdown. This is where the server logs connection events and errors. The most common issues are missing GitHub authentication, a corporate proxy blocking the connection, or the MCP setting not being enabled. If you're stuck, check those three things in order.
Part 4

Working With Issues

The GitHub MCP server exposes a bunch of tools, but the ones you'll use most often revolve around issues and pull requests. Let's see what that workflow looks like.

Reading Issues From Chat

  • Ask Copilot: "What are the open issues in this repo?"
  • Copilot calls the list_issues MCP tool behind the scenes
  • Results come back as structured data that Copilot formats into a readable summary
  • Try more specific queries: "Show me issues labeled 'bug'" or "What issues were opened this week?"
Here's where MCP gets practical. Open Copilot Chat and type: "What are the open issues in this repo?" Copilot recognizes that this question requires external data, so it calls the list_issues tool from the GitHub MCP server. The server fetches the data from the GitHub API and sends it back. Copilot then formats the results into a nice readable summary with issue titles, labels, and authors. You can get more specific too. Ask for issues by label, by assignee, by date range, or by milestone. The MCP server handles the filtering. The key insight here is that you never had to leave your editor, open a browser, or even think about the GitHub API. You just asked a question in natural language and got the answer.

Acting on Issues

  • Ask: "Add a comment to issue #42 saying we'll fix it in the next sprint"
  • Copilot calls the add_issue_comment tool with your message
  • You can also close issues, add labels, and assign people
  • Copilot always asks for confirmation before taking write actions
  • Full audit trail - every action is logged as a normal GitHub API call
Reading is great, but the real power is in taking action. You can tell Copilot: "Add a comment to issue 42 saying we'll fix it in the next sprint." Copilot calls the add_issue_comment tool, and the comment appears on the issue just like it would if you'd written it in the browser. You can close issues, add labels, change assignees, and more. And here's the important safety feature: Copilot always asks for your confirmation before taking any write action. It'll show you exactly what it's about to do, and you have to approve it. So there's no risk of accidental changes. Everything goes through the normal GitHub API, so you get the full audit trail - the activity log, email notifications, webhook events, all of it. It's not a shortcut around your process; it's a faster way to participate in it.

Code Search With MCP

  • Ask: "Find all files that import the AuthService class"
  • Copilot uses the search_code tool to search across your repository
  • Results include file paths, line numbers, and code snippets
  • Combine with chat: "Find all usages of the deprecated API and suggest replacements"
  • Works across repositories - not just the one you have open
The code search tool is another one you'll reach for constantly. Ask Copilot something like "Find all files that import the AuthService class" and it uses the search_code MCP tool to search across your repository. You get back file paths, line numbers, and code snippets - all formatted nicely in the chat. But here's where it gets really powerful: you can combine search with Copilot's reasoning. Ask "Find all usages of the deprecated API and suggest replacements" and Copilot will search the code, analyze the results, and give you a migration plan. And the search isn't limited to your current repo. The GitHub MCP server can search across any repository you have access to. This is incredibly useful for understanding how a library is used across your organization.
Part 5

The MCP Ecosystem

The GitHub MCP server is just the beginning. There's a whole ecosystem of MCP servers out there, and it's growing fast. Let's take a quick tour.

Community MCP Servers

Databases

PostgreSQL, MySQL, SQLite servers let Copilot query your data, explore schemas, and generate migrations.

APIs & Services

Jira, Slack, Linear, Notion - bring your project management and communication tools into chat.

Infrastructure

Docker, Kubernetes, AWS, Azure - manage deployments and check status from your editor.

The MCP ecosystem is exploding. There are community-built servers for databases like PostgreSQL, MySQL, and SQLite that let Copilot query your data, explore schemas, and generate migrations. There are servers for project management tools like Jira, Linear, and Notion, so you can check task status and create tickets without switching apps. Communication tools like Slack have MCP servers too, meaning Copilot can search messages and post updates. And for infrastructure, there are servers for Docker, Kubernetes, AWS, and Azure that let you manage deployments and check service health. The official MCP server directory lives at the Model Context Protocol website, and new servers are being published every day. If a tool has an API, someone is probably building an MCP server for it.

Adding a Third-Party Server

// .vscode/mcp.json
{
  "servers": {
    "postgres": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres"],
      "env": {
        "DATABASE_URL": "postgresql://localhost:5432/mydb"
      }
    }
  }
}
  • Drop a .vscode/mcp.json file in your repo
  • Define the server name, command to start it, and any environment variables
  • Commit the file and your whole team gets the same MCP servers automatically
  • VS Code starts configured servers when you open the workspace
Adding a third-party MCP server to your project is straightforward. You create a mcp.json file in your .vscode directory and define the server configuration. In this example, we're adding a PostgreSQL server. The "command" field tells VS Code how to start the server - in this case, using npx to run the package. The "args" field passes arguments. And the "env" field sets environment variables like the database URL. Once you save this file and reload VS Code, the server starts automatically. And here's the team angle: if you commit the mcp.json file to your repo, every developer who opens the workspace gets the same MCP servers configured automatically. No manual setup, no documentation to follow. They just open the project and the tools are there in Copilot Chat, ready to use.

MCP in Action

Alright, demo time. Let's bring everything together and see MCP in action. We're going to do a complete issue workflow without ever leaving the editor.

Demo: End-to-End Issue Workflow

  • Step 1: Ask Copilot to list open issues in the repository
  • Step 2: Pick an issue and ask Copilot to summarize it
  • Step 3: Ask Copilot to find related code using code search
  • Step 4: Write a fix with Copilot's help and add a comment to the issue
  • Step 5: Close the issue from chat
Here's what we're going to do. Step one: I'll ask Copilot to list all open issues in this repository. The MCP server fetches them and Copilot formats the results. Step two: I'll pick one of those issues and ask Copilot to summarize it - the description, any linked PRs, and the discussion thread. Step three: I'll ask Copilot to find related code using the search tool. If the issue mentions a specific function or file, Copilot can locate it and bring it into the conversation. Step four: I'll work with Copilot to write a fix, and then ask it to add a comment to the issue describing what we did. Step five: once we're happy with the fix, I'll ask Copilot to close the issue. Five steps, zero browser tabs. This is the workflow that MCP enables.

Demo: Combining MCP With Chat

  • Ask: "Summarize the last 5 closed issues and identify any recurring themes"
  • Copilot fetches the issues via MCP, then uses its reasoning to find patterns
  • Follow up: "Based on those themes, suggest improvements to our codebase"
  • This is the power of MCP + LLM reasoning - external data meets AI analysis
Now let me show you something that really demonstrates what MCP enables that wasn't possible before. I'll ask: "Summarize the last five closed issues and identify any recurring themes." Copilot fetches the issues through MCP, reads through the titles, descriptions, and comments, and then uses its reasoning ability to identify patterns. Maybe it notices that three out of five were related to authentication, or that there's a recurring pattern of null pointer exceptions in a specific module. Then I can follow up: "Based on those themes, suggest improvements to our codebase." Now Copilot is combining external data from GitHub with its ability to analyze code and suggest architectural improvements. This is the real magic of MCP - it's not just about fetching data, it's about giving the AI access to real-world context so its reasoning is grounded in what's actually happening in your project.

Security and Trust

  • MCP servers run with your permissions - they use your GitHub token, your database credentials
  • Write actions always require explicit confirmation in the chat
  • You control which servers are enabled - disable any server at any time
  • Review third-party servers before enabling them - check the source code and permissions
  • Enterprise teams can restrict which MCP servers are allowed via organization policies
Let's talk about security, because I know it's on your mind. MCP servers run with your permissions. The GitHub MCP server uses your authenticated GitHub token, a database server uses your database credentials. They don't get any special elevated access. Every write action - creating issues, posting comments, closing tickets - requires your explicit confirmation in the chat before it happens. You always see what's about to happen and you always approve it. You have full control over which servers are enabled. If you don't want a server running, disable it in settings. For third-party servers, I'd recommend reviewing the source code before enabling them, just like you would with any VS Code extension. And for enterprise teams, organization policies can restrict which MCP servers are allowed, giving admins control over what connects to the AI in their environment.
