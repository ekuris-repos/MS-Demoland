Part 1

What Is MCP?

Let's start at the very beginning. What is the Model Context Protocol, and why should you care?

The Problem MCP Solves

  • AI assistants are powerful, but they're isolated; they can only see what's in their context window
  • Developers constantly switch between tools: editor, browser, terminal, issue tracker, CI dashboard
  • Before MCP, connecting an AI to an external tool required custom integrations for each tool and each AI
  • That's an N × M problem: every new tool needs a new integration for every AI host
Before MCP, AI assistants like Copilot were incredibly capable, but they could only work with information already in front of them: your open files, your chat history, and whatever you pasted in. If you wanted Copilot to look up a GitHub Issue, you had to copy the issue text and paste it into chat yourself. If you wanted it to check your CI status, same story, manual copy and paste. And if a tool vendor wanted their platform to work with an AI assistant, they had to build a separate integration for each AI host. If you had ten tools and five AI assistants, that's fifty custom integrations. That's the N times M problem. MCP solves this with a single standard protocol.

The Model Context Protocol

  • MCP is an open standard for connecting AI assistants to external tools and data sources
  • Think of it as a USB-C port for AI: one protocol, many integrations
  • MCP servers expose tools (actions), resources (data), and prompts (templates) that any MCP-compatible host can use
  • Originally created by Anthropic, now adopted across the industry by Microsoft, OpenAI, Google, and hundreds of tool providers
  • Thousands of pre-built MCP servers are available in the MCP registry for immediate use
The Model Context Protocol is an open standard that creates a universal connection layer between AI assistants and the outside world. The USB-C analogy is the easiest way to think about it. Before USB-C, every device had its own proprietary charger. USB-C gave us one port that works with everything. MCP does the same thing for AI. An MCP server exposes three types of capabilities: tools, which are actions the AI can take like "create an issue" or "run a query"; resources, which are data the AI can read like "list of open issues" or "contents of a database table"; and prompts, which are reusable templates for common tasks. Any AI host that speaks MCP can use any MCP server, and any MCP server works with any compatible host. One protocol, universal connectivity. The spec was originally created by Anthropic and is now adopted industry-wide. Microsoft, GitHub, OpenAI, Google, Amazon, and hundreds of other organizations have embraced MCP. The ecosystem includes thousands of pre-built servers in the MCP registry, ready for you to install and use.
Part 2

How MCP Works

Now that you know what MCP is at a high level, let's look at the architecture. Understanding the moving parts will help you troubleshoot connections and choose the right servers for your needs.

The MCP Architecture

  • Host: the application that contains the AI (VS Code, a CLI tool, a web app)
  • Client: the component inside the host that speaks the MCP protocol
  • Server: an external process that exposes tools, resources, and prompts over MCP
+-----------------------------------------+
|  VS Code (Host)                         |
|                                         |
|   Copilot Chat  -->  MCP Client  -------+--->  GitHub MCP Server
|                                         |
|                      MCP Client  -------+--->  Database MCP Server
|                                         |
|                      MCP Client  -------+--->  Custom MCP Server
+-----------------------------------------+
Here's the architecture. There are three layers. The host is the application that contains the AI; in our case, that's VS Code running Copilot Chat in agent mode. The client is the piece inside the host that speaks the MCP protocol. It handles the JSON-RPC messages, manages the connection lifecycle, and routes tool calls. The server is an external process that exposes capabilities. When Copilot needs to interact with GitHub, the MCP client inside VS Code sends a request to the GitHub MCP server. The server does the work, fetches issues, creates comments, and sends the result back through the client to Copilot. The key point is that VS Code can connect to multiple MCP servers simultaneously. You might have a GitHub server for issues and PRs, a database server for queries, and an infrastructure server for deployments, all running at the same time. Each one is a separate process, and the MCP client manages them all.

What MCP Servers Expose

Tools

Actions the AI can invoke: create_issue, run_query, send_message. The AI decides when to call them based on the conversation.

Resources

Data the AI can read: file contents, database schemas, API documentation. Resources provide context without taking action.

Prompts

Reusable templates exposed by the server: "Summarize this PR," "Generate a migration." Similar to prompt files but provided by the server.

MCP also supports advanced capabilities like sampling and elicitation, covered on the next slide.

MCP servers can expose three core types of capabilities. First, tools: these are actions. When the GitHub MCP server exposes a "create_issue" tool, Copilot can call it to create an issue on your behalf. The AI decides when to use tools based on the conversation context. Second, resources: these are read-only data. A database MCP server might expose your schema as a resource, so the AI can understand your data model without running any queries. Third, prompts: these are reusable templates that the server provides. A GitHub server might expose a "summarize this PR" prompt that's already optimized for working with pull request data. Not every server exposes all three. The GitHub MCP server focuses heavily on tools, while a documentation server might focus on resources. Beyond these three core primitives, MCP also supports advanced capabilities like sampling and elicitation, which we'll cover next.

Beyond the Basics: Sampling and Elicitation

Sampling

An MCP server can ask the host to perform an LLM completion on its behalf. This enables servers to reason about intermediate results using Copilot's AI, then continue processing with the answer.

Elicitation

An MCP server can request structured input from you, the user. When a server needs a decision or additional context, VS Code displays a form or dialog for you to respond.

  • Both capabilities require your explicit approval before they execute
  • Sampling and elicitation are less common than tools, but important to recognize when they appear
  • You control whether sampling and elicitation are allowed in your VS Code settings
Beyond tools, resources, and prompts, MCP defines two additional capabilities that you should know about. Sampling is a mechanism that lets an MCP server ask the host, in this case VS Code, to perform an LLM completion. For example, a code analysis server might need Copilot to summarize a piece of code as an intermediate step before producing its final output. When this happens, you'll see a prompt asking whether you want to allow the server to use the AI. Elicitation is the other advanced capability. It allows an MCP server to request structured input directly from you. If a server needs a decision, like which branch to target or which format to use, it can present a form in VS Code for you to fill out. Both sampling and elicitation require your explicit approval. You'll never be surprised by these interactions; VS Code surfaces them clearly and gives you full control. You can also configure whether these capabilities are allowed at all in your VS Code settings.

How Servers Connect

  • stdio: The server runs as a local process; VS Code communicates over standard input/output
  • Streamable HTTP: The server runs remotely; VS Code sends HTTP requests to a URL
  • GitHub's MCP server uses stdio; it runs locally and talks to the GitHub API on your behalf
  • Configuration lives in .mcp.json (workspace root), .vscode/mcp.json, or VS Code settings.json
  • The .mcp.json file at the workspace root is the recommended portable option
MCP servers connect to VS Code through one of two transport modes. The first is stdio, where the server runs as a local process on your machine and communicates through standard input and output. This is the most common mode and what GitHub's MCP server uses. VS Code spawns the process, sends JSON-RPC messages to its stdin, and reads responses from its stdout. The second mode is Streamable HTTP, where the server runs remotely and VS Code sends HTTP requests to a URL. This is useful for servers that need to run on shared infrastructure or that are too resource-intensive to run locally. There are three places to configure MCP servers. First, a .mcp.json file at the root of your workspace. This is the recommended approach because it's portable and easy to share. Second, a .vscode/mcp.json file. Third, your VS Code settings.json, which works for personal or user-level servers. We'll set this up hands-on in just a moment.
Part 3

Setting It Up

Enough theory. Let's get the GitHub MCP server running. This is where things get fun.

Enabling the GitHub MCP Server

  • MCP support is built into VS Code and enabled by default with the GitHub Copilot extension
  • The GitHub MCP server ships with the GitHub Copilot Chat extension; no separate install needed
  • Open Copilot Chat (Ctrl+Shift+I) and confirm the tools icon is visible
  • If needed, search Settings (Ctrl+,) for chat.mcp.enabled and verify it is checked
  • Reload VS Code (Ctrl+Shift+P → Reload Window) if you changed any settings
Setting up the GitHub MCP server is simple because MCP support is built into VS Code and enabled by default when you have the GitHub Copilot extension installed. The GitHub MCP server is bundled with the Copilot Chat extension, so there's nothing extra to install. Just open Copilot Chat and look for the tools icon. If for some reason MCP tools aren't appearing, open Settings with Ctrl+Comma and search for "chat.mcp.enabled" to confirm it's checked. After changing any settings, reload your window with Ctrl+Shift+P and type "Reload Window." Once the window reloads, the GitHub MCP server starts automatically in the background.

Verifying the Connection

  • Open Copilot Chat (Ctrl+Shift+I) to enter agent mode
  • Click the tools icon (wrench) in the chat input area
  • You should see GitHub MCP tools listed: create_issue, search_code, list_issues, etc.
  • If tools aren't visible, check the Output panel (Ctrl+Shift+U) → select GitHub MCP
  • Common fixes: confirm you're signed in to GitHub, verify chat.mcp.enabled is checked, reload the window
Once you've confirmed the settings, let's verify everything is working. Open Copilot Chat. When MCP tools are available, Copilot Chat operates in agent mode, which means it can autonomously call tools to complete multi-step tasks on your behalf. Look for the tools icon in the chat input area. Click it, and you should see a list of available MCP tools from the GitHub server. You'll see entries like "create_issue," "search_code," "list_issues," and more. If the tools aren't showing up, open the Output panel with Ctrl+Shift+U and select "GitHub MCP" from the dropdown. This is where the server logs connection events and errors. The most common issues are that you're not signed in to GitHub, the MCP setting isn't enabled, or the window needs a reload. Check those three things in order.
Part 4

Working With Issues

The GitHub MCP server exposes a bunch of tools, but the ones you'll use most often revolve around issues and pull requests. Let's see what that workflow looks like.

Reading Issues From Chat

  • Ask Copilot: "What are the open issues in this repo?"
  • Copilot calls the list_issues MCP tool behind the scenes
  • Results come back as structured data that Copilot formats into a readable summary
  • Try more specific queries: "Show me issues labeled 'bug'" or "What issues were opened this week?"
Here's where MCP gets practical. Open Copilot Chat and type: "What are the open issues in this repo?" Copilot recognizes that this question requires external data, so it calls the list_issues tool from the GitHub MCP server. The server fetches the data from the GitHub API and sends it back. Copilot then formats the results into a nice readable summary with issue titles, labels, and authors. You can get more specific too. Ask for issues by label, by assignee, by date range, or by milestone. The MCP server handles the filtering. The key insight here is that you never had to leave your editor, open a browser, or even think about the GitHub API. You just asked a question in natural language and got the answer.

Acting on Issues

  • Ask: "Add a comment to issue #42 saying we'll fix it in the next sprint"
  • Copilot calls the add_issue_comment tool with your message
  • You can also close issues, add labels, and assign people
  • By default, Copilot asks for confirmation before write actions; you can configure auto-approval for trusted tools
  • Full audit trail: every action is logged as a normal GitHub API call
Reading is great, but the real power is in taking action. You can tell Copilot: "Add a comment to issue 42 saying we'll fix it in the next sprint." Copilot calls the add_issue_comment tool, and the comment appears on the issue just like it would if you'd written it in the browser. You can close issues, add labels, change assignees, and more. By default, Copilot asks for your confirmation before taking any write action. It'll show you exactly what it's about to do, and you have to approve it. This is an important safety feature. If you find yourself approving the same actions repeatedly, VS Code also supports auto-approval for trusted tools, which you can configure in your settings. Everything goes through the normal GitHub API, so you get the full audit trail: the activity log, email notifications, webhook events, all of it. It's not a shortcut around your process; it's a faster way to participate in it.

Code Search With MCP

  • Ask: "Find all files that import the AuthService class"
  • Copilot uses the search_code tool to search across your repository
  • Results include file paths, line numbers, and code snippets
  • Combine with chat: "Find all usages of the deprecated API and suggest replacements"
  • Works across repositories, not just the one you have open
The code search tool is another one you'll reach for constantly. Ask Copilot something like "Find all files that import the AuthService class" and it uses the search_code MCP tool to search across your repository. You get back file paths, line numbers, and code snippets, all formatted nicely in the chat. But here's where it gets really powerful: you can combine search with Copilot's reasoning. Ask "Find all usages of the deprecated API and suggest replacements" and Copilot will search the code, analyze the results, and give you a migration plan. And the search isn't limited to your current repo. The GitHub MCP server can search across any repository you have access to. This is incredibly useful for understanding how a library is used across your organization.
Part 5

Discovering MCP Servers

We've been working with the GitHub MCP server, but that's just one of thousands. Let's look at how to discover, evaluate, and install other MCP servers from the community.

The MCP Registry

  • The official MCP server directory lives at github.com/modelcontextprotocol/servers
  • Hundreds of servers organized by category: databases, APIs, cloud platforms, developer tools
  • Each entry includes a description, configuration example, and required permissions
  • VS Code can also discover and manage MCP servers through the Command Palette and built-in MCP server management UI
  • Before installing, always review the server's source code, permissions, and maintenance status
The MCP registry at github.com/modelcontextprotocol/servers is the central directory for finding MCP servers. It's organized by category, so you can browse databases, APIs, cloud platforms, developer tools, and more. Each server entry includes a description of what it does, a ready-to-use configuration example, and a list of required permissions. VS Code also has built-in MCP server management. You can use the Command Palette to add, configure, and manage MCP servers directly from the editor. Note that MCP servers are not VS Code extensions; they are separate processes that communicate over the MCP protocol. Before you install any server, take a moment to review it. Check the source code, look at what permissions it requires, and verify that it's actively maintained. This is especially important for servers that will access sensitive data like databases or cloud accounts. We'll cover security in more detail shortly.

Installing a Registry Server

// .mcp.json (workspace root, recommended)
{
  "servers": {
    "postgres": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres"],
      "env": {
        "DATABASE_URL": "postgresql://localhost:5432/mydb"
      }
    }
  }
}
  • Create a .mcp.json file at your workspace root (the recommended portable option)
  • Copy the configuration example from the registry entry and adjust for your environment
  • Reload VS Code; the server starts automatically and its tools appear in Copilot Chat
  • Commit the file to share the configuration with your entire team
Installing an MCP server from the registry is straightforward. Browse the registry, find a server you want, and copy its configuration example. In this case we're adding a PostgreSQL server from the official MCP servers collection. Create a .mcp.json file at your workspace root. This is the recommended location because it's portable and easy to share with your team. Paste in the configuration, adjust the environment variables for your setup, and reload VS Code. The server starts automatically, and its tools appear in Copilot Chat right away. When you commit the .mcp.json file to your repository, every developer who opens the workspace gets the same MCP servers configured automatically. No manual setup, no documentation to follow. They just open the project and the tools are ready.

Security and Trust

Before we move to the live demo, let's cover security. MCP gives AI powerful access to external systems, so understanding the trust model is essential.

Securing Your MCP Setup

  • MCP servers run with your permissions: your GitHub token, your database credentials, your API keys
  • Write actions require explicit confirmation in Copilot Chat before they execute
  • You control which servers are enabled; disable any server at any time from your configuration
  • Least-privilege principle: grant each server only the permissions it needs
  • Store sensitive values like tokens and connection strings in environment variables, not in committed config files
Security starts with understanding the trust model. MCP servers run with your credentials. The GitHub MCP server uses your authenticated GitHub token, a database server uses your database credentials, and a cloud server uses your API keys. They don't get special elevated access; they operate as you. Every write action requires your explicit confirmation before it executes. Copilot shows you exactly what it's about to do, and you approve or reject it. You control which servers are active. If you don't want a server running, remove or disable it in your configuration. Follow the least-privilege principle: if a server only needs read access, don't give it write credentials. And never commit sensitive values like tokens or connection strings directly in your .mcp.json file. Use environment variables or secrets management instead.

Security Best Practices

  • Review before you install: read the server's source code and check its permissions, just as you would with any extension
  • Prefer official servers: servers in the official MCP registry are reviewed and maintained by the community
  • Monitor tool calls: use the Output panel (Ctrl+Shift+U → select the MCP server) to inspect what tools are being called
  • Keep secrets out of config files: use the "env" object in your MCP config for actual environment variables, or use ${input:variableName} to prompt the user at runtime
  • Enterprise controls: organization policies can restrict which MCP servers are allowed across your team
Here are five security practices to follow. First, review before you install. Read the source code and check what permissions the server requests, just like you would with a VS Code extension. Second, prefer servers from the official MCP registry. These are reviewed by the community and more likely to be well maintained. Third, monitor tool calls. The Output panel in VS Code shows exactly what each MCP server is doing. Select the server name from the dropdown and you can see every request and response in real time. Fourth, keep secrets out of your config files. You have two approaches. The "env" object in your MCP server configuration sets actual environment variables that the server process can read. Alternatively, the ${input:variableName} syntax prompts the user for a value at runtime, which is useful when credentials vary per developer. Both approaches keep sensitive values out of source control. Fifth, for enterprise teams, organization policies can restrict which MCP servers are allowed across your organization, giving administrators control over what connects to the AI in their environment.

MCP in Action

Alright, demo time. Let's bring everything together and see MCP in action with a complete issue workflow.

Demo: End-to-End Issue Workflow

  • Step 1: Ask Copilot to list open issues in the repository
  • Step 2: Pick an issue and ask Copilot to summarize it
  • Step 3: Ask Copilot to find related code using code search
  • Step 4: Write a fix with Copilot's help and add a comment to the issue
  • Step 5: Close the issue from chat
Here's what we're going to do. Step one: I'll ask Copilot to list all open issues in this repository. The MCP server fetches them and Copilot formats the results. Step two: I'll pick one of those issues and ask Copilot to summarize it, including the description, any linked PRs, and the discussion thread. Step three: I'll ask Copilot to find related code using the search tool. If the issue mentions a specific function or file, Copilot can locate it and bring it into the conversation. Step four: I'll work with Copilot to write a fix, and then ask it to add a comment to the issue describing what we did. Step five: once we're happy with the fix, I'll ask Copilot to close the issue. Five steps, zero browser tabs.

Demo: Combining MCP With Chat

  • Ask: "Summarize the last 5 closed issues and identify any recurring themes"
  • Copilot fetches the issues via MCP, then uses its reasoning to find patterns
  • Follow up: "Based on those themes, suggest improvements to our codebase"
  • This is the power of MCP + LLM reasoning: external data meets AI analysis
Now let me show you something that really demonstrates the power of MCP. I'll ask: "Summarize the last five closed issues and identify any recurring themes." Copilot fetches the issues through MCP, reads through the titles, descriptions, and comments, and then uses its reasoning ability to identify patterns. Maybe it notices that three out of five were related to authentication, or that there's a recurring pattern of null pointer exceptions in a specific module. Then I can follow up: "Based on those themes, suggest improvements to our codebase." Now Copilot is combining external data from GitHub with its ability to analyze code and suggest architectural improvements. This is the power of MCP: giving the AI access to real-world context so its reasoning is grounded in what's actually happening in your project.
