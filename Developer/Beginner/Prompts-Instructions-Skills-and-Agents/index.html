Part 1

The Building Blocks

Transition into the conceptual overview. This section introduces all four primitives at a high level before diving into each one individually.

The Four Primitives

Prompts

.prompt.md files that capture reusable tasks — run them from chat like saved commands.

Instructions

.instructions.md files that automatically shape every Copilot response in your workspace.

Skills

SKILL.md files that package domain knowledge agents can invoke on demand.

Agents

.agent.md files that define specialized modes combining instructions, skills, and tools.

Present each card briefly. The key distinction: Prompts are things you run. Instructions are things that run automatically. Skills are knowledge that agents pull in. Agents are the orchestrators that tie everything together. Each one is just a Markdown file — no build step, no compilation, no deployment. You commit them to your repo and Copilot finds them.

Where They Live in Your Repo

my-project/
├── .github/
│   ├── copilot-instructions.md    ← repo-wide instructions
│   ├── prompts/
│   │   ├── generate-api-test.prompt.md
│   │   └── explain-error.prompt.md
│   ├── instructions/
│   │   ├── typescript.instructions.md
│   │   └── testing.instructions.md
│   ├── skills/
│   │   └── azure-deploy/
│   │       └── SKILL.md
│   └── agents/
│       ├── code-reviewer.agent.md
│       └── docs-writer.agent.md
└── src/
Walk through the directory tree. Everything lives under .github/ — the same directory that houses workflows, issue templates, and other GitHub configuration. Copilot-instructions.md at the root of .github/ is the repo-wide instruction file. The subdirectories (prompts/, instructions/, skills/, agents/) are conventions that keep things organized. Copilot discovers files by their extension, not by their folder — so you could put a .prompt.md file anywhere, but the .github/ convention keeps your repo tidy and signals intent to your team.
Part 2

Prompt Files

Transition into the first primitive. Prompt files are the simplest and most approachable entry point — a great place to start for teams that are new to Copilot customization.

What Is a Prompt File?

  • A Markdown file with the .prompt.md extension
  • Captures a reusable task — like a saved chat message you can run again and again
  • Can include variables, file references, and output format instructions
  • Shows up in the chat prompt picker — type / and select it
  • Committed to your repo → shared with your entire team automatically
Explain the concept: a prompt file is essentially a saved chat message that you've refined until it reliably produces the result you want. Instead of typing the same complex prompt every time, you save it as a .prompt.md file and invoke it with a single click. The key insight is that this moves prompt engineering from an individual skill to a team asset — when someone on your team writes a great prompt, everyone benefits.

Anatomy of a Prompt File

---
mode: ask
description: Generate a unit test for the selected function
tools:
  - changes
---

Write a comprehensive unit test for the function in the
current selection.

**Requirements:**
- Use the project's existing test framework
- Cover happy path, edge cases, and error conditions
- Follow the naming convention: `test_<function>_<scenario>`

**Context:**
- Test file: #file:${file}
- Selected code: ${selection}
Break down each section. The YAML frontmatter between the triple dashes sets metadata: "mode" controls which chat mode runs the prompt (ask, edit, or agent), "description" appears in the picker UI, and "tools" lists the VS Code tools the prompt is allowed to use. Below the frontmatter is the prompt body in plain Markdown. Variables like ${file} and ${selection} are replaced at runtime with the current file and selection. The hash-file syntax (#file:) tells Copilot to include that file's contents as context. This is the single most important slide for this section — spend time here.

Running & Sharing Prompts

  • In chat, type / to open the prompt picker
  • Select your prompt file → Copilot fills in the variables and runs it
  • Or right-click a file → Copilot → Run Prompt
  • Prompts committed to .github/prompts/ are available to every team member
  • Build a prompt library — curated, tested prompts for common tasks
Prompt picker in chat
Demonstrate the two ways to run a prompt: the slash command in chat and the right-click context menu. Emphasize the team-sharing angle: once a prompt is committed to the repo, every developer who clones it gets the same prompts in their picker. This is how organizations scale prompt engineering — instead of everyone writing their own variation, you curate a shared library of tested prompts. Common examples include: generating API tests, writing migration scripts, creating documentation from code, and reviewing pull requests against team standards.
Part 3

Instruction Files

Transition to the second primitive. While prompts are things you actively run, instructions are always-on guidance that shapes Copilot's behavior passively. This is the most impactful primitive for team consistency.

What Are Instruction Files?

  • Markdown files that Copilot reads automatically — no need to invoke them
  • Two flavors:
    • .github/copilot-instructions.md — applies to every Copilot interaction in the repo
    • .instructions.md files — scoped to specific file patterns with applyTo
  • Think of them as coding standards that Copilot actually follows
  • No enforcement overhead — they're injected into every prompt behind the scenes
This is the "aha" moment for most teams. Instructions let you encode your coding standards, architectural decisions, and team conventions into a file that Copilot respects in every interaction. Instead of hoping developers remember to say "use TypeScript strict mode" in every chat, you write it once in an instruction file and it applies everywhere. The key distinction from prompts: prompts are active (you choose to run them), instructions are passive (they apply automatically).

Repo-Wide: copilot-instructions.md

# Copilot Instructions

## Language & Framework
- Always use TypeScript with strict mode enabled
- Prefer async/await over raw Promises or callbacks
- Use functional components with hooks (no class components)

## Testing
- Write tests using Vitest, not Jest
- Every public function must have at least one test

## Style
- Use single quotes for strings
- Maximum line length: 100 characters
- Prefer named exports over default exports
Show this example and explain that it lives at .github/copilot-instructions.md — one file, one location, applies everywhere. Walk through each section: language and framework choices, testing standards, and style preferences. Emphasize that these are guidelines Copilot will follow when generating code, writing tests, or answering questions about the codebase. The file is plain Markdown, so it's easy to review in PRs and evolve over time. Pro tip: start small and add rules as you discover recurring issues in Copilot's output.

Scoped: .instructions.md Files

---
applyTo: "**/*.test.ts"
---

# Testing Instructions

- Use `describe` / `it` blocks (not `test`)
- Mock external services with `vi.mock()`
- Assert with `expect().toBe()` — never use `assert`
- Name test cases: "should <expected behavior> when <condition>"
  • The applyTo glob pattern limits when these instructions activate
  • Only loaded when you're working on files that match the pattern
  • Perfect for language-specific, framework-specific, or folder-specific rules
The applyTo field is a glob pattern — standard file-matching syntax that developers already know from .gitignore. In this example, the testing instructions only activate when the developer is working on .test.ts files. You can create multiple instruction files: one for React components (applyTo: "src/components/**"), one for API routes (applyTo: "src/api/**"), one for database migrations. This keeps instructions focused and avoids loading irrelevant guidance. The instructions are additive — scoped instructions combine with the repo-wide copilot-instructions.md.
Part 4

Skills

Transition to the third primitive. Skills are the newest addition to the Copilot customization model. While instructions tell Copilot how to behave, skills give it domain-specific knowledge it can draw on when needed.

What Is a Skill?

  • A domain knowledge package defined in a SKILL.md file
  • Contains expertise that agents can invoke on demand — not always loaded, just available
  • Triggered by a description — when an agent recognizes the task matches a skill, it reads the file
  • Unlike instructions (always-on), skills are pulled in only when relevant
  • Great for: deployment procedures, API patterns, compliance rules, framework guides
The distinction between instructions and skills is subtle but important. Instructions are always injected — they're lightweight rules. Skills are heavier, more detailed knowledge packages that agents pull in only when the task at hand requires them. Think of instructions as "always follow these rules" and skills as "here's a manual you can consult when you need it." This lazy-loading approach keeps the context window efficient: a skill about Azure deployment doesn't consume tokens when you're writing React components.

Anatomy of a SKILL.md

# Azure Deployment Skill

## Description
Use this skill when deploying services to Azure Container Apps.
Covers Bicep templates, azd configuration, and managed identity setup.

## Prerequisites
- Azure CLI installed and authenticated
- azd CLI version 1.5+

## Deployment Steps
1. Run `azd init` to scaffold the project
2. Configure `azure.yaml` with service definitions
3. Run `azd up` to provision and deploy

## Bicep Patterns
- Always use managed identity (no connection strings)
- Tag all resources with `environment` and `team`
- Use Key Vault references for secrets
Walk through the SKILL.md structure. Unlike prompts and instructions, there's no YAML frontmatter — the file is pure Markdown. The description at the top is what agents use to decide whether to invoke the skill. Everything below is the knowledge payload: prerequisites, step-by-step procedures, code patterns, and best practices. Skills can be as long as needed — remember, they're only loaded when relevant, so there's no cost to comprehensive documentation. Teams often create skills for: CI/CD pipelines, database migration procedures, API design standards, and security review checklists.

When Skills Shine

Onboarding

New developers get instant access to institutional knowledge — "How do we deploy?" is answered by a skill, not a person.

Consistency

Every agent uses the same deployment procedure, the same API patterns, the same security rules.

Scaling Expertise

Your team's best practices are codified and available to every developer through every agent.

Frame skills as a knowledge-management tool, not just a Copilot feature. The onboarding benefit is compelling: instead of a new hire spending days reading wikis and asking colleagues, they use an agent that has access to skills covering deployment, testing, architecture, and team conventions. The consistency benefit matters at scale: when ten developers use the same deployment skill, they all follow the same steps. The expertise-scaling angle resonates with engineering leaders: one platform engineer writes a skill once, and every developer on the team benefits.
Part 5

Custom Agents

Transition to the final and most powerful primitive. Agents combine everything we've covered — they can have their own instructions, invoke skills, and access specific tools. This is where the four primitives come together.

What Is a Custom Agent?

  • A specialized Copilot mode defined in an .agent.md file
  • Has its own instructions, tool access, and personality
  • Can reference skills for domain knowledge
  • Appears in the chat participant picker — invoke with @agent-name
  • Think of it as a team member with a specific role — reviewer, architect, tester
Agents are the most powerful primitive because they orchestrate all the others. An agent can have its own instructions that override or supplement the repo-wide ones, access specific tools (like terminal, file editing, or web search), and invoke skills when it encounters a task that matches. You invoke an agent in chat with the @ syntax, just like @workspace. The mental model: each agent is a virtual team member with a defined role, expertise area, and set of capabilities.

Anatomy of an Agent File

---
description: Reviews code for security vulnerabilities and best practices
tools:
  - changes
  - codebase
  - fetch
---

# Security Reviewer

You are a senior security engineer. When reviewing code:

1. Check for OWASP Top 10 vulnerabilities
2. Flag hardcoded secrets or credentials
3. Verify input validation at system boundaries
4. Ensure authentication and authorization are correct
5. Look for SQL injection, XSS, and command injection

**Tone:** Be direct and specific. Reference CWE numbers.
**Output:** Provide a summary table of findings with severity levels.
Walk through the three parts. The frontmatter defines: description (shown in the picker UI), and tools (what the agent is allowed to use). The body is the agent's instructions — its persona, methodology, and output format. Notice that this agent has access to "changes" (can see diffs), "codebase" (can search the project), and "fetch" (can read URLs). You could restrict an agent to fewer tools for safety. The persona and methodology sections are critical: they define how the agent thinks, not just what it knows. A well-crafted agent feels like a real colleague with a specific skill set.

Agents in Action

  • In chat, type @ to see all available agents
  • Custom agents appear alongside built-in participants like @workspace
  • Each agent maintains its own conversation context
  • Agents can invoke skills automatically when they encounter a relevant task
  • Chain agents: ask one to generate code, another to review it
Custom agents in the picker
Demonstrate the agent picker by typing @ in the chat input. Show that custom agents appear in the list alongside built-in participants. If you have a live demo, invoke the security reviewer agent on a file and show the output. Emphasize the chaining pattern: you can ask a code-generation agent to build a feature, then ask a review agent to audit it, then ask a documentation agent to write the README section. Each agent brings its specialized perspective, and they compose naturally through chat.

Agents vs. Chat Modes

Chat Modes

  • Ask — read-only Q&A, no file changes
  • Edit — can modify files you specify
  • Agent — full autonomy: edit, create, run commands

Modes control what Copilot can do.

Custom Agents

  • Have their own instructions and personality
  • Restricted to specific tools
  • Invoke skills for domain knowledge

Agents control how Copilot thinks.

This distinction confuses people, so spend time here. Chat modes (Ask, Edit, Agent) are the capability levels — they determine what Copilot is allowed to do. Custom agents are the persona and expertise — they determine how Copilot approaches a task. A custom agent runs within a chat mode. For example, your security reviewer agent might run in Ask mode (read-only review) or Agent mode (can apply fixes autonomously). The mode is the permission level; the agent is the expertise. You can also create custom chat modes, which combine the two concepts for advanced workflows.
Putting It Together

The Decision Framework

Transition to the synthesis section. Now that we've covered each primitive individually, we'll help the audience decide which one to reach for in any given situation.

When to Use What

Repeating a task?

→ Prompt file
Save it, share it, run it from the picker.

Enforcing a standard?

→ Instruction file
Always-on, automatic, zero effort for developers.

Documenting expertise?

→ Skill
On-demand knowledge that agents pull in when needed.

Need a specialist?

→ Agent
A virtual team member with its own persona and tools.

This is the slide people will photograph. Walk through each scenario and make it concrete: "If you find yourself typing the same prompt every morning to generate your test boilerplate, that's a prompt file. If you've told Copilot to use async/await five times this week, that's an instruction file. If your team's deployment process is a 20-step procedure that lives in a wiki nobody reads, that's a skill. If you want a security reviewer who applies your company's specific standards, that's an agent." Give the audience a moment to internalize and ask questions.

How They Work Together

  • A prompt file kicks off a task: "Generate an API endpoint"
  • Instructions shape the response: "Use TypeScript, async/await, Express patterns"
  • The agent invokes a skill: "Azure deployment" procedures for hosting the endpoint
  • The agent orchestrates: reads code, writes files, runs tests, deploys

Prompts trigger. Instructions guide. Skills inform. Agents execute.

This slide ties everything together with a concrete example. Walk through the flow: a developer runs a prompt to generate an API endpoint. Copilot's response is shaped by the instruction files in the repo (TypeScript, async/await). If the developer is using a custom agent, the agent can invoke the Azure deployment skill when it's time to set up hosting. The agent has the autonomy to read files, write code, run tests, and execute commands. The tagline at the bottom — "Prompts trigger, Instructions guide, Skills inform, Agents execute" — is the one-liner summary of the entire course.
