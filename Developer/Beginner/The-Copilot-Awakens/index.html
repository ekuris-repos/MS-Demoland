Part 1

What Is GitHub Copilot?

Transition into the conceptual portion of the session. This section divider signals Part 1. Keep it brief and advance promptly.

Your AI Pair Programmer

  • GitHub Copilot is an AI-powered code assistant that lives inside your editor
  • Trained on billions of lines of public code — understands patterns, APIs, and idioms across dozens of languages
  • Suggests whole lines, entire functions, tests, documentation, and even terminal commands
  • Works in VS Code, JetBrains IDEs, Neovim, Visual Studio, and github.com
Explain that Copilot is not a standalone product but an integrated feature within the tools developers already use. Stress the breadth of language support (dozens of languages, not limited to Python or JavaScript). If the audience is enterprise-focused, note that Copilot is backed by GitHub and Microsoft, and that it has been generally available since 2022 with continuous model improvements. Avoid deep technical detail on the model architecture at this point; that comes on the next slide.

How Copilot Works

  • Context gathering — Copilot reads your open file, neighboring tabs, comments, and function signatures
  • Model inference — Sends context to a large language model (LLM) hosted by GitHub
  • Suggestion delivery — Returns completions as grey "ghost text" in your editor, or full answers in chat
  • Your code stays yours — Copilot Business and Enterprise do not retain prompts or suggestions for model training
Walk through the three-step process: context gathering, model inference, and suggestion delivery. Emphasize that Copilot sends context to a cloud-hosted LLM and returns suggestions in real time. The privacy point is critical for enterprise audiences: on the Business and Enterprise plans, GitHub does not retain prompts or completions for model retraining. If attendees raise concerns about sensitive code, refer them to the Copilot Trust Center documentation. Common question: "Does Copilot see my entire codebase?" The answer is that it sends a limited context window, not the full repository.

Installing & Activating Copilot

  • Open VS Code and go to Extensions (Ctrl+Shift+X)
  • Search for "GitHub Copilot" and click Install
  • A single extension now bundles both inline completions and Copilot Chat
  • Sign in with your GitHub account when prompted
  • Verify activation: look for the Copilot icon in the status bar (bottom right)

  • If your org uses Copilot Business/Enterprise, ensure your admin has granted you a seat

Part 3

Inline Completions

This section covers the core feature most developers encounter first. Signal to the audience that the next few slides explain how ghost text works and how to get the best results from it.

Ghost Text — Your New Superpower

  • As you type, Copilot shows grey "ghost text" — predicted completions inline
  • Press Tab to accept the full suggestion
  • Press Ctrl+→ to accept word-by-word
  • Press Esc or keep typing to dismiss and get a new suggestion
  • Use Alt+] / Alt+[ to cycle through alternative suggestions
Ghost text and hotkeys
This is typically the most impactful slide for newcomers. If you are presenting live, open a blank JavaScript file and start typing a function signature to trigger ghost text. Walk through each keyboard shortcut as you demonstrate: Tab to accept, Ctrl+Right Arrow for partial acceptance, Escape to dismiss, and Alt+] to cycle alternatives. Emphasize that partial acceptance with Ctrl+Right Arrow is one of the most underused and most valuable shortcuts. Let the audience see the ghost text appear and disappear in real time.

Getting Better Suggestions

  • Write a comment first — describe the function in plain English, then let Copilot generate it
  • Meaningful names — calculateShippingCost gives better results than doStuff
  • Open related files — Copilot uses neighboring tabs as context
  • Type signatures and JSDoc — the more structure you give, the better the completions
Ghost text completing from a comment
Walk through the four techniques for improving suggestion quality. The comment-first approach is especially effective: type a natural-language description of what the function should do, press Enter, and let Copilot generate the implementation. Show the code example on screen and explain that descriptive function names and parameter names act as implicit prompts. If time permits, demonstrate opening a second file tab and showing how Copilot draws context from neighboring files.
Part 4

Copilot Chat

Transition to the conversational interface. Note that many developers use inline completions regularly but have not yet explored Chat, so this section may be new even for experienced Copilot users.

Conversational AI in Your Editor

  • Open Copilot Chat with Ctrl+Shift+I (or click the chat icon in the sidebar)
  • Ask questions in natural language — "How do I read a file in Node?"
  • Copilot responds with explanations, code snippets, and follow-up suggestions
  • Conversations are context-aware — Copilot sees your open file and selection
  • Use inline chat (Ctrl+I) for quick edits right in the editor
Open Chat (Ctrl+Alt+I)
Distinguish between the two chat surfaces. The sidebar chat panel (Ctrl+Shift+I) is a full conversational interface suitable for longer exchanges. Inline chat (Ctrl+I) opens a small input directly in the editor and is designed for quick, targeted edits such as "rename this variable" or "add a try-catch block." Demonstrate both if time allows. Emphasize that Chat is context-aware: it can see the currently open file and any selected text, which means the more precise the context, the better the response.

Useful Chat Commands

  • /explain — Explain selected code in plain language
  • /fix — Suggest a fix for problems in the selection or file
  • /tests — Generate unit tests for the selected code
  • /doc — Generate documentation comments
  • @workspace — Ask questions about your entire project, not just the active file
Chat interface
Walk through each command and its use case. The /explain command is excellent for onboarding onto an unfamiliar codebase. The /fix command analyzes the selection and proposes corrections, which can save significant debugging time. The /tests command generates unit tests aligned with the structure and framework of the project. The @workspace participant is especially powerful because it indexes the entire project and can answer questions like "Where is the database connection configured?" or "Which files import this module?" If the audience has specific questions about available commands, mention that typing / in the chat input shows the full list.
Part 5

Configuring Your Workspace

This section addresses workspace-level settings, which are relevant for teams that want to standardize Copilot behavior across their organization. If the audience is primarily individual developers, you can move through this section more quickly and focus on the custom instructions slide.

Key Copilot Settings

  • github.copilot.enable — Toggle Copilot on/off per language: {"python": true, "markdown": false}
  • editor.inlineSuggest.enabled — Master switch for inline ghost text
  • github.copilot.chat.localeOverride — Force chat responses in a specific language
  • Use Workspace Settings (.vscode/settings.json) to enforce team-wide defaults
// .vscode/settings.json
{
  "github.copilot.enable": {
    "*": true,
    "plaintext": false,
    "markdown": false
  }
}
Open VS Code Settings (Ctrl+Comma) and search for "copilot" to show the available options. Highlight the per-language enable/disable map, which allows teams to turn off Copilot for file types where it may not add value (for example, Markdown or plain text). Show the example settings.json code on screen. Mention that placing this file inside .vscode/settings.json and committing it to the repository ensures that every team member inherits the same configuration automatically.

Custom Instructions

  • Create a .github/copilot-instructions.md file in your repo
  • Copilot reads these instructions and applies them to every interaction
  • Great for enforcing coding standards: "Always use TypeScript strict mode", "Prefer async/await over callbacks"
  • Team consistency — every developer gets the same AI guidance from day one
This is one of the most valuable features for enterprise teams. Explain that the copilot-instructions.md file lives in the .github directory of any repository and is automatically included in every Copilot interaction for that project. Provide concrete examples: "Always use TypeScript strict mode," "Follow our internal API naming conventions," or "Prefer async/await over raw Promises." The key value proposition is that institutional knowledge is codified and applied consistently, which reduces onboarding time and improves code quality across the team. Copilot in Action

Demo: Writing a Function with Copilot

  • Create a new file utils.js
  • Write a comment: // Parse a CSV string into an array of objects
  • Watch Copilot generate the full function
  • Accept with Tab, then ask Chat to add error handling
// Parse a CSV string into an array of objects
// using the first row as headers
function parseCSV(csvString) {
  const lines = csvString.trim().split('\n');
  const headers = lines[0].split(',').map(h => h.trim());
  return lines.slice(1).map(line => {
    const values = line.split(',').map(v => v.trim());
    return headers.reduce((obj, header, i) => {
      obj[header] = values[i];
      return obj;
    }, {});
  });
}
This is the primary demonstration slide. Create a new file called utils.js. Type the comment slowly so the audience can read it on screen, then pause and let Copilot generate the function body. Narrate what is happening: "Copilot has read the comment and is now generating a complete implementation." After the ghost text appears, press Tab to accept it. Walk through the generated code line by line, pointing out the use of split, map, and reduce. If the suggestion does not match what is shown on the slide, that is expected and can be used as a teaching moment about the nondeterministic nature of LLM output. After accepting, transition to Chat by saying: "Now let us ask Copilot to improve this."

Demo: Using Chat to Refactor

  • Select the function → open inline chat (Ctrl+I)
  • Type: "Add error handling for malformed CSV input"
  • Review the diff — Copilot shows additions in green, removals in red
  • Click Accept or Discard
  • Then ask: "Generate unit tests for this function"
Select the parseCSV function, then press Ctrl+I to open inline chat. Type the error-handling prompt and show the diff that Copilot produces. Explain that green lines are additions and red lines are removals, and that the developer always has the final say with Accept or Discard. After accepting the refactored version, open the sidebar chat (Ctrl+Shift+I) and type "/tests" to generate unit tests. Walk through the generated test cases and note how Copilot covers normal inputs, edge cases, and error conditions. This sequence demonstrates a complete workflow: generate, explain, refactor, and test.

Tips for Getting the Most from Copilot

  • Be specific — "Write a Python function that validates an email address using regex" beats "write code"
  • Iterate — Accept a suggestion, then refine it with follow-up prompts
  • Review everything — AI suggestions are starting points, not final answers
  • Use context — Open related files, add comments, use meaningful names
  • Learn the shortcuts — Tab, Ctrl+→, Alt+], Ctrl+I, Ctrl+Shift+I
Read through each tip and provide a brief example for each. "Be specific" is the single most impactful piece of advice: vague prompts produce vague results. "Iterate" reinforces that the AI is a collaborator, not a one-shot answer machine. "Review everything" is a professional responsibility point: AI suggestions can contain bugs, outdated APIs, or patterns that do not match organizational standards. "Use context" emphasizes that Copilot performs better when it has more to work with. Close with the shortcuts, which are the most immediately actionable takeaway from the session.
